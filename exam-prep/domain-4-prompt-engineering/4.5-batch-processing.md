# 4.5 — Design efficient batch processing strategies

> **Domain 4: Prompt Engineering & Structured Output** (20%) · **Task Statement 4.5** | **Scenarios:** Claude Code for CI, Structured Data Extraction | **What it tests:** the ability to match a task's latency profile to the right execution mode — send to the `Message Batches API` only asynchronous, latency-tolerant workloads.

## ⚡ TL;DR

- The `Message Batches API` gives a **~50% savings** on tokens, but it is processed asynchronously and can take **up to 24 hours** (no guaranteed latency SLA, though in practice most batches finish in <1 hour).
- A batch is suitable **only** for workloads off the critical path: nightly reports, weekly audits, overnight test generation, bulk data extraction. Blocking tasks (a pre-merge gate that a developer is waiting on) must stay synchronous.
- A batch **does not support an interactive client-side tool-loop**: you cannot intercept a `tool_use` on the fly, run a function, and return a `tool_result` to continue reasoning. At the same time the batch itself **accepts** `tools` definitions, server-side tools, and a pre-assembled multi-turn conversation — it is exactly the *interactive* loop that is impossible. This is a **control flow** limitation, not a latency one.
- `custom_id` correlates request/response pairs (results do not arrive in order). On failure, **resubmit only the failed `custom_id`s** with fixes (for example, by splitting an oversized document into chunks).
- Compute the submission frequency from the SLA: to guarantee a 30-hour SLA with a 24-hour batch window, submit in windows of about 4 hours.

## 🧠 In plain words

> **Analogy:** 🌙 a nighttime freight truck — cheap and a lot at once, but "I need it right now" does not travel that way.
>
> The `Message Batches API` is "deferred delivery": you send a batch of requests and pick up the finished work later. You pay ~50% (half the cost), but the window is **up to 24 hours**, and nobody guarantees the timing (no latency SLA = no promise of "I'll be done in N minutes"). It is only good for latency-TOLERANT tasks (nightly reports, audits) where nobody is waiting for a reply right now. And one more thing: in a single request you **cannot run a multi-turn tool loop** (Claude won't stop in the middle to call a tool and continue).
>
> 🚩 **Detector:** if the answer batches a **blocking** task (a pre-merge gate that a developer is waiting on) — that's symptom-not-root-cause (they mixed up the latency profile); the right move is to batch only what's off the critical path.
>
> 🧷 **Reflex:** "Someone's waiting right now → synchronous. It can wait until morning → batch. Failed — resend only the failed `custom_id`s."

## 📘 Theory

### Principle: matching workload to mechanism

The key decision here is to classify the task by its **real deadline**. If **nobody is waiting on the result in real time** (off the critical path) and the acceptable delay is wider than the worst-case batch processing window, then it is a candidate for the `Message Batches API`. If the task blocks a person or a pipeline (on the critical path), it stays synchronous (`/v1/messages`).

The `Message Batches API` is a **fire-and-forget** model: you send fully formed requests and pick up the finished results by polling. In exchange for immediacy you get a 50% discount and higher throughput.

### What exactly the batch gives you

- A **~50% discount** on the standard token price (both input and output).
- A **window of up to 24 hours**: the batch expires (`expired`) if it does not finish in 24h; for `expired`/`canceled` requests you **do not pay**. There is no timing guarantee — you cannot count on "it'll be done in N minutes".
- Polling of the batch status; results become available when **all** requests are complete (or after 24h have elapsed).

### custom_id — correlating results

Results are returned **in arbitrary order**, so each request carries a unique `custom_id` (format `^[a-zA-Z0-9_-]{1,64}$`). You use it to match a response to the original request. Correlation is a **solved problem**, not a blocker. When processing large volumes: first debug the prompt on a small sample, then run the full batch; on failures, resubmit **only** the failed `custom_id`s with a modification (chunking oversized documents and so on).

### Why iterative tool-loops break

When a tool is executed **on the client side**, you need a live loop: Claude emits a `tool_use` → the application pauses and runs the tool → returns a `tool_result` → Claude continues. A batch has **no interception point** in the middle of a request, so "pause-and-resume" agentic loops are impossible in a batch. This is a **fundamental control flow incompatibility**, not a "too slow" issue. (Note: a batch *accepts* `tools` definitions and server-side tools just like a regular Messages request — you cannot deny that in an answer; it is exactly the multi-turn client-tool loop that is impossible.)

#### A precise reading of "no multi-turn tool calling support"

The official Appendix uses the wording "no multi-turn tool calling support" — you have to understand it precisely, otherwise it's easy to fall into a trap.

**What the batch API does NOT support:** an interactive client-side tool loop — that is, you cannot intercept a `tool_use` in the middle of a request, run the tool on the client, add a `tool_result`, and continue iterating. A batch request is "fire and forget": send → wait → receive the final answer.

**What the batch API DOES support:**
- `tools` definitions in the request (Claude knows about the tools)
- Server-side tools (e.g., `web_search` — Anthropic runs them on their side)
- A pre-assembled multi-turn conversation as input

The implication for the exam: the distractor "batch can't accept tool definitions at all" is **wrong** (it does accept them). The correct limitation: the client-side loop `tool_use → execute → tool_result → continue` is impossible within a single batch request.

### Getting the results: polling

The batch API is asynchronous — there are no results yet right after the POST request. To get them, you need to **poll the status** (polling):

1. `POST /v1/messages/batches` → returns a `batch_id` and the status `in_progress`.
2. Periodically: `GET /v1/messages/batches/{batch_id}` → check `processing_status`.
3. When `processing_status == "ended"` → `GET /v1/messages/batches/{batch_id}/results` → stream JSONL with the results.

Each result contains a `custom_id` (to correlate with the request) and a `result` (either a successful response or an error). Results can be returned **out of order** relative to the requests — `custom_id` is the only way to match a request to a response.

Important details: expired and canceled requests are **not billed**. Typical processing time is << 24h, but the SLA is not guaranteed — do not use it in blocking scenarios.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **Batching a blocking pre-merge check that a developer is waiting on** | A pre-merge gate is on the critical path; a 24-hour delay would stall every merge. You can only batch nightly/deferred analysis (official Sample Q11: the correct answer is to batch *only* the overnight report). |
| **"Batch can run an interactive client-side tool-loop in a single request"** | There is no point to intercept a `tool_use` in the middle of a request — an iterative client-tool loop is impossible. (The `tools` definitions themselves, server-side tools, and a pre-assembled multi-turn conversation the batch does accept — it is exactly the *interactive* loop that's wrong.) A control flow limitation, not latency. |
| **"Batch results can't be matched to requests"** | False: `custom_id` correlates the pairs even when responses arrive out of order. Correlation is solved. |
| **"The 24h latency just makes the workflow slow but functional"** | This masks the real point: with client-side tool use the loop **won't run at all**, it's not about speed. |
| **A hybrid of "timeout → fallback to real-time"** | Unnecessary complexity. The right move is to match each workload to its API up front: blocking → synchronous, deferred → batch. Don't build a timeout-fallback where classification is enough. |
| **"Neither / batch nothing"** | Over-conservative: it correctly excludes the blocking check, but it throws away the legitimate 50% savings on the nightly workload. |
| **"the batch API doesn't accept tool definitions at all"** | Wrong. A batch accepts `tools`, server tools, and a ready multi-turn conversation. The limitation is exactly the interactive client-side tool loop (intercepting `tool_use` + `tool_result` + continuing within a single request). The confusion comes from the imprecise Appendix wording "no multi-turn tool calling". |

## 🔑 Remember

- `Message Batches API`: **~50% cheaper**, a window of **up to 24h**, **no latency SLA**, endpoint `POST /v1/messages/batches`.
- `custom_id` — `^[a-zA-Z0-9_-]{1,64}$`, unique within the batch, matches a response to a request (response order is not guaranteed).
- For `expired` and `canceled` requests you are **not charged**.
- A batch does **not** run a client-side multi-turn tool loop; but it **does accept** `tools` definitions and server-side tools.
- Recovery: resubmit **only** the failed `custom_id`s with fixes (chunking); debug the prompt on a sample before the full volume.
- Submission frequency from the SLA: for a 30h SLA with a 24h window — send every ~4h.
- **Getting the results:** `POST /v1/messages/batches` → `batch_id` → `GET .../results` when `processing_status == "ended"`; results in JSONL, order not guaranteed → correlation only via `custom_id`.

## 📝 Practice

*Answer on your own first, then expand "Answer".*

**Q1 · easy** — A nightly job summarizes 8,000 support tickets and writes a report read the next morning. You want to cut API cost. What is the most effective approach?

A) Keep it synchronous on `/v1/messages` to be safe.
B) Submit the 8,000 prompts via the `Message Batches API`, polling for results.
C) Split each ticket into a separate synchronous request fired in parallel at peak hours.
D) Use the batch API only if you can guarantee sub-minute completion.

<details><summary>Show answer</summary>

**Correct answer: B.** The workload is off the critical path (the report is read in the morning) and tolerant of delay — the canonical candidate for a batch with 50% savings.
A — over-conservative, throws away legitimate savings. C — synchronous parallel requests give no discount and don't solve the cost problem. D — a false requirement: the batch has no sub-minute guarantee, and it isn't needed here (the archetype "the 24h SLA is perceived as a blocker").
</details>

**Q2 · medium** — A code-review tool lets Claude request related files (imports, base classes, tests) via a client-executed tool, receive contents, and continue analysis over several rounds. You want to run it through the batch API to save 50%. What is the primary technical constraint?

A) The batch API rejects `tools` definitions in request parameters.
B) Batch results cannot be correlated back to their input requests.
C) The asynchronous model cannot execute a tool mid-request and return results for Claude to continue analysis.
D) The 24-hour latency is too slow, though the workflow would otherwise function.

<details><summary>Show answer</summary>

**Correct answer: C.** Client-side tool use requires a synchronous loop `tool_use` → execute → `tool_result` → continue; a fire-and-forget batch has no interception point in the middle of a request. This is a control flow incompatibility.
A — false: a batch accepts `tools` definitions. B — false: `custom_id` correlates the pairs. D — masks the real point: the loop won't run at all, it's not about speed (the archetype "latency as an apparent blocker").
</details>

**Q3 · hard** — Your CI has two Claude modes: a `pre-merge-commit` hook that blocks PR merging until it returns, and a "deep analysis" that already runs overnight and posts suggestions to the PR. You want the 50% batch savings. Which mode(s) should use the `Message Batches API`?

A) Both modes, to maximize savings.
B) Deep analysis only.
C) Neither mode — batch is too risky for CI.
D) The `pre-merge-commit` hook only, since it runs most often.

<details><summary>Show answer</summary>

**Correct answer: B.** Deep analysis is off the critical path and already poll-based — it fits the asynchronous batch perfectly and captures 50%. The pre-merge hook blocks the merge in real time, so it cannot be batched.
A — batches a blocking gate, 24h would stall every merge. C — over-conservative, throws away legitimate savings on the nightly workload. D — inverted: it batches the latency-sensitive gate and skips the latency-tolerant task.
</details>

### Tricky question **

**Q★★ (tricky):** A weekly compliance audit must process ~50,000 documents and deliver results within a **30-hour SLA**. You use the `Message Batches API` (24h window, 50% savings). Some documents exceed the model's context and their requests come back failed. What is the most effective operational design?

A) Submit all 50,000 in one batch at the SLA start; if any request fails, re-run the entire batch synchronously as a real-time fallback.
B) Submit in ~4-hour windows to keep total turnaround inside 30h, validate the prompt on a small sample first, and on failure resubmit only the failed `custom_id`s with oversized documents chunked.
C) Skip the batch API entirely because a 24h window cannot guarantee a 30h SLA.
D) Submit one batch and rely on automatic mid-request tool calls to fetch and split any document that overflows context.

<details><summary>Show answer</summary>

**Correct answer: B.** Two principles combine here: (1) computing the submission frequency from the SLA — windows of ~4h leave margin on top of the 24h window, keeping the total within 30h; (2) failure handling — resubmit **only** the failed `custom_id`s with a modification (chunking oversized documents), having first debugged the prompt on a sample.
A — tempting on "reliability", but a real-time fallback for 50k requests is unnecessary complexity and kills the 50% savings; the right move is to fix targeted by `custom_id`, not re-run everything. C — a false conclusion: a 24h window with margin fits within 30h under windowed submission; abandoning the batch throws away the savings. D — the mid-request tool calling trap: a batch does **not** execute tools in the middle of a request, the chunking is done by your client **before** resubmission, not by the model inside the batch.
</details>
