# 4.2 — Apply few-shot prompting to improve output consistency and quality

> **Domain 4: Prompt Engineering & Structured Output** (20%) · **Task Statement 4.2** | **Scenarios:** CI, Structured Data Extraction, Customer Support | **What it tests:** when a few targeted examples (`few-shot`) are the most effective lever for a stable format, for handling ambiguous cases, and for reducing hallucinations in extraction.

## ⚡ TL;DR

- `few-shot` (also called multishot) is the best tool when detailed instructions already exist but the output is still **inconsistent in format or classification**. Examples show the model what is hard to describe in prose.
- Use **2–4 targeted examples** (the official best-practices range is 3–5); quality and coverage of the ambiguous boundaries matter more than quantity. A pile of examples does not help and bloats the context.
- A good example shows the **reasoning** ("why action A was chosen, not the plausible B"), not just the finished answer — this gives **generalization** to new patterns, not memorization of specific cases.
- In extraction tasks, examples with **different document structures** (inline citations vs reference lists, informal measurements) reduce hallucinations, because they set the boundary of "what to extract and what not to".
- Don't confuse the levers: if the problem is the **absence of clear criteria** — that is 4.1 (explicit criteria); if you need a **guaranteed** format/validation — that is programmatic enforcement (`tool_choice`, schema, hooks, 1.4/1.5), not `few-shot`.

## 🧠 In plain words

> **Analogy:** 🖼️ "show, don't tell". When the format you need or the handling of an ambiguous case is hard to describe in words, 2–4 examples show it more clearly than any prose.
>
> `few-shot` is lever 2: the best choice when instructions already exist but the output is inconsistent **in format** or on ambiguous boundaries. A good example shows the **reasoning** ("why A, not the similar B") — then the model generalizes instead of memorizing specific cases. But if the problem is that the **criterion** is not defined — that is 4.1; and if you need a **guaranteed** format — that is code/schema (lever 4–5), not examples.
>
> 🚩 **Detector:** `few-shot` is suggested where you actually need to (a) define a rule → this is really 4.1 (explicit criteria), or (b) give a hard format guarantee → this is really `tool_choice`/schema/hook. Few-shot WITHOUT a criterion for inconsistent judgment → under-engineered.
>
> 🧷 **Reflex:** first the rule (criterion), then examples as boundary anchors; 2–4 targeted ones with reasoning, not a pile.

## 📘 Theory

### Why few-shot is needed and what principle stands behind it

The principle is **context engineering**: you steer the model's behavior by shaping its input, not by fine-tuning it. When "be clear and direct" has already been applied but the model still drifts (the same null-pointer is sometimes `critical`, sometimes `medium`; the same field is extracted one way then another), the cause is that **the abstract definition does not set the boundaries of the categories**. Examples do set them: they work as anchors to which the model ties new inputs. The official wording: "Examples are one of the most reliable ways to steer Claude's output format, tone, and structure."

### How many examples and what they should be

Task statement: **2–4 targeted examples** for ambiguous scenarios. The official best-practices add: for the best result use 3–5, and each example should be:

- **Relevant** — close to your real case;
- **Diverse** — covers edge cases and varies, so the model does not pick up a side pattern (for example, that "the first item is always = critical");
- **Structured** — wrapped in `<example>` / `<examples>` tags, so the model separates examples from instructions.

The key for the exam: **2–4 targeted** beats "as many examples as possible". Many unstructured examples do not raise accuracy — they add noise and waste context.

### Show the reasoning, not just the answer (generalization)

The most common mistake a candidate makes is examples that **memorize specific cases** instead of teaching how to judge. Correct few-shot for an ambiguous case shows **why** one action was chosen rather than a plausible alternative: "for request X, `get_order_status` was chosen, not `search_orders`, because there is an exact ID". Then the model **generalizes the judgment** to new patterns not foreseen in advance. Also demonstrate the **desired output format** (location, issue, severity, suggested fix) — this fixes both the format and the logic at the same time.

### Few-shot in extraction (reducing hallucinations)

In Structured Data Extraction, documents arrive with a **different structure** (inline citations vs a bibliography at the end; "about 5 ml" as an informal measurement). A detailed instruction does not cover all the forms. 2–4 examples on **different structures** show the model what counts as a valid extraction and what to skip, and thereby reduce inventing fields. This is exactly "reduce hallucination in extraction".

### Where NOT to apply few-shot (the lever's boundaries)

- You need a **clear scale/criteria** (what each severity level means) — first 4.1 (explicit criteria); examples reinforce the criteria but do not replace defining them.
- You need **100%-valid** JSON/format with no exceptions — this is **determinism through code**: `tool_choice`, JSON schema, hooks/validator (1.4/1.5), not probabilistic few-shot.
- The problem is in the **decomposition/tool contract** — fix the root, do not add examples to the prompt.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **"Give as many examples as possible"** | Volume does not equal quality. You need 2–4 (best-practices 3–5) **targeted** examples for the ambiguous boundaries. Many unstructured examples add noise and side patterns, and waste context. |
| **"Memorization" examples of specific cases** | If an example just demonstrates the finished answer without reasoning, the model will match only the cases given in advance and will not generalize to new ones. You need to show **why** the action was chosen. |
| **Few-shot where explicit criteria (4.1) are needed** | If the classification levels are not defined in principle, examples lean on emptiness. Set the criteria first; few-shot is an anchor to them, not a replacement. |
| **Few-shot where determinism (1.4/1.5) is needed** | A guaranteed schema/valid JSON comes from `tool_choice` + schema + hooks, not from probabilistic examples. Few-shot improves "usually", code ensures "always". |
| **Reasoning + manual calibration by people** | Asking the model to explain, and people to fix labels, does not scale and does not remove the original ambiguity; few-shot/criteria fix the source. |

## 🔑 Remember

- `few-shot` = `multishot`; format: examples in `<example>` (several — in `<examples>`) tags.
- **2–4 targeted** examples (task statement); the official range is **3–5** for the best result.
- Signs of a good example: **Relevant, Diverse, Structured**.
- Few-shot improves: **format consistency · handling ambiguous cases · generalization of judgment · reducing hallucinations in extraction**.
- An example for an ambiguous case should show the **reasoning** (why A, not the plausible B), not just the answer.
- The hierarchy of levers: explicit criteria (4.1) → few-shot (4.2) → programmatic enforcement/determinism (1.4/1.5). Few-shot ≠ guarantee.

## 📝 Practice

*Answer on your own first, then expand "Answer".*

**Q1 · easy**
Your extraction prompt has detailed written instructions, yet Claude still formats the `severity` field inconsistently across runs (`HIGH`, `High`, `critical`). What is the most effective next step?

A) Add 2–4 examples showing the exact desired output format for `severity`, wrapped in `<example>` tags.
B) Increase `max_tokens` so Claude has room to format correctly.
C) Add a longer prose paragraph re-explaining the formatting rules.
D) Ask Claude to rate its own formatting confidence after each response.

<details><summary>Show answer</summary>

**Correct answer: A.** Detailed instructions already exist, but the output drifts in format — a classic case for few-shot: a few examples of the desired format stabilize it better than any prose, because they show rather than describe.
B — irrelevant (`max_tokens` does not control format). C — "even more instructions" (repeating prose that already did not work). D — reasoning/self-assessment does not remove the format ambiguity and does not scale.
</details>

**Q2 · medium**
A code-review agent must flag genuine null-pointer bugs but ignore safe patterns that *look* similar (e.g. a value already null-checked upstream). Detailed rules haven't fixed the false positives. What's the best fix?

A) Provide 2–4 examples that contrast an acceptable pattern with a genuine issue, each showing the reasoning for why it was or wasn't flagged.
B) Add 20+ examples of every null-pointer variant you can find so coverage is exhaustive.
C) Tell the agent to flag anything that *might* be a null-pointer risk to be safe.
D) Lower the model temperature so the agent is more deterministic.

<details><summary>Show answer</summary>

**Correct answer: A.** A few contrasting examples with **reasoning** teach the model the **boundary** between a safe pattern and a real bug and give generalization to new cases — this is exactly the job of few-shot for ambiguous cases.
B — the "as many examples as possible" trap: 20+ add noise and do not scale; you need 2–4 targeted ones on the boundary. C — shifts behavior toward false positives, does not resolve the ambiguity. D — `temperature` affects the spread, but does not teach what counts as a real bug.
</details>

**Q3 · hard**
For ambiguous user requests, your support agent sometimes calls `process_refund` when it should call `check_order_status` first. You add few-shot examples. Which set of examples will best generalize to *new* ambiguous requests rather than only the ones you listed?

A) Examples that show, for each ambiguous request, the chosen tool plus the reasoning for why it was chosen over the plausible alternative.
B) One example per exact request string you've seen in logs, mapping it to the correct tool.
C) A single example showing a perfect multi-tool resolution end to end.
D) Examples that list every available tool with its description, but no request-to-tool decisions.

<details><summary>Show answer</summary>

**Correct answer: A.** Showing the **reasoning** (why `check_order_status`, not `process_refund`) carries the selection logic over to new patterns — this is "generalize judgment to novel patterns" from the task statement.
B — memorizing specific strings: it matches only the requests already seen, does not generalize. C — a single "everything is perfect" example does not teach the boundary for ambiguous cases. D — this is a description of the tools (tool docs / 2.x), not a demonstration of "request → tool" decisions.
</details>

### Tricky question **

**Q★★ (tricky):** Your extraction pipeline feeds Claude's output directly into a database that **rejects any row whose JSON is malformed or missing the `amount` field**. With detailed instructions plus 4 good few-shot examples, output is correct ~97% of the time, but the remaining ~3% still break ingestion and halt the pipeline. What is the most effective way to meet the hard requirement?

A) Add more few-shot examples (scale from 4 to ~15) covering the failing document structures until the error rate drops.
B) Keep the few-shot examples, but enforce the contract programmatically — constrain output with a tool/JSON schema (`tool_choice`) and validate before ingestion, rejecting/repairing non-conforming rows.
C) Replace the few-shot examples with a single, very explicit prose instruction stating the JSON must always be valid and always include `amount`.
D) Lower the temperature to 0 so the JSON is always valid.

<details><summary>Show answer</summary>

**Correct answer: B.** This combines two principles. Few-shot is a **probabilistic** lever: it raises "usually" (97%), but never guarantees "always". The requirement here is **hard** (the DB rejects any invalid/incomplete row), and only **determinism through code** gives hard guarantees — structuring the output with a schema/`tool_choice` and a validator before writing (Task 1.4/1.5). Few-shot remains useful for raising the baseline quality, but code ensures the contract.

Why the others are traps:
- **A** — the "as many examples as possible" archetype: scaling to 15 examples both adds noise and still stays probabilistic — it never guarantees 0% errors. This is a confusion of levers (few-shot instead of programmatic enforcement).
- **C** — "even more instructions": prose cannot guarantee validity; the same probabilistic nature, the same residual failure percentage.
- **D** — `temperature=0` reduces the spread, but does **not** guarantee valid JSON or the presence of the field; the model can still produce an incorrect structure. It sounds "deterministic", but it is not control at the contract level.
</details>
