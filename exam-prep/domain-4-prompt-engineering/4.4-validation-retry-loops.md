# 4.4 — Implement validation, retry, and feedback loops for extraction quality

> **Domain 4: Prompt Engineering & Structured Output** (20%) · **Task Statement 4.4** | **Scenarios:** Structured Data Extraction | **What it tests:** the skill to build meaningful validation and retry loops, to tell apart the errors that retry fixes from the ones it cannot, and to design self-correction (semantic vs syntax).

## ⚡ TL;DR

- **A useful retry = evidence, not a repeat.** On the retry, send the model the *original document* + the *failed extraction* + the *specific validation errors*. A bare repeat of the same prompt is not feedback — it is wasted tokens and latency.
- **Retry is powerless when the data is simply not in the source.** Retry fixes format / structural / semantic errors; it does not fix *missing* information (a field that is not in the document). That gets escalated to a human or fetched with an external lookup.
- **Strict tool use / structured outputs remove only the syntax errors of the schema** (constrained decoding guarantees valid JSON and types). They **do not catch semantic errors** — values that do not add up, a value placed in the wrong field.
- **Design self-correction explicitly.** Extract `calculated_total` next to `stated_total` to flag mismatches; add `conflict_detected` (boolean) when the source data itself is contradictory.
- **To analyze dismissals you need `detected_pattern`.** A field that records which construct produced a finding gives you a systematic breakdown of false positives and of the reasons findings get dismissed.

## 🧠 In plain words

> **Analogy:** 🔁 asking again helps only if the other person already answered and you misheard — then you say "say that again, I didn't catch this part." But if the number is simply NOT in the document, no amount of asking again will make it appear.
>
> The point: a useful retry is not "roll the dice again." Into the retry prompt put the *original* + *what the model produced* + *exactly what is wrong* ("`stated_total`=120, sum of line items=100"). The trap: retry fixes format and meaning errors (sums that don't add up), but it is **powerless** when the field is not in the source — that is a gap, you escalate it to a human. And one more thing: the schema (`strict: true`) guarantees only valid JSON and types; it does not check "do the sums add up" — that is caught by your validation.
>
> 🚩 **Detector:** if an answer suggests "retry again" to fetch a missing field, or hopes that a `strict` schema will catch sums that don't add up — that is *symptom-not-root-cause*; the right move is retry with a specific error only for format/semantic, and absent data → escalation / external lookup.
>
> 🧷 **Reflex:** misheard — ask again with the error; no data — you can't invent it, escalate.

## 📘 Theory

### Retry-with-error-feedback: the retry as targeted correction

When a structured extraction fails validation, a productive retry is not "roll the dice again." The model has to be given everything it needs to understand *exactly what* is wrong:

1. **The original document** — the source the data was pulled from.
2. **The failed extraction** — exactly what it produced.
3. **The specific validation errors** — not "invalid," but "`stated_total`=120, sum of line items=100, mismatch of 20."

This works because the model gets a targeted signal and can self-correct on format, structural, and semantic errors. **Principle:** the feedback has to be *specific* — without specifics the model has nothing to build on, and it will repeat the error just as easily.

### When retry cannot help at all: missing data

A key distinction the exam loves to ask about: **missing information is not an error you can fix with a retry.** If the document never had a tax rate or a PO number, no retry will "remember" it — this is not a format/structural/semantic problem, it is just a gap in the source. The right move is to mark it as non-retryable and **escalate to a human** or fetch it from an external source (a lookup, another system). A retry loop on missing data is wasted attempts, money, and latency with no chance of success.

### Semantic vs syntax: what the schema catches and what it doesn't

- **Syntax / schema errors** (invalid JSON, wrong type, a missing required field) **are removed by tool use / structured outputs.** Constrained decoding with `strict: true` guarantees that the output matches `input_schema` — "no retries needed for schema violations."
- **Semantic errors** are **invisible** to the schema: a `stated_total` that is well-typed but not equal to the sum of the lines; a value placed in the wrong field. The JSON is valid by type — but wrong by meaning.

**Design takeaway:** since syntax is closed off by the schema, your validation loop has to catch precisely the *semantic* errors. Hence the self-correction patterns: pull `calculated_total` next to `stated_total` and compare them; add `conflict_detected` when the source itself is contradictory.

### Pydantic: a named technology for semantic validation

**Pydantic** is a Python data-validation library, officially mentioned in the exam Appendix. In the context of structured extraction, Pydantic plays a role that `strict: true` JSON Schema cannot fill: it catches **semantic errors** through field-validators and model-validators.

Why Pydantic and not just a strict schema:
- `strict: true` guarantees: correct types, presence of required fields, absence of extra fields — that is, **syntax/structure**.
- Pydantic, through `@model_validator` or `@field_validator`, checks **meaning**: `calculated_total == sum(line_items)`, a start date earlier than an end date, an enum value within the allowed range.

When `model_validate()` fails, Pydantic raises a `ValidationError` with an exact list of per-field violations — and that is exactly what you attach to the retry prompt. So a **validation-retry loop in a Python pipeline** is usually built like this:

1. Claude returns JSON via `tool_use` → parse it.
2. `MySchema.model_validate(data)` → if `ValidationError` → appended retry.
3. On the retry: the original document + the failed extraction + the specific error messages from `ValidationError.errors()`.
4. If the field is simply **not in the source** → there will be a `ValidationError`, but retry is powerless → escalate.

Principle: Pydantic as a validation layer **does not replace** the strict schema — their roles are different: strict removes syntax, Pydantic catches semantics.

### Feedback-loop design: `detected_pattern` for analyzing dismissals

When a system produces findings (for example, a static code analyzer flags potential problems), developers dismiss some of them as false positives. To understand *why*, each finding gets a **`detected_pattern`** field attached — which exact code construct triggered the firing. This gives a systematic breakdown: you can see which patterns produce a lot of dismissals, and the loop can be tuned. Without `detected_pattern` you have no mechanism to analyze the causes of dismissal — only raw noise.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| **Retry on missing data** | It is tempting to "try again," but if the field is not in the source, this is not a fixable error. The retry wastes attempts; you need to escalate / look it up externally. |
| **Generic retry without a specific error** | Repeating the same prompt without attaching a *specific* error is not feedback. The model has nothing to build on, and it will reproduce the same error. Repetition ≠ feedback. |
| **Relying on tool use / strict schema for semantic errors** | Strict schema / structured outputs remove only syntax (valid JSON, types). A well-typed but semantically wrong value (`stated_total` that doesn't add up) is missed by the schema. |
| **No `detected_pattern` for dismissal analysis** | Without a field that records the construct that produced a finding, you cannot systematically break down why findings get dismissed. "Just log everything" gives no link between a cause and a pattern. |

## 🔑 Remember

- An effective retry = **original document + failed extraction + specific validation errors**, added to the prompt.
- Retry fixes format / structural / **semantic**; it does **not** fix *absent information* → escalate to a human or external lookup.
- `strict: true` / structured outputs (constrained decoding) → a guarantee of **schema syntax**; semantic validation stays in your loop.
- Self-correction patterns: `calculated_total` vs `stated_total` for mismatches; `conflict_detected` (boolean) for a contradictory source.
- `detected_pattern` on a finding → systematic analysis of false positives / dismissals.
- **Pydantic** (`model_validate()` + validators) catches semantic errors that `strict: true` does not see; `ValidationError.errors()` is ready-made feedback for the retry prompt.

**Q1 · easy**
An extraction pipeline returns a schema-valid object, but validation flags that `total_amount` does not equal the sum of the line items. You retry. What is the single most important thing to include in the retry prompt?

A) Just resend the same prompt — one of the retries will eventually produce a consistent total.
B) The original document, the failed extraction, and the specific validation error describing the mismatch.
C) A stricter JSON schema with `additionalProperties: false` so the model cannot make the mistake again.
D) A higher `temperature` so the model explores different totals.

<details><summary>Show answer</summary>

**Correct answer: B.** An effective retry is a targeted correction: the model has to see the source, its own failed output, and the *specific* error so it can self-correct.
A — *generic retry without feedback*: repetition is not feedback, the model will reproduce the same error. C — *relying on a strict schema for a semantic error*: the schema closes off syntax, but a total that doesn't add up is a semantic problem the schema does not catch. D — raising `temperature` adds randomness, not a targeted signal.
</details>

**Q2 · medium**
An invoice extractor produces output where `stated_total` does not match the line-item sum, AND the required `tax_rate` field was never printed on the invoice. Both issues were flagged by semantic validation. What is the best next step?

A) Retry up to 5 times with the original document and the specific errors, expecting the retry to also recover the missing `tax_rate`.
B) Tighten the schema and rely on `strict` tool use to fix both the total and the missing `tax_rate`.
C) Retry with the document + failed extraction + specific errors to fix the total; treat the absent `tax_rate` as non-retryable and route it for human input or external lookup.
D) Skip the retry entirely and re-extract from scratch with a fresh prompt to avoid biasing the model.

<details><summary>Show answer</summary>

**Correct answer: C.** The `stated_total` mismatch is a semantic error, fixable with a targeted retry. The absence of `tax_rate` in the source is a different class of failure: it cannot be "recovered" by a retry, it gets escalated / fetched externally.
A — *retry on missing data*: a retry will not invent a rate that is not in the document — wasted attempts. B — *strict schema for semantic + absent*: the schema removes only syntax, not a total that doesn't add up and not a missing value. D — a retry with attached errors is stronger than a "clean" re-extraction; and for a missing field a fresh prompt is just as powerless.
</details>

**Q3 · hard**
A code-review agent flags potential bugs, but developers dismiss roughly 40% of findings as false positives, and the team has no way to know which kinds of code trigger the bad findings. Which design change most directly enables systematic analysis of why findings get dismissed?

A) Add a `detected_pattern` field to each finding recording the code construct that triggered it, so dismissals can be aggregated by pattern.
B) Lower the model `temperature` so the agent produces fewer findings overall.
C) Force the agent to retry every dismissed finding with the specific dismissal reason appended.
D) Switch the output to `strict` tool use so every finding conforms to the schema.

<details><summary>Show answer</summary>

**Correct answer: A.** To systematically break down the causes of dismissal, you need each finding linked to the construct that produced it — that is `detected_pattern`. Then dismissals can be aggregated by pattern and the loop can be tuned.
B — lowering `temperature` reduces noise but gives no *cause analysis*. C — *generic retry aimed at the wrong target*: a dismissal is not a validation error, it is a signal for pattern analysis, not a reason to spin a retry loop. D — *relying on a strict schema*: the schema gives syntax validity but does not explain why a finding was dismissed.
</details>

**Q4 · hard**
A structured-extraction service uses `strict: true` tool use, so every response is guaranteed valid JSON matching the schema. The team reports that downstream systems still occasionally receive wrong data: amounts that don't add up and a customer name placed in the `address` field. What is the root cause, and the right fix?

A) The schema is too loose; add more `required` fields and `additionalProperties: false` to eliminate these errors.
B) `strict` tool use guarantees only syntactic/structural conformance, not semantic correctness; add a semantic validation layer (e.g., `calculated_total` vs `stated_total`, field-placement checks) with retry-on-error feedback.
C) The model `temperature` is too high; lower it so the JSON values become correct.
D) Retry every extraction twice by default; the second pass will correct the misplaced fields.

<details><summary>Show answer</summary>

**Correct answer: B.** Constrained decoding (`strict: true`) guarantees valid JSON and type conformance — that is, syntax/structure. Sums that don't add up and a value in the wrong field are **semantic** errors that the schema fundamentally cannot see. They are cured by a separate semantic-validation layer plus a retry with a specific error.
A — *relying on the schema*: an even stricter schema still does not catch a well-typed but semantically wrong value. C — `temperature` does not make values semantically correct. D — *generic retry by default* without an attached specific error is not a targeted correction; and it does not answer the question about the *root cause*.
</details>

**Q★ · hard** — A Python data extraction pipeline uses `tool_use` with `strict: true` JSON schema. After switching on strict mode, JSON syntax errors disappear. However, the pipeline still occasionally produces invoices where `line_items` amounts don't sum to `total`. A teammate suggests: "Strict mode guarantees the data is correct — something else must be wrong." What is the accurate assessment and the right next step?

A) The teammate is right: strict mode guarantees correctness, so the discrepancy must be a bug in the downstream display layer.
B) Strict mode guarantees schema structure (types, required fields), not semantic correctness; add a `@model_validator` in Pydantic that checks the sum, then use `ValidationError` output as retry feedback.
C) Raise the `max_tokens` limit so the model has more room to compute the total correctly.
D) Add a few-shot example showing the correct total — few-shot is more effective than Pydantic for numeric consistency.

<details><summary>Show answer</summary>

**Correct answer: B.** Strict schema / structured outputs remove syntactic/type errors (invalid JSON, a missing required field), but **do not check semantics**: a `stated_total` that is well-typed but not equal to the sum of the lines is not a schema problem. Pydantic with `@model_validator` catches exactly this kind of semantics, and `ValidationError.errors()` provides a specific signal for a targeted retry.
A — the trap "strict schema → the data is correct": this is a common misconception (topic 4.3). C — `max_tokens` is about output length, not the accuracy of computations. D — few-shot will raise the probability of a correct result, but it gives no deterministic check and no specific signal for retry; Pydantic validation is the right mechanism.
</details>

### Tricky question **

**Q★★ (tricky):** A document-extraction agent runs `strict` tool use plus a semantic validator. On a batch of contracts, the validator flags a finding: for one contract `effective_date` is *after* `expiration_date`. Engineers add a retry loop that appends the specific error ("effective_date 2027-01-01 is after expiration_date 2026-01-01") plus the original contract and the failed extraction. After three retries the model keeps returning the same inverted dates. What is the most likely root cause and correct response?

A) The retry prompt is missing the specific error, so add the exact validation message — then it will self-correct.
B) `strict` schema is failing; tighten the schema to enforce a date ordering constraint and the problem disappears.
C) The source contract itself states the dates in that inverted order, so this is a `conflict_detected` case in the source data — flag it (e.g., set `conflict_detected: true`) and escalate, rather than looping; retry cannot fix a contradiction that is faithfully extracted from the document.
D) Raise the retry limit to 10; the model will eventually produce a valid ordering.

<details><summary>Show answer</summary>

**Correct answer: C.** Two principles are stacked together here. First, the retry-with-error-feedback is *already done correctly* (the document, the failed extraction, and a specific error are attached) — so answer A is wrong, there is no defect in the feedback. Second, the model stubbornly returns the "inverted" dates because it is *faithfully* extracting what the contract says — the contradiction lies in the source itself, not in the extraction. This is close to the class "information is missing/unusable": retry cannot fix a faithfully extracted source contradiction. The right design is to flag it as `conflict_detected: true` and escalate for review.
B — *hoping a schema constraint will "fix" semantics*: even if the schema could impose date ordering (and ordering is a semantic, not a syntax constraint), it would only force the model to distort faithfully extracted data, hiding the real source contradiction. D — *retry on an unfixable situation*: more attempts on a source contradiction are the same wasted cycles.
</details>
