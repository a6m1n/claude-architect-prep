# 4.3 — Enforce structured output using tool use and JSON schemas

> **Domain 4: Prompt Engineering & Structured Output** (20%) · **Task Statement 4.3** | **Scenarios:** Structured Data Extraction | **What it tests:** the ability to guarantee schema-compliant output via `tool_use` + JSON schema and to understand that a schema removes syntactic errors but not semantic ones.

## ⚡ TL;DR

- The most reliable way to get structured output is to define a tool with a JSON `input_schema` (or use Structured Outputs / `strict: true`) and read the data from the `tool_use` block, and **not** to ask "return JSON" in free text.
- `tool_use` + strict schema remove **syntactic** errors (broken JSON, unclosed brackets), but **NOT** semantic ones (line items don't add up to the total, a value landed in the wrong field). Validation downstream in the pipeline is still required.
- `tool_choice`: `auto` (the model may answer with text instead of a call), `any` (must call some tool, the model picks which), `{"type":"tool","name":"extract_metadata"}` (must call exactly this one), `none` (forbidden).
- Make fields `optional`/nullable if the data may be absent from the document — otherwise `required` forces the model to **fabricate** a value. For extensible categories, use `enum` + an `other` value with a detail string, plus `unclear` for ambiguous cases.
- Strict schema + format-normalization rules in the prompt work together: the schema sets the structure, the prompt sets the format of values (dates, currency).

## 🧠 In plain words

> **Analogy:** 📋 a standard form with ready-made boxes: it guarantees that all cells are filled out in the right format, but it does not guarantee that what's inside the cells is true.
>
> Declare a tool with a JSON `input_schema` (or turn on `strict: true`) and read the data from the `tool_use` block — this removes **syntax** issues (broken JSON, unclosed brackets). But the schema does not check meaning: sums may not add up, a value may land in the wrong box. That's why downstream validation (recomputing, cross-checking) is required. Make fields that may be absent from the document optional/nullable — otherwise `required` forces the model to **make up** a value.
>
> 🚩 **Detector:** if the answer says "strict schema → the data is definitely correct" or "let's make all fields `required` for completeness" — that's a **common-misconception** / **one-size-fits-all (extreme)**; the right view is that the schema catches form only, meaning is checked by validation downstream in the pipeline, and absent-able fields are made nullable.
>
> 🧷 **Reflex:** A form holds the shape, not the truth — fields nullable, check the meaning yourself.

## 📘 Theory

### Why `tool_use` and not "return JSON as text"

When you ask the model to return JSON right in the response text, you get a string that has to be parsed with a regex or `JSON.parse()`. Some responses will have broken syntax — and the pipeline crashes. The principle here is **matching workload to mechanism**: the task "guaranteed structure" matches the tool use mechanism, not coaxing in the prompt. You declare a tool with an `input_schema` (JSON Schema), and the API returns a structured `tool_use` block with the `input` already parsed. With `strict: true` (Structured Outputs), schema conformance is guaranteed via constrained decoding: "No more `JSON.parse()` errors".

### Syntax vs semantics — the key distinction of the exam

A strict schema guarantees: valid JSON, correct field types, the presence of `required` fields. What it does **not** guarantee is that the numbers make sense. The model may return a perfectly valid object where `line_items` don't add up to `total`, or where `tax_id` ended up in the `vendor` field. These are **semantic** errors, and the schema does not catch them. Hence the rule: the schema removes the class of syntactic errors, but downstream validation (recomputing sums, range checks) remains mandatory. This is the classic **root-cause-over-symptom**: if you care about the correctness of the numbers, the schema is not the right medicine.

#### What strict mode does not support

It's important to know the limitations: `strict: true` in Claude's JSON Schema (`input_schema`) does not support all of JSON Schema Draft 7. Specifically, the following are **not supported**:
- Numeric constraints: `minimum`, `maximum`, `exclusiveMinimum`, `exclusiveMaximum`
- String length constraints: `minLength`, `maxLength`
- Patterns: `pattern` (regex constraints on strings)
- Array size: `minItems`, `maxItems`
- Recursive and external references: `$ref` (except simple inline definitions)

The consequence for the exam: if the schema contains `"price": {"type": "number", "minimum": 0}`, the `minimum: 0` constraint with `strict: true` will be **ignored** — the schema will accept negative numbers without error. Semantic and constraint validation is done by a separate validation layer (Pydantic, custom checks).

### `tool_choice`: four modes

Verbatim from the docs, when `tools` are set:

- `auto` — the model **itself** decides whether to call a tool or answer with text (the default value when `tools` are set).
- `any` — the model **must** call one of the tools, but which one — it chooses itself. Use this when the document type/schema are not known in advance.
- `{"type":"tool","name":"..."}` — the model must call the **specific** named tool. Use this to guarantee that extraction is launched (for example, before an enrichment step).
- `none` — tools are forbidden.

The **least privilege / determinism** principle: if business logic requires extraction to definitely happen, `auto` is a mistake (the model may "chat" with text). If you need a guaranteed call — `any` or forced. With `any`/`tool` the API prefills the assistant, so there will be no natural-language preamble before `tool_use`.

### Schema design: required, nullable, enum

- **required vs optional.** A field in `required` the model must fill. If it's not in the source, the model **will invent** a plausible value to satisfy the contract. The solution is to make such fields optional/nullable, so that `null` is a legitimate answer. This prevents fabrication.
- **enum + other.** For categories, use `enum`. To make categories extensible, add an `other` value plus a free-text detail field, and `unclear` — for ambiguous cases. This way the model doesn't force a label that doesn't fit.
- **Format normalization.** The schema sets the *structure*, but not the *format* of values. Rules like "dates in `YYYY-MM-DD`, sums without a currency symbol" go in the prompt next to the strict schema — they complement each other.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| "Strict schema guarantees data correctness too" | The schema catches only **syntax** and types. It does not check semantics (sums, swapped fields) — a separate validation is needed. The most common trap of the topic. |
| "Let's make all fields `required` for completeness" | `required` for a field that may be absent from the document **forces the model to fabricate** a value. Absent-able fields should be optional/nullable. |
| "`tool_choice: auto` — since the tool exists, it will be called" | With `auto` the model is free to answer with **text** instead of a call. If you need a guaranteed call, use `any` (any tool) or forced `{"type":"tool","name":...}`. |
| "Let's ask for JSON in the system prompt + tighten the regex" | This is a nudge, not a guarantee: broken JSON is still possible, and parsing a string with a regex is exactly the fragile path that `tool_use` replaces. It doesn't cure fabrication either. |
| "`temperature: 0` will make the JSON always valid" | Temperature reduces variance, but **does not set the grammar** of the output. Broken JSON remains possible, and a deterministic model with a `required` field will deterministically fabricate a missing value. |
| **"`minimum: 0` in the schema won't let through negative prices"** | With `strict: true`, numeric/string/array constraints (`minimum`, `maxLength`, `pattern`) are not enforced — this is a feature of Claude's strict mode implementation. Only `type`, required/optional, enum, nullable, and the basic structure are guaranteed. |

## 🔑 Remember

- Structured output → a tool with a JSON `input_schema`, read from the `tool_use` block; for a guarantee — `strict: true` / Structured Outputs (`output_format`).
- `tool_use`/strict schema remove **syntax**, not **semantics**. Downstream validation is mandatory.
- `tool_choice`: `auto` (may use text) · `any` (must use some tool) · `{"type":"tool","name":"X"}` (must use exactly X) · `none` (forbidden).
- Absent-able fields → optional/nullable, so that `null` is returned rather than fabricated.
- Extensible categories → `enum` + `other` (+ detail string) + `unclear`; format of values → normalization rules in the prompt.
- Tool `name`: regex `^[a-zA-Z0-9_-]{1,64}$`.
- `strict: true` guarantees: types, required fields, enum values, nullable. It does **not** guarantee: `minimum`/`maximum`, `minLength`/`maxLength`, `pattern`, `minItems`/`maxItems` — these are the semantic/constraint level.

## 📝 Practice

*Answer it yourself first, then expand "Answer".*

**Q1 · easy**
A data pipeline asks Claude, in the prompt, to "return a JSON object with the extracted fields." About 1 in 25 responses fails `JSON.parse()` and crashes the consumer. What is the most reliable fix?

A) Add "Respond ONLY with valid JSON, no prose" to the system prompt and harden the parser.
B) Define an extraction tool with a JSON `input_schema` and read the result from the `tool_use` block.
C) Set `temperature` to 0 so the JSON is always well-formed.
D) Wrap the call in a retry loop that re-prompts whenever parsing throws.

<details><summary>Show answer</summary>

**Correct answer: B.** Routing through a tool with an `input_schema` forces the API to return a structured `tool_use` block — there's no free JSON string that can be corrupted, the class of syntactic errors disappears (with `strict: true` this is a guarantee). A — "ask for JSON in free text" + regex: a nudge, not a guarantee, broken JSON remains. C — `temperature: 0` reduces variance, but doesn't set the grammar. D — a retry loop is a symptomatic band-aid: latency and cost grow, convergence isn't guaranteed.
</details>

**Q2 · medium**
You extract metadata from invoices. Some invoices have no `tax_id`. With `tax_id` marked `required`, the model invents plausible-looking IDs rather than omitting them. What is the best schema change?

A) Keep `tax_id` `required` but add "leave blank if missing" to the prompt.
B) Lower `temperature` to 0 to stop the fabrication.
C) Make `tax_id` optional/nullable so the model can legitimately return `null`.
D) Add a post-processing regex that deletes fabricated tax IDs.

<details><summary>Show answer</summary>

**Correct answer: C.** `required` creates pressure to fill the field at any cost — hence the fabrication. Optional/nullable makes `null` a legitimate answer, and there's no need to make anything up (this is the root cause). A — `required` is still in force, the prompt instruction conflicts with the schema contract; the model more often follows the schema. B — determinism doesn't cure fabrication: the model will *deterministically* make up an ID. D — you can't tell a made-up valid ID from a real one with a regex; this is a symptomatic patch.
</details>

**Q3 · hard**
An extraction service must classify each document into one of several known categories, but new category types appear over time and some documents are genuinely ambiguous. The team also needs every request to actually produce an extraction (never a chat reply). Which design is best?

A) A free-text `category` string field with `tool_choice: auto`.
B) An `enum` `category` field plus `tool_choice: auto`, retrying when the model replies with text.
C) An `enum` `category` with an `other` value (+ a detail string) and an `unclear` value, invoked with forced `tool_choice` on the extraction tool.
D) A `required` `enum` with only the known categories, invoked with `tool_choice: any`.

<details><summary>Show answer</summary>

**Correct answer: C.** `enum` + `other` (with a detail string) gives extensibility for new types, `unclear` correctly handles ambiguous documents, and **forced** `tool_choice` (`{"type":"tool","name":...}`) guarantees that the needed extraction tool will definitely be called. A — free-text loses the advantage of enum, and with `auto` the model may answer with text. B — `auto` + retry: extra latency and no guarantee of a call, plus a rigid enum without `other`/`unclear`. D — an enum without `other`/`unclear` will force a new/ambiguous document into an existing category (forcing a label); `any` guarantees a call to *some* tool, but doesn't cover the gap in the enum design.
</details>

**Q4 · hard**
After switching to tool use with a strict JSON schema, JSON parse errors are gone. But auditors find invoices where the sum of `line_items` does not equal `total`, and occasionally the `vendor_name` value is actually the customer's name. What is the root cause and correct response?

A) The schema is not strict enough; add more `required` fields to force completeness.
B) Strict schemas only prevent syntax/type errors; semantic correctness needs downstream validation (recompute totals, cross-check fields).
C) `tool_choice` was set to `any` instead of forced; switch to the named tool.
D) `temperature` is too high; set it to 0 to make the extraction accurate.

<details><summary>Show answer</summary>

**Correct answer: B.** This is the heart of the topic: a strict schema guarantees valid JSON and types, but **not** semantics — sums that don't add up and values in the wrong fields are not caught by the schema. The cure is validation downstream in the pipeline (recomputing `total`, cross-checking fields). A — more `required` fields will only increase fabrication, not correctness. C — `tool_choice` affects the *fact and choice* of a call, not the semantic correctness of values. D — `temperature: 0` will reduce variance, but swapped fields and sums that don't add up are not about variance.
</details>

### Tricky question **

**Q★★ (tricky):** A team needs guaranteed schema-compliant extraction. They enable strict tool use and want a hard guarantee that a tool is always called *and* that its inputs match the schema, while still letting the model pick among several extractor tools for different document types. They also want Claude to emit a short natural-language note explaining its choice before the tool call. Which configuration meets ALL stated requirements?

A) `tool_choice: {"type":"any"}` with `strict: true` on the tools.
B) `tool_choice: {"type":"tool","name":"extract"}` with `strict: true`.
C) `tool_choice: {"type":"auto"}` with `strict: true`, plus a prompt asking it to explain its choice.
D) `tool_choice: {"type":"any"}` with `strict: true`, plus a system instruction to add a preamble.

<details><summary>Show answer</summary>

**Correct answer: C.** Here two requirements collide, and the trap is to miss that with `any`/`tool` the API **prefills** the assistant message, so the model **will not produce** a natural-language preamble before `tool_use`, even if you ask for it directly. That means options A, B, D cannot simultaneously give both the guarantee and the explanatory text. The documentation directly recommends: if you need natural-language context together with a call — use `tool_choice: {"type":"auto"}` (the default) and an explicit instruction to call the tool. `strict: true` still gives schema conformance of the inputs. The subtlety: a "hard guarantee that the tool will be called" with `auto` is strictly speaking not met — but that's exactly the catch of the question: of the four options, only C satisfies *all the listed* requirements, including the preamble (which A/B/D physically won't give). D is especially tempting: `any` + a preamble instruction looks like "both a guarantee and text," but the assistant prefill suppresses that preamble — which is exactly why the trap fails.
</details>
