# 4.1 — Design prompts with explicit criteria to improve precision and reduce false positives

> **Domain 4: Prompt Engineering & Structured Output** (20%) · **Task Statement 4.1** | **Scenarios:** Claude Code for Continuous Integration, Structured Data Extraction | **What it tests:** the ability to replace a vague instruction with a precise operational criterion and manage false positives at the category level, so that trust is not destroyed.

## ⚡ TL;DR

- **An explicit categorical criterion beats a vague instruction.** `flag a comment only when its claimed behavior contradicts actual code behavior` is far better than `check that comments are accurate`.
- General props like `be conservative` or `only report high-confidence findings` **do not raise precision** — they do not define the decision boundary, so the model still improvises.
- **A precise criterion generalizes** to new cases; few-shot examples only pattern-match known forms and are needed for *format*, not for *defining the rule*.
- **False positives are a trust multiplier:** a noisy category teaches developers to ignore ALL findings, including accurate ones. The quality of mixed output converges to its worst component.
- **The cure for a noisy category is to temporarily turn it off** (quarantine), keep the high-precision categories, fix the prompt offline, then bring it back. Not uniform tightening, and not "leave it on while we fix it."

## 🧠 In plain words

> **Analogy:** 🛡️ a guard at the door. Give them 3 photos of "don't let these people in" (`few-shot`) — a fourth person who isn't in the photos shows up, and the guard guesses. Give them a rule "don't let anyone in born after 2007" (a criterion) — it works for anyone, no photos needed.
>
> A vague instruction ("check that comments are accurate") forces the model to improvise. An explicit criterion ("flag **ONLY IF** the claimed behavior contradicts the code") is a checkable yes/no rule that sorts even unseen cases on its own. And a noisy category with lots of false positives teaches developers to ignore **ALL** findings — so you temporarily turn it off (quarantine) rather than "tighten it a little."
>
> 🚩 **Detector:** the answer says `be conservative` / `only high-confidence` / "raise the confidence threshold" instead of a checkable condition → that's vagueness (lever 1–2 instead of 3); the right move is an explicit categorical criterion. A noisy category → quarantine, not uniform tightening.
>
> 🧷 **Reflex:** not "be more careful," but "flag ONLY IF [checkable condition]"; the two-strangers test — would two people, without conferring, assign the same label? then it's a criterion, not a vibe.

## 📘 Theory

### Why an explicit criterion, not a "vibe"

The root of inconsistent model output is an **underspecified instruction**. When the prompt says `check that comments are accurate and up-to-date`, the decision boundary is not set: the model decides for itself what counts as a problem, and so it flags harmless patterns (false positives) and misses comments that are genuinely out of date (false negatives).

The architectural fix is **root-cause-over-symptom**: replace the directive with a precise, checkable criterion that describes *what counts as a problem*, not *what the problem looks like*. The model applies the criterion `flag iff claimed behavior contradicts actual behavior` uniformly and **generalizes to cases the author did not foresee**. This is a direct consequence of the "be clear and direct" principle: the more precise the instruction, the less improvisation.

### Why general props don't work

`be conservative`, `only report high-confidence findings`, `use a confidence threshold` — these sound like raising precision, but they **don't give an operational definition**. "Conservative" relative to what? "High-confidence" by what criterion? The model still invents the boundary on its own, so the false-positive metric doesn't improve systematically — it just shifts a little at random.

### Few-shot — for format, not instead of the rule

Concrete examples for each level (severity, category) are powerful **few-shot anchors** that fix the boundaries between classes better than abstract definitions. But on their own they **pattern-match** forms already seen and transfer poorly to new types of contradictions. The right order is: **first the explicit criterion, then examples** to refine the format and edge cases.

### False positives as a trust multiplier

The output of a reviewer agent is consumed by a human with finite attention. A category with a 52% false-positive rate doesn't just waste time on its own findings — it **teaches developers to discard all findings** ("half are wrong"), including security/correctness at 8% FP. So precision is a property you manage **at the category level**: noisy sources are isolated (quarantine), not adjusted with one global strictness knob.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| **"Be conservative" / confidence threshold** instead of an explicit criterion | Doesn't set the decision boundary. "Conservatively/confidently" is a vibe, not an operational definition; precision doesn't rise systematically. |
| **Vague `check that X is correct`** without an operational definition | This is exactly the root of the problem: the model decides for itself what "correct" means and diverges from run to run. |
| **Leave the noisy category enabled while you fix it** ("over the coming weeks") | The correct long-term fix (improve the prompt), but it ignores *urgency*: trust erodes the whole time, and developers are already ignoring everything. |
| **Few-shot examples instead of a criterion** | Examples are instances of the problem, not the rule; they don't generalize to unseen types of contradictions. Good as a supplement to a criterion, not as a replacement. |
| **Uniform tightening across all categories** | Drags the already-accurate categories (8% security) down for the sake of the bad ones, suppressing valuable true positives. Hits the wrong category. |
| **Confidence score on each finding** | Shifts the precision problem onto the human; developers who already ignore findings won't start triaging by score. The noise isn't removed. |
| **Git blame / metadata for "out-of-date" comments** | recency ≠ inaccuracy: an old comment can be correct, a new one incorrect. Works around the missing criterion rather than setting it. |

## 🔑 Remember

- **Rule:** for inconsistent output — **define the decision boundary**, don't add props around a vague instruction.
- Criterion formula: `flag iff <claimed behavior> contradicts <actual behavior>` — checkable, generalizes.
- **Severity consistency** = explicit per-level criteria **+ concrete code examples** in the prompt (not human post-calibration, not a relative rating within the PR, not a default mapping in `CLAUDE.md`).
- **High-FP category** → temporarily **disable** (quarantine), keep the high-precision ones, fix the prompt offline, bring it back after proven accuracy.
- Fix categories **selectively**, not with global tightening.
- `be conservative`, `only report high-confidence findings`, confidence score — do **not** raise precision.

## 📝 Practice

*Answer for yourself first, then expand "Answer".*

**Q1 · easy**
Your CI review prompt says: `check that comments are accurate`. It flags harmless `TODO` markers and misses comments describing behavior the code no longer implements. What change best addresses the root cause?

A) Add `be conservative and only report high-confidence findings` to the prompt.
B) Replace the instruction with: `flag a comment only when its claimed behavior contradicts the actual code behavior`.
C) Pre-filter `TODO` and `FIXME` comments before analysis to reduce noise.
D) Include `git blame` data so Claude can flag comments that predate recent code changes.

<details><summary>Show answer</summary>

**Correct answer: B.** Replaces the underspecified instruction with an **explicit checkable criterion** (claimed vs actual behavior) that fixes both false positives and false negatives and generalizes to new cases.
- A — the *"be conservative"* trap: a vibe with no operational definition, doesn't set the decision boundary.
- C — *pre-filter/metadata*: suppresses noise without defining correctness; may hide real problems.
- D — *git blame*: recency ≠ inaccuracy, works around the missing criterion.
</details>

**Q2 · medium**
Your automated review assigns severity inconsistently: an identical null-pointer risk is `critical` in one PR but `medium` in another. Developers can't predict which findings matter. What is the most effective fix?

A) Ask Claude to emit its reasoning for each rating, then have reviewers manually calibrate ratings during review.
B) Tell Claude to rate severity relative to other issues in the same PR, so the worst issue is always `critical`.
C) Add explicit severity criteria to the prompt with concrete code examples for each level.
D) Add a `CLAUDE.md` file listing issue types and their default severities to reference.

<details><summary>Show answer</summary>

**Correct answer: C.** Classification consistency is an **input-design** matter: explicit criteria for each level + concrete code examples as few-shot anchors remove the ambiguity at the source and scale without a human.
- A — *human post-hoc calibration*: doesn't scale, the prompt's ambiguity remains.
- B — *relative rating within the PR*: stable within a PR, but the same issue becomes `critical` in one PR and `medium` in another — cross-PR inconsistency.
- D — *default mapping in `CLAUDE.md`*: useful context, but without explicit in-prompt criteria and examples the edge cases stay vague; weaker than criteria in the prompt that does the classifying.
</details>

**Q3 · hard**
Per-category false-positive rates: security/correctness 8%, performance 18%, style/naming 52%, documentation 48%. Developers now dismiss findings unreviewed because "half are wrong," undermining trust even in the accurate categories. What best restores trust while you improve the system?

A) Keep all categories but show a confidence score with each finding so developers decide what to investigate.
B) Keep all categories enabled and add few-shot examples to improve each one over the coming weeks.
C) Apply a uniform strictness reduction across all categories to bring overall FP rate down.
D) Temporarily disable the high-FP categories (style, naming, documentation), run only high-precision ones, and improve their prompts offline before re-enabling.

<details><summary>Show answer</summary>

**Correct answer: D.** False positives are a trust multiplier; quarantining the noisy categories **immediately** stops the erosion of trust while preserving the security/correctness signal as the prompts are fixed offline.
- A — *confidence score*: shifts precision onto the human; the noise isn't removed, and people who already ignore findings won't start triaging.
- B — *leave it enabled while we fix it*: the correct long-term fix, but ignores urgency — trust erodes the whole time.
- C — *uniform tightening*: drags the accurate 8% categories down, suppressing valuable true positives for the sake of the bad categories.
</details>

### Tricky question **

**Q★★ (tricky):** A team in the Structured Data Extraction scenario extracts `contract_type` from PDFs. The prompt says `extract the contract type accurately`. Output is inconsistent: ambiguous documents get arbitrary labels and trust is falling. An architect proposes: "Add three high-quality few-shot examples of correctly labeled contracts to the prompt — examples are a proven Anthropic-recommended technique for consistency." Is this the *best* root-cause fix, and if not, what is?

A) Yes — few-shot examples are the canonical way to make extraction consistent, so this is the correct root-cause fix.
B) No — first define explicit categorical criteria for each `contract_type` (and what disqualifies each); add few-shot examples only to anchor format and edge cases once the criteria exist.
C) No — instead instruct the model to `be conservative and only assign a label when highly confident`, leaving ambiguous documents unlabeled.
D) No — instead lower `temperature` to 0 so identical inputs always produce the same label.

<details><summary>Show answer</summary>

**Correct answer: B.** The trap is subtle: few-shot **really is** a recommended technique, and option A sounds "correct by the docs." But the root of the problem here is the **absence of a rule definition** (`accurately` is underspecified), and examples only pattern-match the forms shown and don't set the boundary between categories for unseen, ambiguous documents. First come **explicit categorical criteria** (what each `contract_type` is and what disqualifies it), and only then few-shot as format/edge-case anchors. The order "criterion → examples" is the key.
- A — *few-shot instead of a rule*: the technique is real, but applied at the wrong layer; it doesn't define the decision boundary, and on new ambiguous inputs it will diverge again. Precisely because it sounds "by the docs," it's the most dangerous trap.
- C — *"be conservative / high confidence"*: a vibe with no operational definition; "highly confident" by what criterion? precision doesn't rise systematically, and some documents simply end up without a label.
- D — `temperature=0`: gives **determinism** (same input → same output), but not **correctness** — the same *wrong* label on an ambiguous document. Fixes reproducibility, not the decision boundary.
</details>
