# 4.6 — Design multi-instance and multi-pass review architectures

> **Domain 4: Prompt Engineering & Structured Output** (20%) · **Task Statement 4.6** | **Scenarios:** Claude Code for Continuous Integration (CI) | **What it tests:** an independent review instance is stronger than self-review; splitting a large review into per-file passes + a separate cross-file integration pass against attention dilution.

## ⚡ TL;DR
- **Independent review instance > self-review.** A model that just generated (or analyzed) the code carries its reasoning context with it and tends to "rubber-stamp" its own decisions. A fresh instance without that context catches subtle bugs better than an instruction to "check yourself" or extended thinking.
- **Not one pass over 14 files, but per-file passes.** One pass over many files = **attention dilution**: uneven depth, missed obvious bugs, contradictory conclusions (a pattern flagged in one file, approved in another).
- **Per-file local pass + a separate cross-file integration pass.** The local pass gives even depth on every file; the integration pass does what per-file cannot — it traces data flow and interface contracts between files.
- **A larger context window does NOT fix the problem.** The context fits — but the quality of attention distribution over it does not improve. This is an architectural decision (how to slice the work), not a "model knob."
- **Consensus voting (≥2 of N runs) is an anti-pattern for finding bugs.** Real, intermittently-detected bugs often surface in just 1 run out of 3; a "minimum 2 of 3" threshold **suppresses** them (official Sample Q12, option D).

## 🧠 In plain words

> **Analogy:** 👀 the author's eyes have gone blind to the text and read right past their own typos — you need a fresh reviewer who sees it for the first time.
>
> A model that wrote the code itself "remembers" why it decided that way, and subconsciously defends its own train of thought — so it is worse at catching its own bugs. A fresh, independent instance sees only the diff, without those thoughts, and catches more. Don't read a big PR in one go: review it one file at a time (per-file pass), plus a separate pass for the connections between files (cross-file integration pass) — otherwise attention smears out and everything stays shallow.
>
> 🚩 **Detector:** if the answer says "let the same session re-check itself / extended thinking / a larger context window / consensus ≥2 of N runs" — that is `symptom-not-root-cause` (or `common-misconception` about window size); the right answer is an independent instance + per-file passes + a separate cross-file pass, and rare bugs are caught by per-finding confidence, not voting.
>
> 🧷 **Reflex:** fresh eyes instead of self-review; one chapter at a time + a pass for the connections; consensus suppresses the rare bug — don't require it.

## 📘 Theory

### Why self-review is weaker than an independent instance
Within a single session, the model keeps the reasoning context of generation: it "remembers" why it made each decision, and so is less inclined to challenge itself. The instruction "now check your answer" and extended thinking do not remove this bias — they work inside the same context. An **independent review instance** sees only the diff and the criteria, without the chain of reasoning that produced the change, and judges the result "on its own terms." This is the principle of **separation of roles**: whoever does the work should not grade themselves. In Claude Code this is implemented with the Writer/Reviewer pattern (two sessions / fresh context) or a verification subagent that checks the result with a fresh model.

### Why one pass over many files falls apart (attention dilution)
When a single pass has to reason about many files at once, attention **smears out**: depth becomes uneven, obvious bugs slip by, and the same pattern gets flagged in one file and approved in another — contradictory findings. This is the **root cause**, not the symptom "the model is weak." The cure is architectural: slice the work by area so each pass analyzes exactly what it can analyze well.

### Per-file local pass + cross-file integration pass
- **Per-file local pass:** each file is analyzed separately → guaranteed even depth and no contradictions (the model doesn't juggle the context of 14 files while assessing one).
- **Cross-file integration pass:** a separate pass dedicated to the connections — data flow between files, interface contracts, regressions at the seams. A per-file pass cannot see this in principle.
- Generalized rule: **scope each pass to what it can analyze well, then add a separate pass for the relationships between scopes.** Neither pass is sufficient on its own.

### Calibrated review routing
A useful reinforcement: run verification passes where the model **self-reports confidence** alongside each finding. This lets you route the review by calibration — for example, send low-confidence findings for extra verification, post high-confidence ones immediately — instead of drowning real bugs in consensus voting.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| **Self-review: let the generating session check its own output** | The model carries the reasoning context of generation and "rubber-stamps" its own decisions. A self-review instruction and extended thinking do not remove the bias — you need an **independent instance** without the prior context. |
| **One pass over all files at once** | This is exactly **attention dilution** — uneven depth, omissions, contradictory findings. It is the cause of the problem, not the solution. You need to slice into per-file + integration. |
| **Switch to a model with a larger context window** | Classic trap: **context size ≠ attention quality.** All files will fit, but that does not fix the quality of attention distribution over them. The problem is *how* attention is distributed, not *whether* it fits. |
| **Consensus: flag only what surfaced in ≥2 of N independent runs** | Sounds "rigorous," but **suppresses real bugs.** Intermittently-detected bugs (exactly the ones a diluted pass misses) are often visible in 1 run out of 3; the 2-of-3 threshold filters them out, triples the cost, and does not add the cross-file pass that is actually needed. (Sample Q12, D.) |
| **Force developers to split the PR into 3–4 files** | Process band-aid: it shifts the work onto people and changes their workflow, but the review system itself still cannot handle a large PR correctly. It doesn't fix the architecture. |
| **Post-processing filter by lines (file path + finding text)** | Brittle: a string filter cannot tell a genuinely fixed issue from a similar unresolved one. Deduplication is a reasoning task (give the prior findings as context), not string matching. |

## 🔑 Remember
- **Independent review instance > self-review / extended thinking** for catching subtle bugs.
- Symptoms of **uneven depth + contradictory findings** on a multi-file review = **attention dilution** → slice the passes by scope.
- **Per-file local pass** (depth) + **a separate cross-file integration pass** (data flow, contracts). Both are needed.
- **A larger context window does not raise attention quality.** Fits ≠ analyzed well.
- **Consensus ≥2-of-N is NOT a reliability mechanism for finding bugs** (suppresses intermittent bugs, triple the cost).
- In Claude Code: Writer/Reviewer (fresh context improves the review), verification subagent / `/code-review` in a fresh context; managed Code Review runs several agents in parallel + a verification step to filter out false positives.
- Reinforcement: per-finding **confidence** for calibrated review routing.

## 📝 Practice
*Answer on your own first, then expand "Answer."*

**Q1 · easy**
A CI pipeline asks Claude to generate a module and then, in the same session, appends "now carefully review your own code for bugs before finishing." Subtle logic errors still slip through to production. What is the most effective change?

A) Add "think step by step and be extra critical" to the self-review instruction.
B) Run the review in a separate, independent instance that sees only the diff, not the generation reasoning.
C) Switch the generating model to a tier with a larger context window.
D) Re-run the same self-review three times and merge only if all three agree it is clean.

<details><summary>Show answer</summary>

**Correct answer: B.** A model in the same session carries the reasoning context of generation and is less inclined to challenge itself; an independent instance without that context catches subtle bugs better. A — *self-review reinforcement*: extended thinking / "be more critical" work in the same biased context. C — *bigger-context trap*: window size doesn't change the fact that the reviewer and the author are the same session. D — *consensus trap* + still self-review: voting across three self-reviews doesn't remove the bias and suppresses intermittent bugs.
</details>

**Q2 · medium**
A PR touches 12 files. Your single review pass over all of them gives detailed feedback on some files, superficial comments on others, and contradicts itself — flagging a pattern in one file while approving identical code in another. What is the root cause and best fix?

A) The model is too small; move to a higher tier so it has enough capacity for 12 files in one pass.
B) Attention dilutes across many files in one pass; split into per-file local passes plus a separate cross-file integration pass.
C) The PR is too big; require the author to resubmit it as several 3–4 file PRs before review.
D) Findings are noisy; run the full-PR review three times and post only issues appearing in at least two runs.

<details><summary>Show answer</summary>

**Correct answer: B.** Uneven depth + contradictions = classic **attention dilution**; the cure is architectural — per-file passes give even depth, a separate integration pass catches cross-file connections. A — *bigger-model/context trap*: the problem is attention distribution, not "capacity." C — *process band-aid*: it shifts work onto the author, the system still can't handle a large PR. D — *consensus trap*: ≥2-of-3 suppresses intermittent bugs and triples the cost without adding the needed cross-file pass.
</details>

**Q3 · hard**
After adopting per-file passes, cross-file bugs (a function renamed in `auth.py` but still called with the old name in `api.py`) are still missed, because each per-file pass only sees its own file. What should you add?

A) Increase each per-file pass's context to also include all sibling files, so every pass sees the whole PR.
B) Add a separate cross-file integration pass dedicated to tracing data flow and interface contracts across files.
C) Run more per-file passes per file and take a majority vote on each file's findings.
D) Move to a model with a larger context window so all files fit and inter-file links are obvious.

<details><summary>Show answer</summary>

**Correct answer: B.** A per-file pass by design cannot see the connections between files — you need a **separate integration pass** dedicated to cross-file data flow and contracts. A effectively returns to single-pass-over-many-files → reintroduces the attention dilution that per-file was introduced to avoid. C — *consensus trap* at the file level: voting within a single scope adds no cross-file visibility. D — *bigger-context trap*: everything fits, but the quality of attention on the connections does not grow.
</details>

**Q4 · hard**
Your CI review must catch subtle, intermittently-detected bugs while keeping noise manageable. Which design best balances catching real bugs against false positives?

A) Require an issue to appear in at least 2 of 3 independent full-PR runs before it is posted.
B) Use per-file + integration passes in independent instances, and have each pass self-report a confidence score per finding to route low-confidence ones for verification.
C) Post every finding from a single pass and let developers filter the noise via 👍/👎 reactions.
D) Raise the model temperature so reviews are more creative and surface more candidate issues.

<details><summary>Show answer</summary>

**Correct answer: B.** Independent instances + per-file/integration slicing catch subtle bugs, and **per-finding confidence** gives calibrated routing: low-confidence goes for extra verification, rather than being thrown away. A — *consensus trap*: 2-of-3 suppresses exactly the intermittent bugs the requirement is to catch. C shifts filtering onto people and doesn't raise the recall of subtle bugs. D — temperature is not a "reliability tool"; more noise ≠ better calibration.
</details>

### Tricky question **
**Q★★ (tricky):** Your review architecture already uses independent reviewer instances. To cut false positives, a teammate proposes: "run each independent reviewer **three times** and only surface a finding when it shows up in **at least two of the three runs** — independent instances plus consensus voting, best of both." Adopt this?

A) Yes — combining independent instances with 2-of-3 consensus is strictly safer than either alone.
B) No — the consensus threshold suppresses real, intermittently-detected bugs; instead keep independent passes and use per-finding confidence to route uncertain findings to a verification pass.
C) Yes, but only if you also switch to a larger context window so each run sees more.
D) No — independent instances are the actual mistake here; a single self-review run with extended thinking would be cheaper and just as reliable.

<details><summary>Show answer</summary>

**Correct answer: B.** The trap is tempting because "independent instances" is the *right* technique, and it seems like adding consensus on top can only help. But the exact cause of failure is **the ≥2-of-3 threshold itself**: subtle bugs are detected intermittently and often surface in just one run out of three, so the consensus filter **suppresses** exactly them (official Sample Q12, D), while tripling the cost. The right way to cut false positives is **calibrated review routing**: independent passes + per-finding confidence, where uncertain findings go to a separate verification pass rather than being thrown out by voting. A — verbatim consensus trap. C — *bigger-context trap*, unrelated to the threshold problem. D — a reversal: an independent instance is in fact stronger than self-review + extended thinking, so this is wrong for the same reason as Q1.
</details>

## 🔗 Official documentation
- [Building effective agents](https://www.anthropic.com/engineering/building-effective-agents) — the parallelization pattern (sectioning/voting) and the example of "several different prompts review code and flag bugs"; the basis for independent review instances.
- [Best practices for Claude Code](https://code.claude.com/docs/en/best-practices) — the Writer/Reviewer pattern and the adversarial review subagent: fresh context improves the review because the reviewer does not see the reasoning that produced the change (direct justification for independent > self-review).
- [Code Review](https://code.claude.com/docs/en/code-review) — managed review: several specialized agents analyze the diff **in parallel**, then a **verification step** filters out false positives and findings are deduplicated — a real example of a multi-instance + verification architecture.
