# 3.4 — Determine when to use plan mode vs direct execution

> **Domain 3: Claude Code Configuration & Workflows** (20%) · **Task Statement 3.4** | **Scenarios:** Code Generation with Claude Code | **What it tests:** choosing the mode by the *shape* of the task (scale / many files / architecture → plan; small targeted fix → direct), and the role of the Explore subagent for isolating "noisy" reconnaissance.

## ⚡ TL;DR

- Choose the mode by the **shape of the task, not its "difficulty"**: many files / several valid approaches / architectural decisions / unfamiliar code → **plan mode**; something you can describe in a one-line diff → **direct execution**.
- **Plan mode** = safe reconnaissance: Claude reads files and builds a plan, but **does not write edits to disk** until you approve. This prevents expensive rework (`costly rework`) when dependencies surface late.
- Don't "start direct and switch to plan if complexity comes up" — if the complexity is **already stated in the requirements**, you should plan from the start.
- Plan mode for a trivial single-file fix = **overkill** (unnecessary overhead).
- The **Explore subagent** reads many files in *its own* context window and returns only a summary — keeping the main context clean. The combination: plan for research + direct for implementation.

## 📘 Theory

### Principle: matching workload to mechanism

The mode is chosen for the *structure of the work*, not for subjective difficulty. The official best-practices guide states the rule directly: planning is useful when **you are not sure about the approach, when the change touches several files, or when you are unfamiliar with the code being changed**. If the diff can be described in one phrase — skip the plan.

- **Plan mode** — for large, multi-file, architecture-heavy tasks with several valid paths: monolith → microservices, migrating a library across 45+ files, choosing service boundaries and module dependencies.
- **Direct execution** — for small, clearly scoped edits: a single-file bugfix from a clear stack trace, adding one validation, a log line, renaming a variable.

### Why plan mode is safe (determinism-vs-rework)

In plan mode Claude **explores but does not edit the source files** until you approve (`claude --permission-mode plan`, or `Shift+Tab` to switch within a session, or the `/plan` prefix for a single prompt). Permission prompts still work as in default mode — but there are no code changes. The point is to separate *research and design* from *execution*, so you don't "solve the wrong problem" and don't hit rework when a hidden dependency shows up in the middle of the edits. When the plan is ready, Claude shows it and offers to: approve and switch to auto / acceptEdits / manual review of each edit, or keep planning. `Ctrl+G` opens the plan in an editor for direct editing before the start.

### Recommended workflow: Explore → Plan → Implement → Commit

1. **Explore** (in plan mode): reads files, answers questions, changes nothing.
2. **Plan** (in plan mode): you ask for a detailed implementation plan.
3. **Implement** (exit plan mode): Claude writes the code, checking against its plan.
4. **Commit**: a meaningful message + PR.

That is, **plan mode for research + direct execution for implementation** — this is not "either-or" but a sequence.

### Explore subagent: isolating noisy reconnaissance

Large reconnaissance across the codebase clogs the main context with dozens of file reads, and performance drops as the window fills. The solution is to delegate the reconnaissance to a **subagent**: it reads files in a *separate* context window and returns **only a summary of the findings**. This way the main context stays clean for implementation. This is a direct application of context engineering: a cheap summary instead of expensive "raw" output.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **Direct execution for architectural / multi-file work** | "Just start writing incrementally, the boundaries will reveal themselves" sounds agile, but it postpones design and risks expensive rework when dependencies surface late in the middle of the edits. An architectural decision requires reconnaissance *before* the first edit. |
| **"Start direct, switch to plan if complexity comes up"** | Sounds pragmatic, but the complexity is **already stated in the requirements** (dozens of files, service boundaries) — it is not a hypothetical contingency but a given. Postponing the mode choice here is a mistake. |
| **Detailed upfront instructions in direct mode** | "Write out the structure of each service in advance" seems rigorous, but it assumes you *already know* the right structure — and it skips the very dependency reconnaissance that the decision requires. |
| **Plan mode for a trivial single-file fix** | For a typo, a log line, or one validation, plan mode is overhead/overkill. If the diff can be described in one phrase — do it directly. |
| **"Plan mode edits the code itself after the plan"** | No: before the plan is approved there are no edits on disk; editing only begins after you choose how to exit plan mode (auto/acceptEdits/manual review). |

## 🔑 Remember

- Start/switch plan mode: `claude --permission-mode plan` · `Shift+Tab` (cycle `default → acceptEdits → plan`) · the `/plan` prefix for a single prompt.
- In plan mode only **reads** are auto-approved; **source files are not edited** until the plan is approved.
- Project default: `defaultMode: "plan"` in `.claude/settings.json` → `permissions`.
- Choice rule: **multi-file / several approaches / architecture / unfamiliar code → plan; one-phrase diff → direct.**
- Delegate reconnaissance to a **subagent** ("use a subagent to investigate X") — a separate context that returns a summary.
- Plan and direct **combine**: plan for research/design, direct for implementation.

## 📝 Practice

*Answer for yourself first, then expand the "Answer."*

**Q1 · easy**
You need to add a single missing null-check in `parseConfig()` after a clear stack trace points to one line in one file. Which approach is most effective?

A) Enter plan mode to explore the whole config subsystem before touching anything
B) Use direct execution and make the one-line fix
C) Delegate the fix to an Explore subagent and apply its summary
D) Start in plan mode, then switch to direct only if the test fails

<details><summary>Show answer</summary>

**Correct answer: B.** A single-file, clearly scoped fix with a clear stack trace is the canonical case for direct execution: the diff can be described in one phrase. A — *plan mode for a trivial fix* (overkill/overhead). C — the Explore subagent is needed for **noisy reconnaissance of a large codebase**, not for a single line; it is unnecessary ceremony. D — switching modes here is pointless: there is no complexity, neither stated nor expected.
</details>

**Q2 · medium**
You're tasked with migrating a logging library across 45+ files, with several valid call-site patterns and decisions about a shared adapter layer. What's the best first step?

A) Begin direct execution and let the right adapter pattern emerge as you edit each file
B) Write exhaustive upfront instructions specifying every call-site change, then run direct execution
C) Enter plan mode to explore call sites and dependencies and produce a reviewable plan before editing
D) Start direct execution and switch to plan mode only if you hit an unexpected conflict

<details><summary>Show answer</summary>

**Correct answer: C.** 45+ files + several valid approaches + a decision about a shared adapter = the shape of task that plan mode was made for: reconnaissance and design *before* any edit, with no rework. A — *direct for multi-file work* (boundaries "will reveal themselves" → expensive rework). B — *detailed upfront instructions* assume the structure is already known and skip dependency reconnaissance. D — *postponed switching*, even though the scale is already stated in the requirements.
</details>

**Q3 · hard**
A monolith-to-microservices restructuring spans dozens of files and hinges on service-boundary decisions. You'll use plan mode. To keep the main conversation from filling up while Claude maps dozens of modules and their dependencies, what should you add?

A) Raise the context window limit so all file reads fit in the main session
B) Delegate the dependency mapping to a subagent that explores in its own context and returns a summary
C) Run the whole exploration in `bypassPermissions` mode to avoid prompts slowing it down
D) Skip exploration and ask Claude to infer boundaries directly from the directory names

<details><summary>Show answer</summary>

**Correct answer: B.** Noisy reconnaissance of dozens of modules clogs the main context; the subagent reads in a *separate* window and returns only the findings — the main context stays clean for planning/implementation. A — you can't "raise the limit" like a button, and even a large window degrades as it fills; this is not a context-engineering solution. C — `bypassPermissions` is about the safety of edits, not about saving context, and it is dangerous here. D — skipping reconnaissance contradicts the very purpose of plan mode and leads to wrong boundaries.
</details>

### Tricky question **

**Q★★ (tricky):** Your team lead says: "It's a big refactor touching ~40 files, so be safe — start the session in plan mode and just let it run end-to-end; plan mode won't change anything you don't approve, so it'll explore, plan, AND implement safely in one go." You're deciding how to actually execute. Which response is correct?

A) Agree: plan mode safely covers exploration, planning, and implementation in one continuous run, since nothing is written without approval
B) Use plan mode for exploration and planning, then exit plan mode (approve into acceptEdits or manual review) for the implementation phase
C) Skip plan mode entirely and use `acceptEdits` from the start, since the lead wants it to "just run"
D) Stay in plan mode for the entire task and rely on `--dangerously-skip-permissions` to let it write the files at the end

<details><summary>Show answer</summary>

**Correct answer: B.** The subtlety is that the lead's claim sounds plausible ("plan mode changes nothing without approval") — and it is *true*, but that is exactly why plan mode **does not implement**: while you are in plan mode, the source files are not edited at all. The recommended workflow is Explore → Plan **in plan mode**, then **exit** plan mode (approving the plan moves the session into `acceptEdits`/auto/manual review) for the Implement phase. So A is wrong: plan mode does not "see it through" by itself — it stops at the plan. C — *direct/acceptEdits for architectural work* without reconnaissance → expensive rework. D — confuses the modes: `--dangerously-skip-permissions` (`bypassPermissions`) is the removal of safety checks, not "a way to let plan mode write"; the plan is approved by the normal exit from the mode, not by a bypass.
</details>
