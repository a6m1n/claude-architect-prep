# 3.3 — Apply path-specific rules for conditional convention loading

> **Domain 3: Claude Code Configuration & Workflows** (20%) · **Task Statement 3.3** | **Scenarios:** Code Generation with Claude Code | **What it tests:** can you pick the Claude Code memory mechanism for conventions tied to a *file type*, not a *location* — through `.claude/rules/` with a `paths:` glob frontmatter.

## ⚡ TL;DR

- Files in `.claude/rules/*.md` carry YAML frontmatter with a `paths:` field (a list of glob patterns). The rule loads into context **only when Claude reads/edits a file that matches the pattern** → less noise, fewer tokens.
- A rule **without** `paths:` loads unconditionally, with the same priority as `.claude/CLAUDE.md`.
- When a convention is defined by **what kind of file it is** (extension/glob), not by **where it sits** (directory) — use a path-scoped rule. One pattern `**/*.test.tsx` catches tests across the whole tree.
- A glob rule **beats** a directory `CLAUDE.md` for cross-cutting file types scattered across many folders (tests next to source files).
- Antipatterns: everything in the root `CLAUDE.md` (Claude *guesses* the section — unreliable), a `CLAUDE.md` in every folder (directory-bound), skills (loaded by *invocation/relevance*, not deterministically by path).

## 📘 Theory

### Two memory mechanisms and where `.claude/rules/` sits between them

Claude Code keeps persistent context in `CLAUDE.md` files (you write these) and in auto memory (Claude writes this). For large projects the instructions are split into modular files in `.claude/rules/`. Each file is one topic (`testing.md`, `api-design.md`), and all `.md` files are found recursively.

The key principle here is **context engineering**: don't dump everything into the window all the time, load exactly what's needed for the current file. A rule without `paths:` loads at session start (like `.claude/CLAUDE.md`). A rule with `paths:` loads **conditionally**, only when Claude touches a matching file.

### `paths:` glob frontmatter

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API Development Rules
- All API endpoints must include input validation
- Use the standard error response format
```

Patterns (from the official table):

| Pattern | What it catches |
| --- | --- |
| `**/*.ts` | all `.ts` files in any directory |
| `src/**/*` | all files under `src/` |
| `*.md` | markdown in the project root |
| `src/components/*.tsx` | components in a specific folder |

You can list several patterns and use brace-expansion: `"src/**/*.{ts,tsx}"`. The trigger is when Claude **reads a file that matches the pattern**, not on every tool-use.

### Why a glob rule beats a directory `CLAUDE.md` (root-cause / matching mechanism)

Nested `CLAUDE.md` files are *also* path-aware, but they are **directory-bound** — they cover only files in/under their own folder and load when Claude reads a file in that directory. If tests (`Button.test.tsx` next to `Button.tsx`) are scattered across dozens of folders, you'd have to put a `CLAUDE.md` with **duplicated** test rules in each one. A glob rule collapses this into a single `**/*.test.tsx` file, maintained centrally.

A mental rule:
- convention by *file type* → path-scoped rule in `.claude/rules/`;
- convention by *location* → directory `CLAUDE.md`;
- always-on project standard → root / `@`-imported `CLAUDE.md`.

### Why not skills and not "everything in the root"

Skills (`.claude/skills/`) load **on invocation or when Claude itself decides they're relevant** — this is not a deterministic path binding. A root `CLAUDE.md` with section headers requires Claude to *infer* the right section for the current file (inference — unreliable), and the whole block hangs in context every session, burning tokens. Neither gives automatic, path-driven application.

## ⚠️ Traps and distractors

- **Everything in the root `CLAUDE.md` under headers, Claude guesses the section (inference-trap).** The conventions all seem to be in place, but application depends on the model's *inference*, not on a deterministic path match; plus the whole block is in context every session → wasted tokens, "lost in the middle". No path guarantee.
- **A separate `CLAUDE.md` in every subdirectory (directory-bound-trap).** The closest trap. A directory `CLAUDE.md` covers only its own folder. For a file type scattered across the whole tree, this means duplicating the rules in dozens of folders — not scalable. A glob `**/*.test.tsx` solves it with one file.
- **Skills for each code type (manual-invocation-trap).** A `SKILL.md` *can* hold conventions, but skills load by invocation/relevance, not by path. This contradicts the "automatically by path" requirement.
- **"Just rely on Claude knowing the conventions" (no-mechanism-trap).** `CLAUDE.md` and rules are *context*, not enforced configuration; without an explicit loading mechanism the rules are unpredictable. For hard enforcement you need a `PreToolUse` hook, not memory.

## 🔑 Remember

- Directory: `.claude/rules/*.md`; frontmatter field: `paths:` (a list of glob patterns).
- `paths:` present → the rule is **conditional** (loads when a matching file is read). No `paths:` → loads unconditionally, priority like `.claude/CLAUDE.md`.
- Patterns: `**/*.ts`, `src/**/*`, `*.md`, brace-expansion `"src/**/*.{ts,tsx}"`. Cross-cutting example: `paths: ["**/*.test.tsx"]`.
- Rules are found recursively in all `.md` files under `.claude/rules/`; one topic — one file (`testing.md`, `api-design.md`).
- To see which memory/rule files are actually loaded into a session, use the `/memory` command (not guesses about loading "magic").
- Memory is context, not enforcement. A hard block on an action → a `PreToolUse` hook.

## 📝 Practice

*Answer it yourself first, then expand "Answer".*

**Q1 · easy**
A team wants its TypeScript files anywhere in the repo to follow a shared set of conventions, loaded automatically only when Claude edits a `.ts` file. Where does the `paths` glob pattern belong?

A) In a `.claude/rules/*.md` file's YAML frontmatter
B) In the root `CLAUDE.md` under a `## TypeScript` header
C) In `.claude/settings.json` under a `paths` key
D) In a `SKILL.md` file's frontmatter

<details><summary>Show answer</summary>

**Correct answer: A.** The `paths:` field with glob patterns lives in the YAML frontmatter of a file inside `.claude/rules/`; such a rule loads only when working with matching files.
B — inference-trap: the root `CLAUDE.md` relies on guessing the section and stays in context all the time. C — `settings.json` manages permissions/hooks/env, not path-scoped conventions; there's no `paths` field there for this. D — manual-invocation-trap: skills load by invocation/relevance, not deterministically by path.
</details>

**Q2 · medium**
Your repo has unit test files (`*.test.tsx`) sitting next to the components they test, scattered across dozens of feature directories. You want one set of testing conventions applied automatically whenever Claude touches a test file, regardless of its directory. What is the most maintainable approach?

A) A single `.claude/rules/testing.md` with `paths: ["**/*.test.tsx"]`
B) A `CLAUDE.md` in every feature directory repeating the test conventions
C) A `## Testing` section in the root `CLAUDE.md` describing the conventions
D) A `testing` skill that the developer invokes before writing each test

<details><summary>Show answer</summary>

**Correct answer: A.** The convention is defined by *file type* (a test), not by location, and the files are scattered across the tree — this is exactly the case for a glob rule: one pattern `**/*.test.tsx` catches all tests, maintained in a single file.
B — directory-bound-trap: a directory `CLAUDE.md` covers only its own folder → duplication across dozens of folders. C — inference-trap: depends on the model's inference and burns tokens every session. D — manual-invocation-trap: a skill requires manual invocation, it doesn't fire automatically by path.
</details>

**Q3 · hard**
A monorepo has three convention sets: React components (functional + hooks), API handlers (async/await + specific error handling), and DB models (repository pattern). Components and handlers each live under their own top-level directory, but `*.test.ts` files are spread throughout. A developer proposes putting all three conventions plus the test rules in the root `CLAUDE.md`. Why is a path-scoped rule for the test conventions the better design here?

A) It loads test conventions deterministically by file path and only when a test file is edited, instead of relying on Claude to infer the right section and carrying the rules in context every session
B) `.claude/rules/` files are enforced by the client, so test conventions become hard constraints that Claude cannot violate
C) Root `CLAUDE.md` cannot contain testing instructions, so the rules must move to `.claude/rules/`
D) Path-scoped rules load faster than `CLAUDE.md` because they are cached separately on disk

<details><summary>Show answer</summary>

**Correct answer: A.** A glob rule is applied *deterministically by path* and only when a matching file is edited — this removes the dependency on inference and gets rid of the constant token cost of the root `CLAUDE.md`. (Components/handlers live in their own folders — a directory `CLAUDE.md` is appropriate for them; but the tests are scattered → they need a glob.)
B — wrong: both `CLAUDE.md` and rules are *context*, not enforcement; hard enforcement comes only from a `PreToolUse` hook. C — factually false: the root `CLAUDE.md` can perfectly well contain test instructions, it's only a question of efficiency. D — an invented "speed/cache advantage"; the point of path-scoped rules is relevance and context savings, not latency.
</details>

### Tricky question **

**Q★★ (tricky):** You add `.claude/rules/security.md` *without* any `paths` frontmatter, intending it to apply only to files under `src/auth/`. You also keep a `src/auth/CLAUDE.md` with auth-specific notes. During a session editing `src/utils/format.ts`, you notice the security rules are influencing Claude's output even though `format.ts` is unrelated to auth. What is the root cause and the correct fix?

A) A rule file with no `paths` field loads unconditionally (same priority as `.claude/CLAUDE.md`), so `security.md` is always in context; add `paths: ["src/auth/**/*"]` to scope it
B) `src/auth/CLAUDE.md` is being loaded for every file in the repo; delete it and move its content into `security.md`
C) Rules and `CLAUDE.md` conflict, so Claude merged them; resolve by setting `tool_choice` to restrict the rule
D) The `**/*` default glob is implicitly applied to every rule file; override it with an empty `paths: []`

<details><summary>Show answer</summary>

**Correct answer: A.** The subtlety: a rule **without** `paths:` loads **unconditionally** at session start (the same priority as `.claude/CLAUDE.md`) — that's exactly why the security rules surfaced while editing the unrelated `format.ts`. The root cause is the missing `paths:`, not the directory `CLAUDE.md`. The fix is to add `paths: ["src/auth/**/*"]` so the rule becomes conditional.
B — a false root-cause: the nested `src/auth/CLAUDE.md` loads *only* when Claude reads files under `src/auth/`; when working with `src/utils/` it's not active, so there's no reason to delete it. C — `tool_choice` relates to API tool use and has nothing to do with loading rules; it's an off-domain distractor. D — invented behavior: there's no "implicit `**/*`", and an empty `paths: []` is not documented; the absence of `paths` means unconditional loading, not a default glob.
</details>
