# 3.6 — Integrate Claude Code into CI/CD pipelines

> **Domain 3: Claude Code Configuration & Workflows** (20%) · **Task Statement 3.6** | **Scenarios:** Claude Code for Continuous Integration | **What it tests:** running Claude Code non-interactively in a pipeline, machine-readable output, independent review instead of self-review, and removing duplicates by passing prior findings into context.

## ⚡ TL;DR

- In CI, run Claude Code in non-interactive mode with the `-p` flag (also `--print`): it reads the prompt, prints the result to stdout, and exits with a return code. A bare `claude "prompt"` waits for terminal input and **hangs** in the pipeline.
- For machine-readable output add `--output-format json` (result + session metadata), and for a strict schema use `--json-schema '<schema>'` (the validated payload arrives in the `structured_output` field). Parse it with `jq`.
- `CLAUDE.md` is the mechanism for passing project context (test standards, fixture conventions, review criteria) into a CI run, so you raise generation quality and cut out low-value output.
- **Independent** review beats self-review: the same session that wrote the code is worse at finding problems in it. Run the review as a separate instance.
- On repeat reviews after new commits, **pass the prior findings into context** and ask it to report only new or unaddressed problems — otherwise you get duplicate comments.

## 📘 Theory

### Why `-p`, not "workarounds"
By default `claude` starts an **interactive** session and blocks waiting for input. In a pipeline no one is at the keyboard → the job hangs forever. The documented way is **headless / print mode** via `-p` (`--print`): one result to stdout, a clean exit code, support for `--output-format` and `--json-schema`. This is exactly the contract a CI step needs. The principle: **matching workload to mechanism** — for automation you pick the mode designed for automation, not "treat the symptom" with a shell trick.

For reproducibility in CI there is `--bare`: it skips auto-discovery of hooks, skills, plugins, MCP servers, auto memory, and `CLAUDE.md` from the working directory and `~/.claude`, so the result is identical on any machine. In bare mode, context is passed only through explicit flags (`--append-system-prompt`, `--mcp-config`, `--settings`, etc.).

### Structured output for gates
A CI step that gates the build wants to branch on a structured verdict, not parse prose. `--output-format json` returns the result in the `result` field plus a session ID and usage; `--json-schema` additionally forces conformance to a JSON Schema, and the validated object is placed in `structured_output`. The canonical PR-review pattern: `gh pr diff "$1" | claude -p --append-system-prompt "...review for vulnerabilities" --output-format json`.

### Review independence (context isolation)
The same session that generated the code is **worse at reviewing its own changes**: its context is already "convinced" the solution is correct, and it tends to confirm its own assumptions. The principle **independent review beats self-review** is a special case of separation of roles: the review is done by a separate instance/session with its own clean context and criteria from `CLAUDE.md`.

### Stateless API → state-aware review through context
The Messages API is **stateless**: each review run is an independent call with no memory of prior findings or of what the developer has already fixed. To make iterative review state-aware, the architect themselves **reifies the missing state** — the prior findings — and passes it into context, instructing the model to report only new or still-relevant problems. Deduplication here is a task of **semantic reasoning** ("has issue X been fixed by these commits?"), not string matching on text. The same goes for test generation: give the existing test files into context so the model does not produce duplicate scenarios.

## ⚠️ Traps and distractors

- **Invented flags/variables (`CLAUDE_HEADLESS=true`, `--batch`).** They sound plausible but do not exist. Non-interactive mode is selected with the `-p`/`--print` flag, not an env variable. `--batch` is easy to confuse with the server-side **Message Batches API** — that has nothing to do with the Claude Code CLI.
- **`< /dev/null` as a "workaround".** A real shell trick to keep stdin from blocking, but it is **treating the symptom**: it does not give print behavior (one result to stdout, a clean exit, support for `--output-format`/`--json-schema`). The documented path is `-p`.
- **Self-review by the generating session.** Asking the same session to review its own code violates context isolation: it confirms its own assumptions and misses defects. You need an **independent** review.
- **Repeat review without the prior findings.** Running "from scratch" reports already-fixed issues again → duplicate comments. The fix is passing the findings into context, not a post-filter.
- **Post-filter on text/paths (string matching).** Brittle: it cannot tell a truly fixed issue from one that just reads similar; it suppresses on a surface match and risks dropping valid new findings. Deduplication is a reasoning task — give the context to the model.
- **"Skip intermediate commits" / "review only the most recently changed files".** They reduce noise at the cost of coverage (regressions and issues in earlier files get skipped) — and break the "maintain thorough analysis" requirement.

## 🔑 Remember

- Non-interactive mode: `-p` (a synonym for `--print`). A bare `claude "prompt"` will hang in CI.
- Structured output: `--output-format json` (there is also the default `text` and `stream-json`); strict schema: `--json-schema '<schema>'` → object in `structured_output`. Parsing — `jq`.
- Reproducibility in CI: `--bare` (skips hooks/skills/plugins/MCP/auto memory/`CLAUDE.md`).
- Project context for CI → `CLAUDE.md` (test standards, review criteria).
- Independent review ≠ self-review. Repeat review — with prior findings in context; test generation — with existing test files in context.
- Do NOT exist: `CLAUDE_HEADLESS`, `--batch`. `< /dev/null` is not a supported mechanism.
- GitHub path: `anthropics/claude-code-action@v1` (input `prompt`, CLI flags via `claude_args`).

## 📝 Practice

*Answer it yourself first, then expand "Answer".*

**Q1 · easy** — A CI step runs `claude "Review the diff for security issues"` and the job hangs; logs show Claude Code waiting for interactive input. What is the correct fix?

A) Set `CLAUDE_HEADLESS=true` in the job environment before the command.
B) Add the `-p` flag: `claude -p "Review the diff for security issues"`.
C) Redirect stdin from `/dev/null`: `claude "Review the diff for security issues" < /dev/null`.
D) Add the `--batch` flag: `claude --batch "Review the diff for security issues"`.

<details><summary>Show answer</summary>

**Correct answer: B.** `-p` (`--print`) is the documented non-interactive mode: it reads the prompt, prints the result to stdout, and exits with a return code.
- A — **invented env variable**: `CLAUDE_HEADLESS` does not exist; the mode is chosen with a flag, not a variable.
- C — **treating the symptom**: `< /dev/null` removes the stdin block but does not give print behavior and support for `--output-format`/`--json-schema`.
- D — **invented flag**: `--batch` does not exist; confusion with the server-side Message Batches API.
</details>

**Q2 · medium** — A build gate must branch on whether Claude approved a PR. The step currently scrapes the approval verdict out of Claude's prose reply, which is brittle. What is the most effective change so the pipeline can reliably parse a machine-readable verdict?

A) Add `--output-format json` with `--json-schema` and read the validated verdict from `structured_output`, parsing with `jq`.
B) Pipe the prose output through `grep "APPROVED"` and branch on the exit code.
C) Switch the model to a larger one so the prose format is more consistent.
D) Run the review in interactive mode and copy the result into the next step manually.

<details><summary>Show answer</summary>

**Correct answer: A.** `--output-format json` + `--json-schema` forces schema conformance, and the validated object arrives in `structured_output`; the gate branches on a structured field, not on prose. Parsing — `jq`.
- B — **brittle string matching**: `grep` on text breaks at the slightest change in wording; the same problem as with post-filters.
- C — **not the root cause**: a larger model does not make prose a contract; you need an enforced schema.
- D — **interactive mode is not for a pipeline**: manual transfer is incompatible with automation and brings back hanging / manual labor.
</details>

**Q3 · hard** — An automated review produced 12 findings. A developer pushed commits fixing several issues; the next review run posts 8 comments, 5 of which duplicate earlier comments on code that is already fixed. What most effectively removes the redundant feedback while keeping the analysis thorough?

A) Only review the PR at creation and at the final pre-merge state, skipping intermediate commits.
B) Add a post-processing filter that drops findings whose file path and description match a previous comment.
C) Include the prior review findings in context and instruct Claude to report only new or still-unaddressed issues.
D) Restrict the review scope to files changed in the most recent push.

<details><summary>Show answer</summary>

**Correct answer: C.** The Messages API is stateless — pass the prior findings into context and ask the model to report only new or unaddressed ones. Deduplication is a reasoning task ("has the issue been fixed given the new commits?"), and only the model has the information for it; the thoroughness of the analysis is preserved.
- A — **gap in coverage/iterativeness**: you lose fast iterative feedback, and the final run can still surface stale findings.
- B — **brittle post-filter**: matching on text/path cannot tell fixed from similar, suppresses on the surface, and risks dropping valid new findings.
- D — **loss of coverage**: regressions and issues in earlier files go unreviewed — the "maintain thorough analysis" requirement is broken.
</details>

### Tricky question **

**Q★★ (tricky):** A CI job generates unit tests with `claude -p "Generate tests for payment.py" --output-format json`, then a **second** `claude -p` invocation reviews the generated tests for gaps. Reviewers complain about two recurring problems: (1) the generator keeps producing tests that duplicate scenarios already present in the existing `tests/` suite, and (2) the reviewer sometimes rubber-stamps weak tests. Which single change most effectively addresses the **root cause** of both problems at once?

A) Replace the second invocation with `--continue` so the reviewer reuses the generator's session — it already knows the code, so it reviews faster and more cheaply.
B) Provide the existing test files in the generator's context and keep the review as an independent `-p` instance with the review criteria from `CLAUDE.md`.
C) Add `--bare` to both invocations so the runs are deterministic and stop producing duplicates.
D) Add a post-processing filter that deletes any generated test whose name matches a test already in `tests/`.

<details><summary>Show answer</summary>

**Correct answer: B.** Here there are two different root causes, and both are fixed with context engineering + isolation:
- Duplicate tests — because the generator **does not see the existing suite**. Fix: give it the existing test files in context so it avoids already-covered scenarios.
- Weak review — you need **independence**: a separate `-p` instance with its own clean context and criteria from `CLAUDE.md`.

Why the rest are traps:
- A — **the most tempting and for exactly that reason wrong**: `--continue` merges the review into **the same session** that wrote the code. This violates context isolation (the generating session is worse at reviewing its own changes and tends to confirm its assumptions) and, worse, **aggravates problem #2** (rubber-stamping) instead of solving it. "Cheaper and faster" is the bait.
- C — **`--bare` is about reproducibility, not dedup**: it only skips auto-discovery of configuration; it gives the generator no knowledge of existing tests and does nothing to strengthen the review. Misses both root causes.
- D — **brittle post-filter on name**: matching on names cannot tell a real scenario duplicate from a test with a similar name, does not touch the weak-review problem, and treats the symptom rather than the cause.
</details>
