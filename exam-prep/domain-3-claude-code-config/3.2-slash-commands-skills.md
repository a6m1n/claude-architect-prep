# 3.2 — Create and configure custom slash commands and skills

> **Domain 3: Claude Code Configuration & Workflows** (20%) · **Task Statement 3.2** | **Scenarios:** Code Generation with Claude Code | **What it tests:** where commands/skills live (project vs user scope), the `SKILL.md` frontmatter (`context: fork`, `allowed-tools`, `argument-hint`), and the difference between "on-demand skills vs `CLAUDE.md` always-loaded".

## ⚡ TL;DR
- **Scope follows audience.** A command/skill that the whole team should share → put it in the repository's `.claude/` (`.claude/skills/<name>/SKILL.md` or legacy `.claude/commands/<name>.md`). Just for you → `~/.claude/skills/` or `~/.claude/commands/`.
- **Skills load on-demand**: only the `description` is always in context, and the full body of `SKILL.md` loads only when the skill is invoked (`/name` or an auto-trigger). `CLAUDE.md` loads **fully in every session** — it is for always-applicable standards, not for procedures.
- `context: fork` runs the skill in an **isolated sub-agent**, so noisy output (codebase analysis, brainstorming) does not clutter the main conversation.
- `allowed-tools` **grants permission** for tools without an approval prompt (it does not "narrow" access; for a hard restriction use `disallowed-tools`). `argument-hint` suggests the expected arguments in autocomplete.
- Make the personal version of a team skill in `~/.claude/skills/` **under a different name**, so you don't affect your colleagues. The command name comes from the **directory name**, not from the `name` field.

## 📘 Theory

### Where commands and skills live (distribution = choosing scope)
Custom slash commands and Agent Skills are resolved from two places:

| Scope | Location | Available to | Shared via |
|---|---|---|---|
| **Project** | `.claude/skills/<name>/SKILL.md` (or legacy `.claude/commands/<name>.md`) | everyone in this repository | version control (clone/pull) |
| **Personal** | `~/.claude/skills/<name>/SKILL.md` (or `~/.claude/commands/<name>.md`) | only you, in all projects | nothing (local) |

The principle is **least surprise / scope follows audience**: a team `/review` or `/deploy` that everyone should get on `git clone` must live in the repository's `.claude/`. Put it in `~/.claude/` and it will work on your machine and "silently" be missing for your colleagues. The command name comes from the skill's directory name (`.claude/skills/deploy-staging/` → `/deploy-staging`), and for legacy commands from the file name (`deploy.md` → `/deploy`). Custom commands were merged into Skills: `.claude/commands/*.md` still works, but Skills add frontmatter, supporting files, and auto-loading.

### `SKILL.md` frontmatter
`SKILL.md` = YAML frontmatter (when to apply) + markdown instructions (what to do). The key fields for the exam:

- **`context: fork`** — run the skill in a **forked sub-agent**: the skill body becomes the sub-agent's prompt, it has no access to the main conversation history, and its (often wordy) result **does not pollute** the main context. Ideal for a verbose codebase analysis or brainstorming. (`agent:` sets the sub-agent type.)
- **`allowed-tools`** — a list of tools that can be called **without a permission prompt** while the skill runs. This is a **permission grant**, not a restriction: the other tools stay available and are governed by your permission settings. For an actual ban, use **`disallowed-tools`**.
- **`argument-hint`** — a hint in autocomplete about the expected arguments (for example `[issue-number]` or `[filename] [format]`) when a developer invokes the skill. The arguments themselves are substituted through `$ARGUMENTS` / `$0`.

#### How `argument-hint` works: how an argument reaches the skill body

`argument-hint` in the SKILL.md frontmatter does two things:
1. **In the interface:** shows a hint in autocomplete (for example `[issue-number]`, `[filename] [format]`), indicating that the skill expects input.
2. **At runtime:** Claude Code passes the supplied argument into the command/skill body through special variables:
   - `$ARGUMENTS` — everything the user wrote after the command name (for example, `/review src/auth.ts` → `$ARGUMENTS = "src/auth.ts"`).
   - `$1`, `$2`, ... — individual positional arguments (if there are several separated by spaces).

Example frontmatter and usage:
```yaml
argument-hint: "[filename] [format]"
```
In the SKILL.md body: `Analyze the file $1 and output results in $2 format.`
When invoked as `/myskill auth.ts json` → the body becomes: `Analyze the file auth.ts and output results in json format.`

**Important for the exam:** if `argument-hint` is set but the user invokes the skill without arguments, Claude Code will ask them to enter them. If there are no arguments in the body (`$ARGUMENTS` is not used), the hint only helps with the suggestion in the UI.

- Invocation control: `disable-model-invocation: true` (only you can invoke it — for commands with side effects like `/deploy`); `user-invocable: false` (only Claude, for background knowledge).

### Skills (on-demand) vs `CLAUDE.md` (always-loaded)
This is a decision about the **context budget**. `CLAUDE.md` is injected in full into **every** session — the place for **universal, always-applicable** standards (code style, test conventions). A skill: only the short `description` is always in context, **the full body loads only when invoked** — the place for **episodic, task-specific** multi-step procedures (PR review, deploy, migrations), triggered by keywords or `/name`. The rule: classify guidance by **when it should be active**, not by topic.

A similar-looking but wrong alternative is to scatter the instructions across `.claude/rules/` with a path-glob. Path rules fire by the **type of files opened**, not by **activity** (deploy, migration), so for a task-level workflow that is the wrong trigger semantics. `@import` does not help either: imported files still load in full at startup, so there is no context saving.

### Conflict resolution and names
When skill names collide, the priority is: **enterprise > personal > project**; and a skill beats a legacy command with the same name. The name you type after `/` comes from the skill's **directory name** (or the file name for `.claude/commands/*.md`) — the `name` frontmatter field only sets the displayed label in the listing (the exception is a plugin-root `SKILL.md`). So, to keep a personal version from replacing the team one, put it in `~/.claude/skills/` under a **different directory name**.

## ⚠️ Traps and distractors

- **"Move universal standards into Skills."** Wrong: skills are on-demand — the standards would stop loading by default and Claude would start ignoring them when the skill does not fire. Always-on standards stay in `CLAUDE.md`.
- **"Put a team command in `~/.claude/commands/`."** Wrong scope: `~/.claude/` is personal and not in version control, so colleagues will not get it. The most tempting trap — the location is real, the scope is wrong.
- **"Skills are always loaded into context."** Wrong: only the `description` is always present; the full body only loads when invoked. This is confusion with `CLAUDE.md`, which loads in full.
- **"`allowed-tools` restricts/bans tools."** Subtly wrong: `allowed-tools` **grants** auto-approve, it does not take away access. The hard removal of tools is `disallowed-tools`.
- **"A personal version of a skill under the same name."** Conflict: when names collide, personal overrides project — it would affect the team's expectations. Make the version **under a different name** in `~/.claude/skills/`.
- **"`CLAUDE.md` can register `/review`."** No: `CLAUDE.md` carries instructions/context, not invocable commands.
- **"`argument-hint` is only for the UI hint, the argument does not reach the body"** — Wrong: Claude Code injects the argument through `$ARGUMENTS` (or `$1`, `$2`) into the command/skill body at runtime. `argument-hint` sets both the UI hint and the argument injection point.

## 🔑 Remember
- Project (shared): `.claude/skills/<name>/SKILL.md` or `.claude/commands/<name>.md`. Personal: `~/.claude/skills/...` / `~/.claude/commands/...`.
- Command name = the skill's **directory name** (for legacy, the file name), not the `name` field (the exception is plugin-root).
- `context: fork` → isolated sub-agent, no access to history, output does not pollute the main context.
- `allowed-tools` = auto-approve without a prompt (not a restriction); `disallowed-tools` = real removal of tools.
- `argument-hint` = autocomplete hint; arguments are available through `$ARGUMENTS`, `$0`, `$1`.
- `argument-hint: "[param]"` → at runtime Claude Code injects the argument through `$ARGUMENTS` / `$1` into the skill body; without `$ARGUMENTS` the hint is only a UI suggestion.
- `CLAUDE.md` — always loaded in full; Skills — the description is always in context, the body is on-demand.
- On a skill name conflict: enterprise > personal > project; a skill beats a command with the same name.

## 📝 Practice
*Answer on your own first, then expand the "Answer".*

**Q1 · easy**
Your team standardizes on a `/review` slash command that runs a code-review checklist. Every developer must get it automatically on `git clone`/`git pull`. Where do you put the definition?

A) `~/.claude/commands/review.md` in each developer's home directory
B) `.claude/commands/review.md` (or `.claude/skills/review/SKILL.md`) in the project repository
C) In the project-root `CLAUDE.md`
D) In `.claude/settings.json` under a `commands` array

<details><summary>Show answer</summary>

**Correct answer: B.** Project-scoped commands/skills live in the repository's `.claude/` and reach everyone through version control — scope follows audience.
- A — *wrong scope*: `~/.claude/` is personal, not in VCS, so colleagues will not get it.
- C — `CLAUDE.md` carries instructions, not invocable commands (*wrong mechanism*).
- D — fabrication: there is no `commands` array in `settings.json`; commands are files.
</details>

**Q2 · medium**
A skill performs a long, verbose codebase audit whose output is only needed as a short summary. You don't want pages of analysis filling the main conversation. Which `SKILL.md` frontmatter is the most effective?

A) `context: fork`
B) `allowed-tools: Read Grep`
C) `disable-model-invocation: true`
D) `user-invocable: false`

<details><summary>Show answer</summary>

**Correct answer: A.** `context: fork` runs the skill in an isolated sub-agent: the wordy output does not pollute the main context, and only the result is returned upward.
- B — `allowed-tools` only auto-approves tools, it does not isolate context (*wrong axis*).
- C — `disable-model-invocation` controls who invokes the skill, not context pollution.
- D — `user-invocable: false` hides it from the `/` menu; it has nothing to do with context noise.
</details>

**Q3 · hard**
You wrote a team skill `/refactor` in `.claude/skills/refactor/SKILL.md`. You personally want a tweaked version with extra steps, but teammates must keep the original behavior. What's the best approach?

A) Edit the project `.claude/skills/refactor/SKILL.md` and add a `personal: true` field
B) Create `~/.claude/skills/refactor-mine/SKILL.md` with your variant under a different name
C) Create `~/.claude/skills/refactor/SKILL.md` so your version overrides the project one
D) Move the skill into your `CLAUDE.md` so it loads only for you

<details><summary>Show answer</summary>

**Correct answer: B.** The personal version goes in `~/.claude/skills/` **under a different name** (`refactor-mine`), so it does not affect the team `/refactor`.
- A — editing the project file changes the behavior for everyone; there is no `personal` field (*fabrication*).
- C — a same-named personal skill **overrides** the project one (personal > project) — your `/refactor` would silently replace the team one for you and break consistency.
- D — `CLAUDE.md` is always loaded and does not register commands; a multi-step procedure does not belong there.
</details>

### Tricky question **
**Q★★ (tricky):** A skill must run an autonomous cleanup loop and must **never** be able to delete files via `Bash(rm *)`, even though `rm` is otherwise permitted by the developer's `settings.json`. Which frontmatter change actually enforces this?

A) `allowed-tools: Read Write Edit` (omit `Bash`)
B) `disallowed-tools: Bash(rm *)`
C) `context: fork`
D) `disable-model-invocation: true`

<details><summary>Show answer</summary>

**Correct answer: B.** Only `disallowed-tools` **removes** the tool from the available pool while the skill is active — that is the real ban.
- A — the most tempting trap: it seems that "I didn't list `Bash` in `allowed-tools` → so I banned it". But `allowed-tools` only **grants auto-approve**; tools that are not included **stay available** and are governed by the normal permission settings. Since `rm` is permitted in `settings.json`, the skill can still call it. `allowed-tools` ≠ an allowlist restriction.
- C — `context: fork` isolates context, but does not restrict the sub-agent's tool set.
- D — `disable-model-invocation` changes who starts the skill, not which tools are available to it.
</details>
