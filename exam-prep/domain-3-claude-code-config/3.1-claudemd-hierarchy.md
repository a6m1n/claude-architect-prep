# 3.1 — Configure CLAUDE.md files with appropriate hierarchy, scoping, and modular organization

> **Domain 3: Claude Code Configuration & Workflows** (20%) · **Task Statement 3.1** | **Scenarios:** Code Generation with Claude Code | **What it tests:** the correct hierarchy level for `CLAUDE.md` (user / project / directory), understanding that the user level is NOT shared through version control, and modularity through `@import` and `.claude/rules/`.

## ⚡ TL;DR

- **Scope = the sharing boundary.** `~/.claude/CLAUDE.md` (user level) is only for you on your machine, and it **never goes into version control**. Team standards must live at the **project level** (`./CLAUDE.md` or `./.claude/CLAUDE.md`), which is committed and shared through source control.
- The hierarchy loads **from broad to narrow**: managed policy → user → project → directory (subdirectory `CLAUDE.md`). Everything is **concatenated**, not overwritten; the narrower one is read last.
- To keep the file from getting too big, **make it modular**: `@path/to/file` (`@import`) pulls in external standards files, and `.claude/rules/` holds topic files (`testing.md`, `api-design.md`) — an alternative to a single big `CLAUDE.md`.
- The symptom "a new team member's Claude ignores the standards" = the diagnosis **wrong scope**: the rules ended up at the user level instead of the project level. Fix the root, not the symptom.
- `/memory` shows which memory files are actually loaded into the session — the main tool for diagnosing inconsistent behavior between sessions or people.

## 📘 Theory

### Hierarchy and load order
`CLAUDE.md` exists at several levels, and Claude Code loads them **from the broadest scope to the narrowest** (and they appear in the context in that same order, so the project instruction is read after the user instruction):

| Scope | Location | Shared with |
|---|---|---|
| Managed policy | system path (e.g. `/Library/Application Support/ClaudeCode/CLAUDE.md`) | all users in the organization |
| **User** | `~/.claude/CLAUDE.md` | **only you** (all your projects) |
| **Project** | `./CLAUDE.md` or `./.claude/CLAUDE.md` | **the team via source control** |
| Local | `./CLAUDE.local.md` (in `.gitignore`) | only you (this project) |

Files in directories **above** the working directory are loaded in full at startup. Files in **subdirectories** (directory level) are loaded **on demand**, when Claude reads files in that folder. All files found are **concatenated** into the context rather than overriding one another.

### Why scope is the sharing boundary (the key principle)
The main design idea of this task: **the level = who will see it**. `~/.claude/CLAUDE.md` physically sits on one machine and is **not under version control**, so colleagues will never get it. `./.claude/CLAUDE.md` is committed to the repository and loads automatically for anyone who clones the project. Hence the rule: personal preferences → user level; team standards → project level. This is an application of the **least surprise / correct scope** principle: place config exactly as broadly as it should take effect.

### Modularity: `@import` and `.claude/rules/`
A long, single `CLAUDE.md` (>200 lines) eats context and lowers adherence. Two ways to break it up:

- **`@import`** — the `@path/to/file` syntax pulls in external files (e.g. `@docs/git-instructions.md` or per-package standards). Handy for modular organization. Important: imported files are still **loaded into the context at startup** — there is no context saving, this is about organization.
- **`.claude/rules/`** — a directory of topic files (`code-style.md`, `testing.md`, `security.md`). Without `paths` frontmatter they load at startup with the same priority as `.claude/CLAUDE.md`. With `paths` frontmatter (glob) they load **only when Claude works with matching files**, which really does save context.

### Diagnosis: `/memory`
When behavior is inconsistent between sessions or between people, `/memory` lists all loaded `CLAUDE.md`, `CLAUDE.local.md`, and rules files. If a file is not in the list, Claude does not see it, and the problem is in the location/scope, not in how the rule is worded.

## ⚠️ Traps and distractors

- **"Put team rules in `~/.claude/CLAUDE.md`."** Wrong: the user level is not under version control, so colleagues will not get it. This is exactly the root cause of the "the new hire's standards are ignored" bug — the fix is to move them to the project level, not to duplicate them.
- **"Have each developer copy the standards into their own `~/.claude/CLAUDE.md`."** Works for a while, but leaves the rules at the user level: no versioning, drift, and the next new hire again does not get them. This treats the symptom, not the root (root-cause-over-symptom).
- **"Assuming the user level is shared with the team."** Directly false: `~/.claude/CLAUDE.md` is shared **only with you**. Sharing with the team comes solely from the project level via source control.
- **"Leave one monolithic `CLAUDE.md` instead of `@import` / `.claude/rules/`."** Wrong for large projects: it bloats the context and lowers adherence; the right move is modularizing via `@import` or `.claude/rules/`.
- **"Inflate the max context window so the standards definitely fit."** Red herring: the problem is not truncation, it is that the rules sit in a non-shared location. Context size has nothing to do with the scope problem.
- **"Demote always-on standards to Skills."** Skills load on demand by triggers; always-needed team standards must be always-loaded at the project level `CLAUDE.md` (or `.claude/rules/`), otherwise they silently stop being applied.

## 🔑 Remember

- User level: `~/.claude/CLAUDE.md` — **personal, NOT in version control**, applies to all your projects.
- Project level (shared with the team): `./CLAUDE.md` **or** `./.claude/CLAUDE.md`.
- Personal for one project (do not commit): `./CLAUDE.local.md` → add to `.gitignore`.
- Load order: managed → user → project → directory (subdirectory loads on demand); everything is concatenated.
- Modularity: `@path/to/file` (`@import`, up to 4 hops deep) and `.claude/rules/*.md` (topic files; `paths` frontmatter = path-scoped loading).
- `@import` files and rules without `paths` load **at startup** (no context saving; only a path-scoped rule saves context).
- Diagnosis: **`/memory`** shows the memory files that are actually loaded.
- The size target for a single `CLAUDE.md` is **< 200 lines**.

## 📝 Practice

*Answer for yourself first, then expand "Answer".*

**Q1 · easy**
A team's coding standards are followed when a senior engineer runs Claude Code, but a newly onboarded teammate reports Claude ignores those same standards on the identical repo. You discover the standards live in the senior engineer's `~/.claude/CLAUDE.md`. What is the most effective fix?

A) Have each developer paste the standards into their own `~/.claude/CLAUDE.md`.
B) Move the standards into the project's `./.claude/CLAUDE.md` (or root `CLAUDE.md`) and commit it.
C) Convert the standards into a Skill so they load on demand.
D) Increase the model's context window so the standards always fit.

<details><summary>Show answer</summary>

**Correct answer: B.** `~/.claude/CLAUDE.md` is user level, not under version control, so colleagues do not see it. Moving it to the project level (`./.claude/CLAUDE.md` or root `CLAUDE.md`) and committing it puts the standards in the shared, versioned part of the hierarchy that anyone who clones the repo will get.
A — treating the symptom: the rules stay at the user level, with no versioning, they drift over time, and the next new hire again does not get them (root-cause-over-symptom).
C — Skills load on demand; always-on team standards must be always-loaded at the project level (wrong mechanism).
D — red herring: the problem is the location, not context truncation.
</details>

**Q2 · medium**
Your project `CLAUDE.md` has grown past 400 lines: universal coding standards plus large, package-specific style guides for each of five packages in a monorepo. You want the file lean and modular while keeping all guidance loaded. What is the best approach?

A) Delete the package-specific sections and rely on the model to infer per-package conventions.
B) Move everything into `~/.claude/CLAUDE.md` so it applies across all your projects.
C) Keep universal standards in `CLAUDE.md` and pull in per-package standards via `@import` to separate files (e.g. `@packages/api/standards.md`).
D) Put the whole file behind a Skill triggered by the keyword "standards".

<details><summary>Show answer</summary>

**Correct answer: C.** `@import` (`@path/to/file`) pulls in external standards files, making `CLAUDE.md` modular and maintainable, while everything needed is still loaded into the context at startup.
A — you lose the explicit conventions, hoping the model will guess (unreliable).
B — moves team standards to the user level: not shared with the team and loses the project specifics.
D — Skills load on demand by trigger; always-needed standards must not depend on a trigger firing (wrong mechanism / demoting always-on).
</details>

**Q3 · hard**
Two developers on the same repo see Claude apply different style rules in the same files, and you suspect duplicated or stray instructions across memory files. Before changing any rule text, what is the most effective first diagnostic step?

A) Run `/memory` in each developer's session to list which `CLAUDE.md`, `CLAUDE.local.md`, and rules files are actually loaded, then compare.
B) Rewrite the style rules to be more emphatic so Claude follows them more strictly.
C) Move all rules into a single monolithic `CLAUDE.md` to eliminate ambiguity.
D) Increase the context window so no instruction gets dropped.

<details><summary>Show answer</summary>

**Correct answer: A.** `/memory` lists the memory files actually loaded in the current session. By comparing the two developers' lists, you will see which file is present/absent or conflicting (often the cause is a personal `CLAUDE.local.md` or a user-level rule on one of them). This is root-cause diagnosis before edits.
B — rewriting the text without diagnosis does not explain why the files differ for the two people (treating the symptom).
C — premature "fixing" blind: a monolith will not show where the rules currently diverge, and it breaks modularity.
D — red herring: the divergence is due to different loaded files, not context truncation.
</details>

### Tricky question **

**Q★★ (tricky):** In a monorepo, the root `CLAUDE.md` says "use tabs," and a subdirectory `services/billing/CLAUDE.md` says "use 2-space indentation." A developer runs Claude Code from the repo root and opens a file in `services/billing/`. Which statement is the most accurate about how these instructions are applied?

A) Only the root `CLAUDE.md` is loaded at launch from the root, so Claude uses tabs and never sees the billing rule.
B) The subdirectory `CLAUDE.md` is loaded on demand when Claude reads files in `services/billing/`, and being the more specific (narrower) instruction read later, it governs files there (2-space).
C) Both load at launch and conflict unresolved, so Claude picks tabs vs. 2-space at random every time.
D) The subdirectory file overrides the root globally for the whole session, so the entire repo switches to 2-space once that file opens.

<details><summary>Show answer</summary>

**Correct answer: B.** The subtlety is in the **directory-level loading semantics**: files in directories *above* the working directory load in full at startup, while a subdirectory `CLAUDE.md` loads **on demand**, when Claude reads files in that folder. Instructions are concatenated from the root down, and the narrower one (closer to the edited file) is read **last**, so for files in `services/billing/` the 2-space rule applies.
A — wrong: subdirectory files are not ignored, they are pulled in when working with files in that folder (just not at launch).
C — wrong: this is not an unresolved random conflict; the hierarchy gives a deterministic order (narrower is read last), and a "random pick" happens with *contradictions at the same level*, not here.
D — wrong, and this is the most tempting distractor: the subdirectory rule applies **scoped** to its own subdirectory, not globally to the whole repo; the root rule (tabs) keeps applying outside `services/billing/`.
</details>
