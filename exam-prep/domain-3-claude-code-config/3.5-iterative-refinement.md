# 3.5 — Apply iterative refinement techniques for progressive improvement

> **Domain 3: Claude Code Configuration & Workflows** (20%) · **Task Statement 3.5** | **Scenarios:** Code Generation, Developer Productivity | **What it tests:** how to iteratively get Claude Code's result to where you need it — through concrete I/O examples, the test-driven loop, the interview pattern, and the right grouping of fixes.

## ⚡ TL;DR

- If prose is interpreted inconsistently (each run gives slightly different code), the problem is **under-specification**, not "not enough pressure." The cure is demonstration: give **2-3 concrete input/output examples**, not a more emotional description.
- **Test-driven iteration:** first write the test suite (expected behavior, edge cases, performance), then iterate by feeding Claude **concrete test failures** — that is an objective convergence signal.
- **Interview pattern:** in an unfamiliar domain, ask Claude to ask questions (`AskUserQuestion`) to surface what you did not think about — cache invalidation, failure modes, edge cases — **before** writing code.
- **Grouping of fixes:** **interacting** bugs (where fixes affect each other) go in **one** message; **independent** bugs go **one after another**.
- For an edge case, always give a concrete example: input + expected output (for example, `null` in a migration script).

## 📘 Theory

### Demonstrate, don't describe

A prose description of a transformation ("flatten nested fields, coerce types, drop empties") leaves gaps: drop `null` or keep it? which date format? The model fills those gaps differently each time — which is where the non-reproducibility comes from. **Principle: concepts that are hard to put into words are best conveyed with examples.** Give 2-3 `input → expected output` pairs, and make sure to include a record with the problematic edge case (for example, with `null` fields). This pins the exact behavior more reliably than any wording. Repeating the same prose more "insistently" or with "be very careful" is useless — you are not adding any new information about *what* the output should be.

### Test-driven iteration (evaluator-optimizer loop)

The best way to improve progressively is to give Claude a **verifiable signal**. Write the tests first (covering expected behavior, edge cases, performance), then iterate by passing in **concrete failing test cases**. This closes the loop "did it → ran the test → read the result → fixed it" without a human acting as the verifier. The official best practice: give Claude a way to check its own work (a test suite, a build exit code, a linter, a screenshot) — otherwise the only signal is "looks done," and you become the verifier yourself.

### Interview pattern (surface the unknown)

In an **unfamiliar domain** you do not know which questions to ask, so the spec is incomplete by design. The solution: ask Claude to run an interview (`Interview me in detail using the AskUserQuestion tool`) to raise design considerations you did not think about — cache invalidation, failure modes, tradeoffs. That way the gaps are closed *before* implementation, instead of showing up as bugs afterward.

### Interacting vs independent fixes

When there are several bugs, **how** you submit them matters:

- **Interacting** (fixing one changes the behavior of another) → **one message** with all the problems, so Claude sees them together and finds a consistent solution. Fixing them separately leads to fix #2 breaking fix #1.
- **Independent** → **one after another**, one at a time. Dumping independent fixes into one "tangled" message bloats the context and confuses priorities for no benefit.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| **Strengthen the prose** ("repeat more emphatically", "be very careful and precise") | Adds no information about the *required output*; the source of variability (under-specification) stays. It tempts you because it feels like "trying harder." |
| **Switch the model on the same prose** | Switching the model fixes a lack of capability, not a gap in the specification. The same ambiguous prose will give the same ambiguity. |
| **Delegate to "best practices"** ("use best practices for X") | Increases ambiguity instead of reducing it and kills verification — there are no examples and no tests. |
| **Fix interacting bugs one after another** | Fixing one bug will break another; you need shared context → one message. |
| **Dump independent bugs into one message** | Bloats the context and confuses priorities; fix independent bugs one at a time. |
| **Skip the interview pattern in an unfamiliar domain** | You do not know what to ask → the spec is incomplete → edge cases (cache invalidation, failure modes) show up as bugs. |

## 🔑 Remember

- Prose is inconsistent → **2-3 concrete input/output examples** (including an edge case like `null`). Demonstrate, don't describe.
- Test-driven: **tests first** (behavior + edge cases + performance), then iterate on **failing tests**.
- Give Claude a way to check itself: test suite, build exit code, linter, screenshot.
- Unfamiliar domain → **interview pattern** (`AskUserQuestion`), questions before code.
- **Interacting** bugs → one message; **independent** → one after another.
- Always specify an edge case with an `input → expected output` example (for example, `null` in a migration script).

## 📝 Practice

*Answer on your own first, then expand "Answer."*

**Q1 · easy**
A developer describes a data-cleaning transformation to Claude Code in prose. Each run produces subtly different code: sometimes empty strings are dropped, sometimes kept. Re-running the same prose prompt does not converge. What is the most effective next step?

A) Re-send the same prose prompt with "be very careful and precise about edge cases."
B) Provide 2-3 concrete input/output example pairs, including a record with empty-string fields.
C) Switch to a larger Claude model and resubmit the identical prose.
D) Tell Claude to "apply standard data-cleaning best practices."

<details><summary>Show answer</summary>

**Correct answer: B.** Inconsistent output from prose = the spec is under-specified. Concrete input/output examples pin the exact behavior (including how to handle empty strings) more reliably than any wording. A — "strengthen the prose": no new information about the required output, the variability stays. C — "switch the model on the same prose": fixes capability, not the spec gap. D — "delegate to best practices": increases ambiguity and removes verification.
</details>

**Q2 · medium**
A developer is asked to harden an existing module. Code review surfaced two issues: (1) a date parser returns wrong results in a non-UTC timezone, and (2) a retry wrapper around that same parser swallows the resulting exception. The fix for the parser changes what the retry wrapper sees. How should the developer prompt Claude Code most effectively?

A) Report both issues in a single message so Claude can fix them together with shared context.
B) Fix the retry wrapper first, then in a separate later message report the parser bug.
C) Ask Claude to "improve the module's robustness" and accept whatever it produces.
D) Re-emphasize in prose that the code must be "very robust and handle all timezones."

<details><summary>Show answer</summary>

**Correct answer: A.** The bugs **interact** (fixing the parser changes what the retry wrapper sees), so they must be submitted together — in one message — so Claude finds a consistent solution. B — "fix interacting bugs one after another": fixing the parser will then break the already "fixed" wrapper. C — "delegate to best practices": ambiguous, no verification. D — "strengthen the prose": emotional pressure adds no information about the required behavior.
</details>

**Q3 · hard**
A team must add a distributed rate-limiter to their API — a domain none of them has built before. They want Claude Code to produce a correct first implementation rather than discovering missing requirements during review. What is the most effective approach before implementation?

A) Write a single detailed prose paragraph describing the rate-limiter and ask Claude to implement it in one pass.
B) Ask Claude to interview them with `AskUserQuestion` about edge cases, failure modes, and tradeoffs, write a spec, then implement from it.
C) Tell Claude to implement it using "industry-standard rate-limiting best practices."
D) Implement a naive version, then send Claude many small independent prose corrections one at a time.

<details><summary>Show answer</summary>

**Correct answer: B.** In an **unfamiliar domain** the team does not know which questions to ask, so any prose spec will be incomplete. The interview pattern forces Claude to surface what was not accounted for (failure modes, consistency, clock skew) *before* code — and that is exactly what reduces the risk of "discovering gaps during review." A — a prose spec in an unfamiliar domain is inevitably incomplete. C — "delegate to best practices": maximum ambiguity. D — treats the symptoms with a scattering of fixes instead of closing the requirements gap at the start.
</details>

### Tricky question **

**Q★★ (tricky):** A developer is writing migration scripts to move records to a new schema. Claude's first version passes the happy-path tests but corrupts rows where the legacy `phone` column is `null`. The developer also wants to fix two unrelated cosmetic issues (a log message typo and an unused import) found in the same file. What is the single most effective next prompt?

A) Provide one message with the `null`-edge-case test (input row with `null` `phone` + expected output), the log typo fix, and the unused-import fix, so everything in the file is corrected together.
B) Provide a concrete failing test case for the `null` edge case (example input + expected output) and address the typo and unused import as separate, independent follow-ups.
C) Re-prompt with prose: "make sure the migration handles null and empty values carefully and clean up the file."
D) Ask Claude to interview the developer about migration edge cases before changing anything.

<details><summary>Show answer</summary>

**Correct answer: B.** Two principles meet here. First — **an edge case is fixed with a concrete example** (input with `null` `phone` + expected output), not with prose about "handle null carefully"; this rules out C. Second, and decisive — **grouping by interaction**: the `null` bug and the cosmetic fixes (typo, unused import) are **independent**, so they should not be dumped into one message. A — tempting ("it's all in one file — fix it all at once"), but "one file" ≠ "interacting problems": the grouping criterion is **whether the fixes affect each other**, not a shared file; the cosmetics do not affect the migration logic, so combining them only bloats the context and dilutes the priority of the real bug. D — the interview pattern is overkill: the domain here is known, the specific defect is already pinned down by a test, there is nothing to "surface."
</details>
