# 1.4 — Implement multi-step workflows with enforcement and handoff patterns

> **Domain 1: Agentic Architecture & Orchestration** (27%) · **Task Statement 1.4** | **Scenarios:** Customer Support Resolution Agent | **What it tests:** when the order of steps is critical for safety — enforce it with a programmatic constraint (prerequisite gate / hook) instead of prompt instructions, plus a structured handoff on escalation.

## ⚡ TL;DR

- If the order of tool calls **must** be followed 100% of the time (for example, verify the customer before `process_refund`) — move the invariant into **code** (`PreToolUse` hook / prerequisite gate) that **blocks** the downstream call until the prerequisite is done.
- Prompt and few-shot are **probabilistic**: they lower the violation rate but never bring it to zero. A measured 12% of skips is direct proof that prompt-only is not enough.
- A routing classifier controls tool **availability** (which subset is enabled), not their **order**. That is a different lever — do not confuse them.
- A complex multi-concern request: **decompose → fetch shared context once → parallel investigation → synthesize** a single answer. Do not serialize independent subtasks.
- When escalating to a human who has no access to the transcript, build a **structured handoff**: `customer ID`, root cause, refund amount, recommended action.

## 🧠 In plain words

> **Analogy:** 🪜 a barrier gate that physically will not lift without a pass — versus a sign that says "please check the pass."
>
> When the order of steps matters for safety (first verify the customer, then refund the money), a request in the prompt is just a sign: the model *usually* obeys it, but not always. Those 12% of skips are the "sometimes did not obey." Only a code gate (`PreToolUse` hook) gives a guarantee: it **physically blocks** `process_refund` until `get_customer` has returned a verified ID. A routing classifier does not help here — it decides *which* tools are enabled, not in *what order* to call them.
>
> 🚩 **Detector:** if the answer is "strengthen the prompt / add few-shot / lower the temperature" for a safety-critical order — that is `symptom-not-root-cause` (a probabilistic nudge instead of a guarantee); the right answer is a code gate (`PreToolUse`) that blocks the downstream call until the prerequisite is done.
>
> 🧷 **Reflex:** need 100% order — put up a barrier gate (lever 5), not a sign; the prompt *asks*, the code *does not let through*. On escalation — not a raw transcript, but a ready summary: `customer ID` + root cause + amount + recommendation.

## 📘 Theory

### Programmatic enforcement vs prompt-based guidance

The prompt **steers** the model, the code **constrains** the system. An LLM is probabilistic by nature: even a strong instruction "always verify first" is followed *most* of the time, and "most" ≠ "always." For irreversible or high-blast-radius actions (refunding money, changing an order) the residual error rate is unacceptable.

The solution is to raise the invariant to the **tool execution layer**. In Claude Code this is a `PreToolUse` hook that, before a tool runs, returns `permissionDecision: "deny"` with a reason if the prerequisite is not met. In the Agent SDK the same `PreToolUse` is available as a callback and can also block the call. This way the model physically cannot call `process_refund` before `get_customer` has returned a verified `customer ID`.

> Anthropic *Building effective agents*: workflows are "systems where LLMs and tools are orchestrated through **predefined code paths**," with programmatic "gate" checks at intermediate steps. A deterministic path where reliability is needed; agentic freedom where flexibility is needed.

### Prerequisite gate: what it looks like

The gate does three things: (1) it checks that the dependency is satisfied (a verified `customer ID` exists); (2) if not — it returns an **error result** that forces the model to call the prerequisite first; (3) it lets the downstream call through only after success. This is a deterministic guarantee, independent of the model's behavior.

### Decompose → parallel → synthesize

A compound request ("charged twice, discount didn't apply, want to cancel") is a bundle of **independent** subproblems with shared context (one customer). The right structure: break it into separate concerns, fetch the shared context **once**, investigate the concerns **in parallel**, then synthesize a single answer. A sequential concern-by-concern pass re-fetches the same context and inflates the number of tool calls. The principle: the agent's control flow should mirror the task's dependency graph — parallelize the independent, reuse the shared.

### Structured handoff on escalation

A human operator who receives the escalation does not see the conversation transcript. So the agent must compile a **structured summary**: `customer ID`, root cause, refund amount, recommended action. This removes the need to re-investigate and passes exactly what is needed for a decision — context engineering at the human-handoff boundary.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| **Strengthen the system prompt** ("verification is required before any operations") | Sounds right — it literally names the rule. But the instruction is probabilistic: those same 12% of skips are the failure mode of prompt-only enforcement. For a safety-critical order it gives no guarantee. |
| **Add few-shot examples** of the correct sequence | Raises the probability of the right order, but leaves a nonzero residual rate of violations on unseen inputs. This is steering, not a constraint. |
| **A routing classifier enables a subset of tools** | Swaps the concepts: the classifier controls *availability* (which tools are enabled for a request type), not *order* among the enabled ones. `lookup_order` can still fire before `get_customer`. This is official Sample Q1: D (the classifier) is wrong, A (the prerequisite gate) is right. |
| **Verification gates/checkpoints between concerns** for a multi-concern request | Forces a *sequential* flow — exactly the opposite of what is needed: independent concerns should be parallelized, not serialized with checkpoints. |
| **Merge the tools into one `investigate_issue`** | Hides the redundancy behind a coarse tool, but does not remove it and kills observability/parallelism. This treats a symptom, not the root cause. |

## 🔑 Remember

- Safety-critical order ⇒ **`PreToolUse` hook / prerequisite gate in code** that returns `"deny"` until the prerequisite is met. A deterministic guarantee, not a nudge.
- A measured nonzero % of violations (for example 12%) ⇒ proof that prompt-only is not enough.
- A routing classifier = tool **availability**, not **order**.
- A compound request ⇒ **decompose → shared context once → parallel → synthesize**; checkpoints/gates only for genuinely dependent steps.
- Escalation to a human ⇒ structured handoff: `customer ID` + root cause + refund amount + recommended action.

## 📝 Practice

*Answer it yourself first, then open the "Answer."*

**Q1 · easy**
A support agent must always call `verify_identity` before `process_refund`. In testing, the model usually does this, but occasionally issues a refund without verification. The action is irreversible. What is the most reliable fix?

A) Add a sentence to the system prompt making verification mandatory.
B) Add a `PreToolUse` gate that blocks `process_refund` until `verify_identity` has returned a verified ID.
C) Add few-shot examples that always verify before refunding.
D) Lower the temperature so the model follows instructions more consistently.

<details><summary>Show answer</summary>

**Correct answer: B.** An irreversible action requires a deterministic guarantee: a gate in code blocks `process_refund` until verification succeeds, independent of the model's behavior. A and C are probabilistic prompts (steering), leaving a nonzero % of violations. D (lowering the temperature) does not make the order guaranteed — it is still the same probabilistic nudge.
</details>

**Q2 · medium**
Production data shows the agent skips `get_customer` in 12% of cases and calls `lookup_order` using only the customer's stated name, sometimes returning the wrong account. What change most effectively addresses this reliability issue?

A) Implement a routing classifier that enables only the tools appropriate for each request type.
B) Strengthen the system prompt to state that `get_customer` is mandatory before any order operation.
C) Add a programmatic prerequisite that blocks `lookup_order` and `process_refund` until `get_customer` has returned a verified customer ID.
D) Add few-shot examples showing the agent always calling `get_customer` first.

<details><summary>Show answer</summary>

**Correct answer: C.** This is the only option that makes the invariant a hard deterministic guarantee at the execution layer. A (the routing classifier) controls tool *availability*, not their *order* — `lookup_order` can still outrun `get_customer`. B and D are probabilistic (prompt/few-shot); the 12% of skips is exactly their failure mode.
</details>

**Q3 · medium**
For simple requests the agent resolves in 3–4 tool calls (91% success). For compound requests ("charged twice, discount didn't apply, want to cancel") it averages 12+ tool calls at 54%, investigating concerns sequentially and re-fetching customer data for each. What is the most effective change?

A) Add few-shot examples of ideal tool-call sequences for multi-part billing scenarios.
B) Decompose the request into distinct concerns, fetch shared customer context once, investigate them in parallel, then synthesize a unified resolution.
C) Add verification gates between steps so the agent checkpoints after each concern before the next.
D) Merge `get_customer`, `lookup_order`, and billing lookups into one `investigate_issue` tool.

<details><summary>Show answer</summary>

**Correct answer: B.** Both failures named in the logs — sequential investigation and re-fetching data — are symptoms of serializing the compound request. Decompose + one shared-context fetch + parallel + synthesize hits the root. A teaches better *sequences*, not a better *structure*. C forces an even more sequential flow (the opposite of what is needed). D hides the redundancy behind a coarse tool, with no parallelism and no observability.
</details>

**Q4 · hard**
After resolving most of a billing dispute, the agent determines a $400 goodwill credit exceeds its limit and must escalate to a human, who does NOT have the conversation transcript. What should the agent do to make the handoff most effective?

A) Forward the full raw conversation transcript so the human can re-derive the situation.
B) Escalate immediately with the customer's name so the human can start a fresh investigation.
C) Compile a structured handoff summary: customer ID, root cause, refund amount, and recommended action.
D) Add a system-prompt rule telling the agent to always escalate disputes over $300.

<details><summary>Show answer</summary>

**Correct answer: C.** A human with no transcript needs a ready, structured summary (`customer ID`, root cause, refund amount, recommended action) to decide without re-investigating — this is context engineering at the handoff boundary. A hands over a raw, noisy transcript (lossy, forces the human to dig out the essence). B loses all the work already done and requires re-investigation. D answers a different question (the escalation threshold), not *how* to hand over the case.
</details>

### Tricky question **

**Q★★ (tricky):** A team adds a `PreToolUse` gate that blocks `process_refund` until `verify_identity` succeeds. To "make verification even more reliable," they ALSO rewrite the system prompt with a strong rule and three few-shot examples enforcing verification-first. Refund-without-verification incidents drop to zero. What is the most accurate assessment of this design?

A) Correct and well-layered: the prompt and few-shot examples are what actually drove incidents to zero, with the gate as backup.
B) The gate alone provides the 100% guarantee; the added prompt/few-shot enforcement is redundant for the invariant and, if relied upon, would still be probabilistic — incidents hit zero because of the gate, not the prompt.
C) Risky: the gate and the prompt rule can conflict, so the prompt should be removed to avoid nondeterministic behavior.
D) Insufficient: a routing classifier is still needed to guarantee the ordering between `verify_identity` and `process_refund`.

<details><summary>Show answer</summary>

**Correct answer: B.** The subtlety is in attributing the cause. The gate at the execution layer already gives a deterministic 100% guarantee — it is what brought incidents to zero. The added prompt rule and few-shot for *this invariant* are redundant: on their own they stay probabilistic, and without the gate they would not guarantee zero. The danger is to wrongly credit the prompt with the success (A) and then weaken or remove the gate, bringing back a nonzero failure rate.

Why the others are traps: **A** — a causal swap: the gate produced zero, not the prompt. **C** — the "conflict" is imaginary: the prompt only duplicates what the gate enforces anyway; it introduces no nondeterminism (though for *this* invariant it is simply unnecessary). **D** — the classic *order ↔ availability* swap: a routing classifier controls tool availability, not order; the sequence guarantee comes precisely from the prerequisite gate.
</details>
