# 1.3 — Configure subagent invocation, context passing, and spawning

> **Domain 1: Agentic Architecture & Orchestration** (27%) · **Task Statement 1.3** | **Scenarios:** Multi-Agent Research System | **What it tests:** how a coordinator spawns subagents through `Task`, passes them context explicitly, and runs them in parallel.

## ⚡ TL;DR

- The spawning mechanism for a subagent is the `Task` tool (also called `Agent` in newer versions). For the coordinator to call subagents without asking for permission, `Task`/`Agent` must be in its `allowedTools`. No `Task` in `allowedTools` → the coordinator spawns no one.
- A subagent starts in a **fresh, isolated context**: it does NOT inherit the parent's conversation history, tool results, or memory. The only parent→child channel is the `Task` prompt string.
- Pass the **full findings of earlier agents directly in the prompt**, not a reference/pointer/"summary". Whatever is not in the prompt, the subagent does not know.
- Use a **structured format** that separates content from metadata (source URLs, document names, page numbers), so attribution survives the hand-off.
- **Parallel = several `Task` calls in ONE coordinator response.** Sequential calls across different turns are NOT parallel.
- Configure each subagent through `AgentDefinition`: `description` (when to call it), `prompt` (system prompt/expertise), `tools` (limit privileges). The coordinator's prompt sets the **goals and quality criteria**, not a step-by-step procedure.

## 🧠 In plain words

> **Analogy:** 🧳 a new hire does not have telepathy; they get a brief in hand — they know nothing that you did not put in their folder yourself.
>
> A subagent starts with EMPTY memory: it does NOT inherit the parent's history, tool results, or memory. The only channel is the `Task` prompt string: put the ready findings right there, and keep the metadata separately (URL/document name/page) so you do not lose "who said this". Want three of them to work at once — hand out three assignments in ONE letter: several `Task` calls in ONE coordinator response (not one after another across different turns). And `Task` itself must be in the coordinator's `allowedTools`, otherwise it spawns no one.
>
> 🚩 **Detector:** if the answer says "shared memory / inherits context / raise `max_tokens` / pass a reference-pointer" — that is a common misconception (confusing the output budget with a missing input); the right move is to embed the full findings inline in the `Task` prompt. If "calling subagents one after another across different turns" is called parallel — that is under-engineered; parallel = several `Task` calls in one response.
>
> 🧷 **Reflex:** empty head, brief in hand: whatever is not in the `Task` prompt, the subagent does not know; three at once = three `Task` calls in one letter.

## 📘 Theory

### How a subagent is spawned

The coordinator delegates a focused subtask by calling the `Task` tool (renamed to `Agent` in Claude Code v2.1.63+; the SDK still shows `Task` in the `system:init` tool list and in `permission_denials[].tool_name`). For subagent calls to be **auto-approved** without a permission prompt, the tool name (`Task`/`Agent`) must be in the coordinator's `allowedTools`. This follows directly from the permission model: a subagent is a "tool action", and without permission the call goes to the `canUseTool` callback or is rejected.

### Context isolation — the key principle (context engineering)

Each subagent works in **its own fresh conversation**. From the official "What subagents inherit" table: a subagent gets its system prompt (`AgentDefinition.prompt`) + the `Task` prompt string + the project `CLAUDE.md` + tool definitions; it does **NOT** get the parent's conversation history, tool results, or system prompt. Intermediate tool-calls stay inside the subagent — only the final message returns upward.

From this comes the main design rule: **context is passed EXPLICITLY**. If a synthesis subagent needs the findings of the web-search and document-analysis agents, the coordinator must **embed those findings in full into the `Task` prompt**. Neither "shared memory", nor "inheriting history", nor raising `max_tokens` replaces this — the data simply is not in the subagent's input.

### Structured hand-off and attribution

During synthesis it matters which source supports which claim. So findings are passed in a **structured format** that separates content from metadata (source URL, document name, page number). This way attribution survives the parent→subagent "jump", and the final report stays reference-verifiable.

### Parallelism = several `Task` calls in one response

The real speed-up (Anthropic reports up to −90% time on complex queries) comes from running 3–5 subagents **at the same time**. Technically this means: the coordinator emits **several `Task` calls in a single response of its own**. If it calls subagents one after another, one per turn, that is sequential execution, not parallel, and there is no speed gain.

### AgentDefinition and "goals, not procedure"

Each subagent type is described by `AgentDefinition`: `description` (Claude uses it to decide when to delegate), `prompt` (role/expertise/constraints), `tools` (least privilege — for example, read-only `Read`/`Grep`/`Glob`). The coordinator's prompt should set the **research goals and quality criteria**, not a step-by-step algorithm — this leaves subagents free to adapt to what they find. (A note from the docs: a subagent cannot spawn its own subagents — do not add `Agent` to a subagent's `tools`.)

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| "Subagents automatically inherit the parent's context" | Directly contradicts the "What subagents inherit" table: the parent's history and tool results are NOT passed. Context only through the `Task` prompt string. |
| "Run subagents one after another across different turns and call it parallel" | Parallel means several `Task` calls in ONE response. Turn-by-turn calls run sequentially, no speed-up. |
| "Pass a reference/pointer/short summary instead of the findings themselves" | The subagent sees only what is literally in the prompt. A pointer to the data ≠ the data; synthesis comes out empty. You need the full findings inline. |
| "Forget `Task`/`Agent` in `allowedTools`" | Then the subagent call is not auto-approved → the coordinator effectively cannot spawn anyone (or every call hits a permission prompt). |
| "Add a shared memory store so agents read each other's outputs" | Subagents do NOT share memory across calls; this is extra infrastructure instead of fixing the real gap — explicit context passing. |
| "Raise the subagent's `max_tokens` so it 'remembers' earlier results" | Confuses the output budget with a missing input. The data was never in the context — no token budget brings it back. |

## 🔑 Remember

- Spawning tool: `Task` (newer versions — `Agent`); it must also be in the coordinator's `allowedTools` for auto-approval.
- Subagent = fresh, isolated context. Gets: its own `AgentDefinition.prompt` + the `Task` prompt + the project `CLAUDE.md` + tool definitions. Does NOT get: the parent's history/results/system prompt.
- The context channel is exactly one — the `Task` prompt string. Pass the full findings inline.
- Parallel = several `Task` calls in one coordinator response.
- Attribution: structure the data, separating content from metadata (URLs, doc names, page numbers).
- `AgentDefinition`: `description`, `prompt`, `tools` (least privilege). The coordinator sets goals and quality criteria, not a step-by-step procedure. A subagent does not spawn subagents.

## 📝 Practice

*Answer for yourself first, then expand the "Answer".*

**Q1 · easy**
A coordinator agent in a Multi-Agent Research System is configured but never spawns any subagents — it always answers research questions itself. Its options set `allowedTools: ["Read", "Grep", "WebSearch"]` and it defines two `AgentDefinition` subagents. What is the most likely root cause?

A) The subagents' `description` fields are too long, so Claude ignores them.
B) `Task` (the subagent-invocation tool) is missing from the coordinator's `allowedTools`, so subagent calls are not auto-approved.
C) The coordinator needs a higher `max_tokens` budget to fit the subagents.
D) Subagents must be defined in `.mcp.json`, not via `AgentDefinition`.

<details><summary>Show answer</summary>

**Correct answer: B.** For the coordinator to call subagents without a permission prompt, the `Task`/`Agent` tool must be in `allowedTools`. It is not in the list → calls are not auto-approved, and the agent solves the task itself.
A — plausible, but the length of `description` affects subagent selection, not the ability to call at all; this is a "symptom instead of root cause" distractor. C — confuses the output budget with the calling mechanism. D — subagents are defined through `AgentDefinition`/`agents`, while `.mcp.json` is about MCP servers; a swapped mechanism.
</details>

**Q2 · medium**
A coordinator runs a web-search subagent and a document-analysis subagent, then spawns a synthesis subagent whose `Task` prompt says only: "Write the final research report based on the findings." The report comes back generic and ignores everything the first two agents discovered. What is the most effective fix?

A) Add a shared memory store so every subagent automatically reads the others' outputs.
B) Explicitly include the prior agents' complete findings in the synthesis subagent's `Task` prompt, formatted as structured data that keeps source URLs, document names, and page numbers attached to each finding.
C) Increase the synthesis subagent's `max_tokens` so it has room to recall the earlier results.
D) Re-run the synthesis subagent so it inherits the coordinator's conversation history on the second attempt.

<details><summary>Show answer</summary>

**Correct answer: B.** A subagent sees only what is literally in its `Task` prompt; the history and results of earlier agents are not inherited. You need to embed the full findings inline, structured so the source metadata (URLs, doc names, page numbers) stays attached — this preserves attribution for synthesis.
A — subagents do not share memory across calls; extra infrastructure instead of explicit passing. C — confuses the output budget with a missing input. D — inheriting the parent's history does not exist; a re-run adds nothing.
</details>

**Q3 · hard**
A research coordinator must investigate three independent subtopics as fast as possible. Each subtopic is handled by a separate subagent. Which orchestration produces genuinely parallel execution?

A) The coordinator emits three `Task` tool calls in a single response, one per subtopic.
B) The coordinator emits one `Task` call, waits for its result, then emits the next, repeating across three turns.
C) The coordinator defines all three subagents in `AgentDefinition` and relies on the SDK to run any defined agent automatically in parallel.
D) The coordinator spawns one subagent and instructs it to spawn the other two subagents itself.

<details><summary>Show answer</summary>

**Correct answer: A.** Parallelism is achieved by emitting several `Task` calls in one coordinator response — then the subagents run at the same time.
B — turn-by-turn calls across different turns = sequential execution, no speed-up (the classic "sequential passed off as parallel"). C — defining in `AgentDefinition` only makes the subagents available; the mere existence of a definition does not run them in parallel — you need explicit multiple `Task` calls. D — a subagent cannot spawn its own subagents; architecturally impossible.
</details>

### Tricky question **

**Q★★ (tricky):** A coordinator spawns five subagents by emitting five `Task` calls in one response. Each subagent searches a different source set and returns findings. To "save coordinator context," an engineer changes the design so each subagent, instead of returning its full findings, writes them to a shared scratch file and returns only the file path; the synthesis subagent is then told to "read the findings from the paths the others produced." Synthesis still comes back thin and mis-attributed. What is the precise reason this design is flawed?

A) Five parallel `Task` calls exceed the subagent limit, so some subagents silently fail to run.
B) The synthesis subagent's `Task` prompt carries only path pointers, not the findings themselves — and without the structured content+metadata passed explicitly, the only reliable parent→subagent channel (the `Task` prompt) is empty of the actual data and attribution.
C) Writing to a shared file is the correct pattern, so the real bug must be that `Task` was renamed to `Agent` and the old calls no longer fire.
D) The synthesis subagent inherited the coordinator's history but not the subagents', so it sees half the data.

<details><summary>Show answer</summary>

**Correct answer: B.** This is a combination of two principles at once: (1) the only reliable context channel is the `Task` prompt string, and (2) attribution is preserved only by structured passing of content together with metadata. By passing only path-pointers, the engineer repeated the "reference/pointer instead of the findings themselves" archetype: the data and its source metadata are not in the prompt, so synthesis comes out thin and with lost attribution. The right move — embed the full findings inline in a structured format (content + URLs/doc names/page numbers).
A — there is no "limit of 5" that silently drops subagents; a false cause. C — the rename `Task`→`Agent` is real (v2.1.63) and sounds "technically informed", but it is NOT the root cause: the problem is not the tool name but the empty context channel; plus "writing to a shared file is the correct pattern" is falsely presented as fact. D — inheriting history (neither the parent's, and certainly not other subagents') does not exist; a common misconception that the trap is built on.
</details>
