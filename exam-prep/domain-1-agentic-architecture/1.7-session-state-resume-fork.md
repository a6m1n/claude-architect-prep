# 1.7 — Manage session state, resumption, and forking

> **Domain 1: Agentic Architecture & Orchestration** (27%) · **Task Statement 1.7** | **Scenarios:** Developer Productivity, Code Generation | **What it tests:** when to resume a session (`--resume`), when to fork (`--fork-session` / `forkSession`), and when to start fresh and inject a structured summary because the tool results are stale.

## ⚡ TL;DR

- **A session = the conversation history**, not a snapshot of the file system: it holds the prompt, every tool call, and **their cached results** (file reads, grep, command output). Resuming *replays* that history.
- **`claude --resume <name>`** resumes a named session directly; `--continue` resumes the most recent one in the directory; `--from-pr <number>` resumes the one tied to a PR. In the SDK it is the `resume=<session-id>` option.
- **Resume when the prior context is mostly still valid.** If there were only a few edits, resume and **explicitly list the changed files** so the agent does a targeted re-analysis instead of a full re-exploration.
- **Stale tool results → start fresh + inject a structured summary** (conclusions + open questions). The docs themselves call this more reliable than dragging along a stale transcript.
- **`--fork-session` / `forkSession`** copies the history into a **new, independent** session from a *valid* shared baseline (two strategies in parallel). The original is unchanged; the link is **one-way**. It is a tool for divergent exploration, **not** a way to fix stale context.

## 📘 Theory

### What a session is and what resume does

A session is the accumulated history of the agent's work: your prompt, every tool call, **every tool result**, and every response. The SDK writes it to disk automatically (`~/.claude/projects/<encoded-cwd>/<session-id>.jsonl`). Resuming gives the agent the full prior context: the files it already read, the analysis it already did, the decisions it already made. CLI entry points:

| Command | What it does |
|---|---|
| `claude --continue` | resumes the most recent session in the current directory |
| `claude --resume` | opens the session picker |
| `claude --resume <name>` | resumes a **named** session directly |
| `claude --from-pr <number>` | resumes the session tied to a PR |
| `/resume` | switch between conversations from inside a session |

Naming: `claude -n auth-refactor` at start, `/rename` while running. In the Agent SDK the equivalent is `resume=<session-id>` (the ID comes from `session_id` in the init message / `ResultMessage`).

### Fresh session vs resume: the key design decision (context engineering)

Resuming **replays the cached tool results**. That is exactly what you want when the context is *mostly valid*: the agent does not re-read what it already knows. But after big code edits those cached reads go **stale** — the agent reasons about a snapshot that is no longer on disk. Resume does **not diff** the disk against the history: a human has to report the changes.

The context-engineering principle is — **durable facts matter more than lossy/stale raw material**. When the tool results are stale, it is more reliable to **start fresh and inject a structured summary** (what was concluded, what is still open), forcing the agent to re-read the *current* files. The official docs on resume across hosts recommend exactly this: "do not rely on session resume — capture the results you need as application state and pass them into the prompt of a fresh session; this is often more reliable than carrying transcript files around."

### The middle option: targeted re-analysis

If there were only a few edits, you need neither a blind resume nor a full fresh start. Resume the session and **explicitly name the changed files** — the agent does a targeted re-analysis instead of an expensive re-exploration of the whole module. This is the cheapest correct path when the context is almost valid.

### Forking: divergent branches from a shared baseline

`--fork-session` (CLI: `/branch`, or `--continue/--resume` + `--fork-session`; SDK: `forkSession: true` / `fork_session=True` together with `resume`) creates a **new** session that starts from a *copy* of the original's history and then diverges. The fork gets **its own** session ID; the original and its history **do not change**. You get two independent sessions that resume separately. The purpose is to explore an alternative without losing the original branch (for example, comparing two refactoring or testing strategies from one valid baseline).

Important details:
- A fork branches the **conversation**, not the file system. If the forked agent edits files, the edits are real and visible to any session in the same directory.
- The link is **one-way**: the branches do not sync with each other. Changes in the fork do not flow back into the original, and vice versa.
- Permissions granted "allow for this session" **do not carry over** to the new branch.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **"Always resume"** even when the tool results are stale | Resume replays a two-day-old cache: the agent reasons about renamed/moved files that no longer exist. With stale results, a fresh session + structured summary is more reliable. |
| **Resume after edits without naming the changed files** | Resume does **not** diff the disk against the history. Without an explicit list of changed files, the agent treats the old snapshot as current. You must either name the files (targeted) or start fresh. |
| **"Fork syncs both branches"** | A fork copies the history once and then diverges. The link is one-way; the branches are independent, with no back-sync. |
| **`fork_session` as a cure for stale context** | Right tool, wrong job. A fork is for divergent exploration from a *valid* baseline. Forking a stale session = copying the staleness into a new branch without updating anything. |
| **"Resume will detect the disk changes itself"** | There is no auto-detection. The SDK replays the saved history and does not compare it against the disk. |

## 🔑 Remember

- `claude --resume <name>` — named session; `--continue` — most recent; `--from-pr <number>` — by PR; `/resume` — picker. SDK: `resume=<session-id>`.
- Fork: CLI `/branch` or `--fork-session` (with `--continue`/`--resume`); SDK `forkSession: true` / `fork_session=True` together with `resume`. New ID, original untouched, link is one-way.
- A session persists as a **conversation**, not as a file system: the transcript lives in `~/.claude/projects/<encoded-cwd>/<session-id>.jsonl`.
- Stale tool results → **fresh session + structured summary** (durable conclusions) is more reliable than resuming raw stale context.
- A few edits → resume + **explicitly name the changed files** for targeted re-analysis.
- Resume does **not** auto-detect disk changes; session permissions are not inherited by the fork.

## 📝 Practice

*Answer on your own first, then expand the "Answer".*

**Q1 · easy.** You named a session `payments-investigation` yesterday and want to continue exactly that conversation today (not the most recent one in the directory). Which command resumes it directly?

A) `claude --continue`
B) `claude --resume payments-investigation`
C) `claude --fork-session payments-investigation`
D) `claude --from-pr payments-investigation`

<details><summary>Show answer</summary>

**Correct answer: B.** `claude --resume <name>` resumes a named session directly (an exact name match resumes it right away).
A — `--continue` takes the *most recent* session in the directory, not a specific named one (the "wrong entry point" trap). C — `--fork-session` does not resume; it creates a *new* branch-copy (tool substitution). D — `--from-pr` expects a **PR number**, not a session name (wrong argument).
</details>

**Q2 · medium.** You want to compare two refactoring strategies (extract-class vs. introduce-interface) starting from the *same* analysis your session already produced, keeping the original thread intact so you can return to it. What's the most effective approach?

A) Run `--resume` on the session twice in two terminals and let both threads interleave into one transcript.
B) Use `--fork-session` from the analysis session to create an independent branch per strategy; the original stays unchanged.
C) Start two fresh sessions from scratch and re-run the full analysis in each.
D) Resume the session, try strategy one, then `/clear` and try strategy two in the same session.

<details><summary>Show answer</summary>

**Correct answer: B.** A fork from a valid shared baseline is exactly the "divergent branches" scenario: each strategy in its own independent session, the original untouched.
A — resume without a fork in two terminals **mixes** the messages into one transcript (the docs warn about this directly) — no clean comparison. C — throws away the analysis already done, expensive and not "from a shared baseline" (the re-exploration anti-pattern). D — `/clear` wipes the context, so the analysis baseline is lost for the second strategy.
</details>

**Q3 · hard.** Two days ago a session read and analyzed ~15 files in `payments/`. Since then a teammate merged a large refactor that renamed functions and moved several of those files. You must continue the investigation reliably. What's the **root-cause-correct** move?

A) `--resume` the session and immediately ask the follow-up, trusting the cached reads — the agent will auto-detect that disk changed.
B) `--fork-session` the prior session so the stale analysis is branched before you continue.
C) Start a fresh session and seed it with a structured summary of prior findings, letting the agent re-read the current files from disk.
D) `--resume` the session and ask the follow-up without mentioning the refactor, since resume always reflects the latest disk state.

<details><summary>Show answer</summary>

**Correct answer: C.** The two-day-old tool results describe code that no longer exists; resume would just *replay* them. A fresh session + structured summary keeps the durable conclusions and forces a re-read of the *current* files — the staleness is fixed at the root.
A — there is no auto-detection of disk changes (the "resume will notice it itself" trap). B — forks the staleness into a new branch without updating anything (right tool, wrong job). D — resume replays the *saved history*, not "the latest disk state" (false assumption).
</details>

### Tricky question **

**Q★★ (tricky):** Your session analyzed a service and you `--fork-session` it to prototype an aggressive change. In the fork, the agent runs `Edit` and rewrites `config.yaml` on disk. You decide the fork's direction is wrong, so you `--resume` the **original** session by its ID to continue the safe path. Why might the original session now behave incorrectly, and what's the precise reason?

A) Resuming the original automatically reverts `config.yaml`, so nothing is wrong — the fork's edits were rolled back when you switched.
B) The fork edited the real file on disk; forking branches the **conversation**, not the filesystem, so the original session's next disk read sees the fork's mutated `config.yaml` even though its conversation history is untouched.
C) The original session is corrupted because forks sync bidirectionally and the fork's messages were merged back into the original transcript.
D) Resuming by ID fails entirely because a fork deletes the original session's `.jsonl` file.

<details><summary>Show answer</summary>

**Correct answer: B.** The subtlety is in separating two things: a fork isolates the **conversation history** (the original transcript really is untouched), but **not** the file system. The `Edit` in the fork wrote the real `config.yaml`, and any session in the same directory — including the resumed original — will see the changed file on its next read. The "conversation" context is clean, but the "state of the world" (the disk) is not; this is the same kind of stale context as at the start, but in reverse: the history is valid while the cache/disk have diverged.
A — there is no auto-revert: rolling back file changes needs a separate mechanism (file checkpointing); resume itself rolls back nothing. C — the fork link is **one-way**, there is no merge back into the original (the classic "fork syncs" distractor). D — a fork creates a *new* `.jsonl` and does not delete the original; resume by ID works.
</details>
