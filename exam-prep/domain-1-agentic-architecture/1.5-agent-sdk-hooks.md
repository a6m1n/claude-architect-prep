# 1.5 — Apply Agent SDK hooks for tool call interception and data normalization

> **Domain 1: Agentic Architecture & Orchestration** (27%) · **Task Statement 1.5** | **Scenarios:** Customer Support Resolution Agent | **What it tests:** whether you can pick a deterministic mechanism (`hooks`) instead of a probabilistic one (prompt) when a business rule must never be broken, and tell apart `PreToolUse` (intercept the call) and `PostToolUse` (normalize the result).

## ⚡ TL;DR

- A "never" rule → code, not a prompt. `hooks` give a **deterministic** guarantee; system prompt / few-shot / `temperature` only give **probabilistic** compliance.
- `PreToolUse` intercepts the **outgoing** tool call: it reads `tool_input`, returns `permissionDecision: "deny"` to block (for example, refund > $500), and reroutes to human-escalation.
- `PostToolUse` intercepts the tool **result**: it normalizes mixed formats (Unix timestamps, ISO 8601, numeric status codes from different MCP tools) into one format before the model sees it — via `updatedToolOutput`.
- `deny` always wins: if even one hook returns `deny`, the call is blocked regardless of the other hooks and permission rules.
- MCP tools are matched as `mcp__<server>__<action>`, so you can attach the interception precisely to one specific tool (`mcp__refunds__process_refund`).

## 🧠 In plain words

> **Analogy:** 🛂 a hook = a customs officer at the model's border. On the way in, the officer brings the goods up to standard (`PostToolUse` cleans the tool result); on the way out, the officer does not let forbidden things through (`PreToolUse` blocks a bad call).
>
> A hook is code that the SDK runs around every tool call, OUTSIDE the model loop, so the model cannot talk it out of its decision. `PostToolUse` catches the **result** and normalizes the mixed formats (Unix time, ISO 8601, numeric statuses) into one before the model sees it. `PreToolUse` catches the **outgoing call** before it runs and returns `permissionDecision: "deny"` if it breaks a rule (refund > $500 → escalation). The catch: a "never" rule in a prompt is only a request (probabilistic); only code gives a guarantee.
>
> 🚩 **Detector:** if the answer to a "must never" rule suggests a system prompt / few-shot / lowering `temperature` — that is **symptom-not-root-cause** (treating it with probability); the right answer is a `PreToolUse` hook with `deny` (determinism, lever 5).
>
> 🧷 **Reflex:** Pre — does not let a bad call through; Post — tidies up the result; "never" lives in code, not in the prompt.

## 📘 Theory

### Why a hook, not a prompt: determinism vs probability

The core of this topic is **determinism-vs-probabilistic**. An instruction in a prompt ("Never issue a refund above $500") only shifts the probability distribution toward the wanted behavior. The model will *usually* obey, but an edge-case input, a long context, an ambiguous wording, or a jailbreak can override it. There is no guarantee.

A `hook` is callback code that the SDK runs in response to an agent event (a tool call, a session start, a stop). It runs **outside the model loop**, as plain code, and the model cannot override its verdict. So when a rule must always hold (financial limits, compliance, protecting `.env` and system directories), its place is in the hook's code, not in the prompt text. This is the same principle as in "Building effective agents": the predictability of well-defined tasks comes from deterministic workflow code, not from a model-driven decision.

### `PreToolUse` — intercept the outgoing call (enforcement)

`PreToolUse` fires **before** the tool runs and can block it or change its input. The callback receives `tool_name` and `tool_input`, checks the arguments with plain code, and returns in `hookSpecificOutput`:

- `permissionDecision: "deny"` — hard-blocks the call;
- `permissionDecisionReason` — explains the reason to the model so it does not try again;
- a redirect into an alternative workflow (human-escalation).

Example from the task: a hook on `process_refund` reads the amount, and when it is `> $500` returns `deny` and routes to escalation. That is a guarantee, not a wish.

### `PostToolUse` — normalize the result (data normalization)

`PostToolUse` fires **after** the tool runs successfully and intercepts its **result** before the model processes it. This is a *context engineering* tool: different MCP tools return time as a Unix timestamp, as an ISO 8601 string, statuses — as numeric codes. If the model has to sort out this zoo of formats itself, that is noisy and unreliable. `PostToolUse` brings everything to one format in code:

- `updatedToolOutput` — fully replaces the tool output with the normalized version;
- `additionalContext` — appends context to the result without replacing it.

### Decision priority and matchers

When several hooks/permission rules apply, the priority is: **deny > defer > ask > allow**. A single `deny` blocks the operation regardless of the rest. A matcher is a regex over the **tool name** (not over the arguments); to filter by amount/path, check `tool_input` inside the callback. MCP tools are named `mcp__<server>__<action>`, which lets you intercept one specific tool precisely.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **A hard rule in the system prompt** ("Never refund > $500") | A prompt is probabilistic guidance, not a guarantee. Under an edge-case input, a long context, or a jailbreak the model may issue the refund. For "must never" this is always the wrong answer. |
| **A few-shot example** of declining a $600 refund | The same failure as the system prompt: an example only shifts the distribution, it does not *enforce* anything. One example does not cover all cases > $500. |
| **Lower the `temperature`** for "reliable" policy compliance | `temperature` reduces sampling randomness, but it **does not encode the $500 rule at all**. A confidently wrong refund will still be issued. Tuning sampling is not control over compliance. |
| **Confusing `PreToolUse` with `PostToolUse`** | `PreToolUse` is intercepting the *call* before it runs (enforcement, blocking). `PostToolUse` is intercepting the *result* after it runs (normalization). Blocking a refund in `PostToolUse` is too late: the call has already run. There is nothing to normalize in `PreToolUse` — the result does not exist yet. |

## 🔑 Remember

- **`PreToolUse`** = before execution, intercept the **call** → `permissionDecision: "deny"` + `permissionDecisionReason`, optionally `updatedInput` (with `"allow"`).
- **`PostToolUse`** = after execution, intercept the **result** → `updatedToolOutput` (replace) or `additionalContext` (append).
- Priority: **`deny` > `defer` > `ask` > `allow`**. Any `deny` blocks.
- A matcher = regex over the **tool name**; MCP = `mcp__<server>__<action>`; filter by arguments inside the callback via `tool_input`.
- Hooks = a deterministic guarantee; prompt / few-shot / `temperature` = probabilistic. "Must never" → hook.

## 📝 Practice

*Answer it yourself first, then expand "Answer".*

**Q1 · easy**
A Customer Support agent calls an MCP tool `mcp__refunds__process_refund`. Company policy: a refund over $500 must NEVER be issued by the agent and must go to human escalation. Which approach guarantees the cap cannot be violated?

A) Add to the system prompt: "Never issue a refund above $500; escalate instead."
B) Register a `PreToolUse` hook on the tool that returns `permissionDecision: "deny"` when the amount exceeds $500 and redirects to escalation.
C) Add a few-shot example showing the agent declining a $600 refund.
D) Lower the model's `temperature` so it follows the policy more reliably.

<details><summary>Show answer</summary>

**Correct answer: B.** A "never" rule requires a deterministic guarantee, which only code outside the model loop can give. `PreToolUse` reads `tool_input`, and when it is `> $500` returns `deny` and redirects to escalation.
- A — *guaranteed-rule-in-prompt*: the system prompt is probabilistic, not a guarantee.
- C — *few-shot*: an example shifts the distribution, it enforces nothing.
- D — *temperature-as-control*: sampling does not encode the $500 rule at all.
</details>

**Q2 · medium**
Three MCP tools return time fields inconsistently: one as a Unix timestamp, one as an ISO 8601 string, one as a numeric status code. The agent keeps misreading them. What is the most effective fix?

A) Use a `PreToolUse` hook to rewrite each tool's input arguments into a common format.
B) Add a `PostToolUse` hook that normalizes each tool's result into one format via `updatedToolOutput` before the model sees it.
C) Add a system-prompt section explaining all three formats so the model can convert them itself.
D) Lower `temperature` so the model parses the formats more consistently.

<details><summary>Show answer</summary>

**Correct answer: B.** This is normalizing the **result** — the job of `PostToolUse`, which replaces the output via `updatedToolOutput` before the model processes it.
- A — *PreToolUse/PostToolUse confusion*: on the way in there is no result yet, there is nothing to normalize.
- C — *guaranteed-rule-in-prompt*: format parsing by the model is probabilistic and noisy; deterministic normalization is done by code.
- D — *temperature-as-control*: sampling does not control format parsing.
</details>

**Q3 · hard**
A compliance rule forbids the agent from ever writing to the `mcp__db__execute_write` tool with table `audit_log`. The team added a strongly worded system-prompt rule and lowered `temperature`, but an edge-case prompt still triggered a forbidden write. What is the root-cause fix?

A) Strengthen the system-prompt wording and add two few-shot refusal examples.
B) Move the rule into a `PostToolUse` hook that flags the write after it happens.
C) Register a `PreToolUse` hook on `mcp__db__execute_write` that inspects `tool_input`, returns `permissionDecision: "deny"` for `audit_log`, and sets `permissionDecisionReason`.
D) Reduce `temperature` further and add `top_p` capping to make refusals deterministic.

<details><summary>Show answer</summary>

**Correct answer: C.** The root cause is that enforcement was handed to a probabilistic mechanism. A guarantee is achieved only by code: `PreToolUse` blocks the call before it runs, based on the contents of `tool_input`.
- A — *prompt/few-shot*: strengthening the text does not change the probabilistic nature.
- B — *Pre/Post confusion*: `PostToolUse` fires after the write — the forbidden write has already happened.
- D — *temperature-as-control*: neither `temperature` nor `top_p` makes rule compliance deterministic.
</details>

### Tricky question **

**Q★★ (tricky):** A `PreToolUse` hook on `mcp__refunds__process_refund` returns `permissionDecision: "deny"` for amounts over $500. To also redirect those cases, an engineer adds a SECOND `PreToolUse` hook (same matcher) that returns `permissionDecision: "allow"` plus `updatedInput` rerouting the call to a manager-approval queue. In testing, over-$500 refunds are still blocked and never reach the queue. What is the precise reason, and the correct fix?

A) The two hooks run in a guaranteed order, so the first `deny` always executes before the `allow`; reorder them so `allow` is registered first.
B) When multiple hooks match, `deny` takes priority over `allow` regardless of order, so the blocking hook always wins; do the redirect inside the single blocking hook's logic (or in the downstream escalation path), not as a competing `allow`.
C) `updatedInput` only works in `PostToolUse`, so the redirect silently fails; move the second hook to `PostToolUse`.
D) The matcher `mcp__refunds__process_refund` is too specific; broaden it to `mcp__refunds__.*` so both hooks fire.

<details><summary>Show answer</summary>

**Correct answer: B.** This combines two principles: decision priority and the non-deterministic order of hooks. When several hooks match, the priority matrix is **`deny` > `defer` > `ask` > `allow`**: a single `deny` blocks the operation regardless of what the others returned. So the competing `allow` from the second hook will never "beat" the `deny`. You cannot do the redirect as a competing `allow` — it must be built into the logic of the blocking hook itself (return `deny` + a `permissionDecisionReason` that routes the flow into the escalation queue), or handled in a downstream workflow.

- A — tempting, but wrong on the precise reason: hooks for one event run in parallel, and the completion order is **non-deterministic**; reordering guarantees nothing. And even in any order, `deny` still has priority.
- C — a factual error: `updatedInput` belongs to `PreToolUse` (together with `permissionDecision`), not to `PostToolUse`; moving it to `PostToolUse` loses the very ability to change the input.
- D — broadening the matcher only makes the hooks fire on more tools; it does not resolve the `deny` vs `allow` conflict.
</details>
