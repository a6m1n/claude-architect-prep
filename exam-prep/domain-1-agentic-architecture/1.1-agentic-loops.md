# 1.1 — Design and implement agentic loops for autonomous task execution

> **Domain 1: Agentic Architecture & Orchestration** (27%) · **Task Statement 1.1** | **Scenarios:** Customer Support, Multi-Agent Research, Developer Productivity | **What it tests:** agent loop control is built on `stop_reason`, not on text parsing or an arbitrary iteration limit.

## ⚡ TL;DR

- The agent loop is a contract, not a guess: `while stop_reason == "tool_use"` run the tools, append `tool_result` to the history, repeat; on `"end_turn"` — stop and give the final answer.
- Every `tool_result` MUST be appended to `messages`, otherwise the model cannot reason about the next step based on the new data.
- The model itself decides which tool to call next (model-driven) — not a hard-coded decision tree and not a fixed sequence of calls.
- Anti-patterns: parsing prose for words like "done", using `max_iterations` as the MAIN stop signal, treating the presence of any text block as a sign of completion.
- `max_iterations` is a safety backstop, not a completion mechanism.

## 📘 Theory

### The agent loop lifecycle

One pass of the loop: (1) you send a request to Claude with a `tools` array; (2) you read `stop_reason`; (3) if it is `"tool_use"` — you take the `tool_use` blocks from the response, run the tools, and build `tool_result` blocks; (4) you append both the assistant response and the `tool_result` to `messages`; (5) you send a new request and go back to step 2. The loop ends when `stop_reason` becomes anything that is NOT `"tool_use"`: `"end_turn"` (the final answer is ready), and also `"max_tokens"`, `"stop_sequence"`, `"refusal"` — you must handle the last ones explicitly.

Why `stop_reason` specifically? It is a contract field that the API emits exactly for this decision point. It is deterministic: either the model asks for tools, or it does not. Any other signal (text, a counter) is a probabilistic or arbitrary proxy on top of a deterministic fact you already have.

### Why tool results are appended to the history

The model does not "remember" calls between requests — the state lives only in `messages`. If you do not add the `tool_result`, the next request goes out without the new data, and the model will either repeat the same call or start to hallucinate the result. The principle is **context engineering**: a new observation from the environment (tool output) must become part of the context so the decision about the next step is well-founded. In the official wording, an agent is "an LLM using tools based on environmental feedback in a loop".

### Model-driven versus a fixed sequence

The key difference from "Building effective agents": **workflows** orchestrate the LLM along paths written in code in advance; **agents** are systems where the LLM itself dynamically directs its own process and tool choices. When the number of steps is unpredictable (a refund may need 1 call or 5, a research agent — an unknown number in advance), a fixed decision tree breaks. A model-driven loop on `stop_reason` handles this open-ended uncertainty naturally: the model decides the next action from the accumulated context on each turn.

### Where this is already implemented

In the "raw" Client SDK you write `while response.stop_reason == "tool_use": ...` yourself. The Claude Agent SDK runs this same loop for you (the same contract, just encapsulated) — but the exam checks that you understand which signal it spins on.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **Loop until `max_iterations`** | The limit is a backstop against an infinite loop, not a completion signal. As the MAIN mechanism it either cuts off a legitimate multi-step case or wastes extra calls on a simple one. Completion is determined by `stop_reason == "end_turn"`. |
| **Stop on any assistant text** | Claude often emits explanatory text IN THE SAME turn as a `tool_use` block. The presence of text does not mean the turn is over — only `"end_turn"` means that. |
| **Parse prose for "done"/"finished"** | Free text is unreliable: the model may write "let me check" in the middle of the work, or conversely fail to write the keyword. If you are writing a regex to extract the decision from the model's output, that decision should have been read from `stop_reason`. |
| **Hard-code a fixed sequence of calls / a decision tree** | This turns the agent into a workflow. With an unpredictable number of steps, a rigid sequence does not cover the real branches. The decision "which tool next" is delegated to the model. |
| **Not appending `tool_result` to the history** | The model will not see the result and will loop on re-calling or hallucinate the answer. |

## 🔑 Remember

- Loop control: `while stop_reason == "tool_use"` — by the API field, not by prose, a counter, or the presence of text.
- On each iteration, BOTH the assistant response AND the `user` message with `tool_result` blocks are appended to `messages`.
- `"end_turn"` is the authoritative completion signal; handle `"max_tokens"`, `"stop_sequence"`, `"refusal"` explicitly.
- Server-side tools (web_search, etc.) spin their own internal loop; their pause on a limit is `stop_reason: "pause_turn"` (don't confuse it with the client-side `"end_turn"`).
- `max_iterations` = safety backstop, not the primary stop.
- Model-driven > a fixed sequence when the number of steps is unknown in advance.

## 📝 Practice

*Answer on your own first, then expand "Answer".*

**Q1 · easy**
You are writing the control loop for a Claude Agent that uses client-executed tools. After each API response, what is the correct primary signal to decide whether to execute another tool or return the final answer to the user?

A) Check whether the response contains any assistant `text` block; if it does, return it as the final answer.
B) Continue while `stop_reason` is `"tool_use"`; stop when it is `"end_turn"`.
C) Keep looping until a `max_iterations` counter reaches its limit, then return the last response.
D) Scan the assistant's text for a completion phrase like "task complete" and stop when it appears.

<details><summary>Show answer</summary>

**Correct answer: B.** `stop_reason` is the API contract field for this decision point: `"tool_use"` means "run the tools and continue", `"end_turn"` means "the final answer is ready".
A — the "text = completion" trap: text often comes in the same turn as the `tool_use` block. C — the "iteration cap as primary stop" trap: the limit is a backstop, not a completion signal. D — the "prose parsing" trap: an unreliable probabilistic proxy on top of the deterministic `stop_reason`.
</details>

**Q2 · medium**
A Customer Support agent intermittently stops mid-resolution and replies to the customer before finishing a refund. The loop currently terminates as soon as the model's response includes a text block. What is the root cause and the fix?

A) The model is under-prompted; add "do not stop until the refund is done" to the system prompt and keep the text-based termination.
B) The conversation history is too long; truncate it and keep terminating on text presence.
C) Termination keys on text presence, but Claude emits text alongside `tool_use`; drive termination off `stop_reason == "end_turn"` instead.
D) The `max_iterations` cap is too low; raise the cap so the loop runs longer.

<details><summary>Show answer</summary>

**Correct answer: C.** The root cause is that the terminator looks at the presence of text, while Claude normally emits explanatory text IN THE SAME turn as the `tool_use` block, so the loop breaks prematurely. It is fixed by switching the condition to `stop_reason`.
A — treats the symptom with a prompt instead of the root control-flow error (root-cause-over-symptom). B — irrelevant, the length of the history has nothing to do with it. D — the iteration limit does not control this termination at all; that is a different mechanism.
</details>

**Q3 · medium**
Your loop executes the tools Claude requested but on the next iteration Claude repeatedly asks for the same tool with the same inputs and never converges. What is the most likely defect in the loop?

A) `tool_choice` is set to `"auto"`; force `{"type": "tool"}` to stop the repetition.
B) The `tool_result` blocks are not being appended to the conversation history, so the model never sees the outcome.
C) The model needs a higher `max_tokens`; raise it so it can finish.
D) The tools should be run in a fixed predetermined sequence to avoid re-asking.

<details><summary>Show answer</summary>

**Correct answer: B.** The agent's state lives only in `messages`. If the `tool_result` is not appended to the history, the next request goes out without the result — the model does not see that the call has already been made, and it repeats it (context engineering: an observation from the environment must reach the context).
A — `tool_choice` controls WHETHER to call a tool, not the passing of the result; forcing it will make the loop worse. C — `max_tokens` is about the length of the output, not about a lost result. D — a fixed sequence replaces the model-driven loop and does not fix the cause.
</details>

**Q4 · hard**
A Multi-Agent Research orchestrator spawns subagents whose number of tool calls is unpredictable per query. The team proposes hardcoding a decision tree: "call `search`, then `read`, then `summarize`, then stop." Why is the `stop_reason`-driven model-driven loop the more effective design here?

A) A decision tree is always faster, so it should be preferred whenever latency matters.
B) The number of steps is unpredictable; a fixed sequence can't cover the real branches, whereas the model decides the next action from context each turn and the loop stops on `"end_turn"`.
C) Decision trees can't call MCP tools, so they are technically impossible here.
D) The model-driven loop guarantees fewer tokens than a decision tree in every case.

<details><summary>Show answer</summary>

**Correct answer: B.** When the number and order of steps are unknown in advance, a fixed sequence (workflow) does not cover the real branches; a model-driven agent dynamically chooses the next tool from the accumulated context, and the loop ends on `stop_reason == "end_turn"`.
A — about speed, and wrong as an absolute (a workflow is not "always faster"). C — factually false: decision trees can also call MCP tools. D — overcomplication / a false guarantee: nobody promises fewer tokens "in every case".
</details>

### Tricky question **
**Q★★ (tricky):** Your research agent enables the **server-side** `web_search` tool. After one API request, the response comes back with `stop_reason: "pause_turn"` and partial results. A teammate says: "Our loop only continues while `stop_reason == "tool_use"`, so on `pause_turn` we should treat the turn as finished and return what we have." What is the correct handling?

A) Treat `pause_turn` as completion and return the partial results — only `"tool_use"` should ever continue the loop.
B) Re-send the conversation including the paused response so the model continues its server-side work; `pause_turn` means the work is not finished.
C) Switch the agent to a `max_iterations` cap, since `stop_reason` is unreliable for server tools.
D) Append an empty `tool_result` block and resend, the same way you would for a client tool.

<details><summary>Show answer</summary>

**Correct answer: B.** The subtlety is in mixing up two loops. The client-side loop does indeed continue on `"tool_use"`, but `web_search` is a server-side tool: Anthropic spins its own internal loop, and when it hits the iteration limit, `stop_reason: "pause_turn"` is returned — this is "the work is not finished", NOT completion. The right move is to re-send the conversation together with the paused response so the model continues from where it stopped.
A — the trap: applies the client-loop rule to a server tool and loses the result; `pause_turn` ≠ `end_turn`. C — replacing a deterministic signal with an arbitrary limit (that very anti-pattern), and besides `stop_reason` is in fact reliable here. D — for server-side tools you do NOT construct a `tool_result` (the server does that); an empty `tool_result` is the wrong continuation mechanism.
</details>
