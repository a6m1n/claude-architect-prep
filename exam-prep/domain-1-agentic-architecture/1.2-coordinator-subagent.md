# 1.2 — Orchestrate multi-agent systems with coordinator-subagent patterns

> **Domain 1: Agentic Architecture & Orchestration** (27%) · **Task Statement 1.2** | **Scenarios:** Multi-Agent Research System | **What it tests:** Hub-and-spoke — the coordinator owns routing, decomposition, and error handling; decomposition that is too narrow is a typical root cause of incomplete topic coverage.

## ⚡ TL;DR

- **Hub-and-spoke:** one coordinator (lead agent) routes *all* inter-agent communication — this gives observability, consistent error handling, and control over what each subagent sees.
- **Context is isolated:** a subagent starts in a fresh conversation and does **not** inherit the coordinator's history. The only "parent → subagent" channel is the prompt string of the `Agent` tool. Pass paths, errors, and decisions into it explicitly.
- **The coordinator owns** decomposition, delegation, result aggregation, and the **dynamic** decision about which subagents to call (don't always push everything through the full pipeline).
- **Root cause #1 of incomplete coverage:** the coordinator's decomposition of the topic is too narrow — even when *every* subagent performed flawlessly.
- **Iterative loop:** the coordinator evaluates the synthesis for gaps → re-issues targeted search/analysis requests → re-runs synthesis until coverage is sufficient.

## 🧠 In plain words

> **Analogy:** 🗼 the airport control tower: every crew talks only through it, and the tower also does the task slicing ("you build the roof, you build the walls").
>
> The hub's strength is **structural**: it sees everything, fixes errors in one place, decides who knows what, and **owns the slicing of the task** into pieces. The catch: retry, batch, and cache are operational conveniences — you can bolt them on anywhere, they are not unique to the hub. And: a subagent starts "from a blank slate" — it does not inherit the coordinator's history, so put what it needs into its prompt explicitly.
>
> 🚩 **Detector:** if an answer says "retry/batch/cache is possible **only** through the coordinator," that's a **false dichotomy** (true as a perk, but not structural); the right answer is that the hub's structural advantage = visibility + consistent error handling + information gating + ownership of decomposition.
>
> 🚩 **Detector 2:** all subagents performed correctly, yet the topic is covered incompletely → blaming the worker is **symptom-not-root-cause**; the right answer is that the root is in the coordinator's **narrow decomposition** (read the failure backwards).
>
> 🧷 **Reflex:** the crews did their work, the house is incomplete → blame whoever sliced the tasks, not the bricklayer.

## 📘 Theory

### Why hub-and-spoke (orchestrator-workers)

A "star" topology: the lead agent (coordinator) at the center, specialized subagents on the spokes. The coordinator is not just a dispatcher — it is the system's **control plane**. Routing *all* traffic through one node gives three structural properties that are impossible to get from direct agent↔agent edges:

1. **Observability** — all interactions are visible in one place.
2. **Consistent error handling** — a single point for detecting, logging, and recovering from failures.
3. **Controlled information flow** — the coordinator decides which *slice* of context each worker gets (context isolation as policy, not chance).

Principle: choose centralization when you need a single source of truth and a single policy; swap it for direct edges only if point-to-point latency clearly outweighs the loss of control.

### Isolated context of subagents

Each subagent runs in its own fresh context window. It **receives**: its own system prompt (`AgentDefinition.prompt`), the prompt string of the `Agent` tool, the project `CLAUDE.md`, and tool definitions. It does **not** receive: the parent's conversation history and its tool results, or the parent's system prompt. Intermediate tool calls stay inside the subagent — only the final message is returned outward. This reduces bloat in the main context, but it means: **there is no shared memory by default** — whatever a subagent needs, you put into its prompt explicitly.

### The coordinator owns decomposition and routing

The coordinator's first job is to map *the whole* problem space and break it into subtasks that **together** cover every relevant domain of the topic. Each subagent is given an objective, an output format, a set of tools/sources, and clear task boundaries. The coordinator also **dynamically** decides which subagents are needed for the complexity of the query — a simple query does not have to go through the full pipeline.

### Root cause of incomplete coverage — narrow decomposition

Coverage is determined **at the decomposition stage, not at execution**. If the coordinator never assigned a domain, no one is responsible for it, and no flawless downstream agent will recover it. That is why the pattern "all subagents succeed, but the aggregate is incomplete" is a symptom of *decomposition quality*, not of worker quality. Read the failure mode "backwards": components are correct, but scope was under-covered ⇒ the defect is in *how the task was sliced*, further upstream.

### Iterative refinement (refinement loop)

The coordinator does not have to stop after the first synthesis. It evaluates the synthesis output for coverage gaps, **re-delegates** targeted requests to search/analysis subagents, and **re-runs** synthesis until coverage is sufficient. This cures the risk of narrow decomposition at execution time, not only at design time.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| :-- | :-- |
| **Blame the synthesis / web-search / document-analysis subagent** for incomplete coverage | This is classic Sample Q7 (answer B / "narrow decomposition"). If the logs show a domain was *never assigned*, then search did not search for it, analysis did not filter it, and synthesis synthesized from what it got. The root is the coordinator, not the worker. |
| **"Subagents share memory / inherit context automatically"** | No. A subagent's context is fresh; the parent's history and tool results are **not passed**. The only channel is the `Agent` prompt string. Any answer relying on "shared memory" is a trap. |
| **"Always push the full pipeline regardless of complexity"** | The coordinator should dynamically choose subagents for the complexity of the query. A rigid full pipeline on every query is unnecessary latency/cost and an orchestration anti-pattern. |
| **False dichotomy: "retries/batching are possible only through the coordinator"** | Retry and batching are operational techniques implementable at any layer (even a direct call can wrap them). They are *not* a structural advantage of hub-and-spoke. The structural one is observability + consistent error handling + information gating. |
| **"The synthesis agent just lacks an instruction to look for gaps"** | Sounds like a best practice (and a gap-check is useful!), but if a domain is excluded *before* synthesis, the omission is already "baked in." This is not the *originating cause*, just treating a symptom downstream. |

## 🔑 Remember

- The advantages of hub-and-spoke are **structural**: observability, consistent error handling, controlled information flow — a consequence of routing through one node.
- Subagent: **fresh context**, does not inherit the parent's history/tool results; the channel is only the `Agent` tool prompt. Intermediate steps don't come out; the final message is returned.
- The coordinator owns: task decomposition, delegation, result aggregation, **dynamic** selection of subagents.
- "All subagents succeed, but coverage is incomplete" ⇒ the coordinator's **narrow decomposition** (Sample Q7 = the B/D equivalent of "decomposition too narrow").
- Treatment at execution: the coordinator evaluates the synthesis → re-delegates targeted requests → re-runs synthesis until coverage is sufficient.
- For **dozens to hundreds** of agents use the `Workflow` tool (orchestration in a script, outside the conversation context), not turn-by-turn delegation with subagents.

## 📝 Practice

*Answer on your own first, then expand "Answer".*

**Q1 · easy**
A colleague proposes that the document-analysis agent send its output directly to the synthesis agent, bypassing the coordinator. What is the main advantage of keeping the coordinator as the central hub for all subagent communication?

A) It batches subagent requests together to reduce the total number of API calls and latency.
B) The coordinator can observe all interactions, handle errors consistently, and control what information each subagent receives.
C) Routing through the coordinator enables automatic retry logic that direct agent-to-agent calls cannot support.
D) Subagents have isolated memory, so only the coordinator can perform the serialization needed for them to communicate.

<details><summary>Show answer</summary>

**Correct answer: B.** These are the structural advantages of the topology: visibility into all interactions, consistent error handling, and control over what each subagent sees.
- A — *batching* — an operational optimization available at any layer, not a defining property of hub-and-spoke.
- C — *false dichotomy*: retry can be implemented at any level, it is not unique to the coordinator.
- D — *invented constraint*: memory isolation does not make serialization an exclusive function of the coordinator.
</details>

**Q2 · medium**
A multi-agent research system processes the topic "Impact of AI on creative industries." Every subagent succeeds: web search finds relevant articles, document analysis summarizes papers correctly, synthesis produces coherent output. Yet the report covers only visual arts and omits music, writing, and film. The coordinator's logs show it split the topic into "AI in digital art," "AI in graphic design," and "AI in photography." What is the most likely root cause?

A) The web-search agent's queries are not comprehensive enough and need to cover more sectors.
B) The document-analysis agent filtered out non-visual sources due to overly restrictive relevance criteria.
C) The coordinator's task decomposition was too narrow, so subagent assignments don't cover all relevant domains.
D) The synthesis agent lacks instructions to detect coverage gaps in the findings it received.

<details><summary>Show answer</summary>

**Correct answer: C.** The logs show it directly: the split covered only the visual arts; music/writing/film were never assigned. Coverage is determined at decomposition — the root is at the coordinator.
- A — search only searches within the assigned subtasks; the limitation is in the assignment, not in the breadth of the queries.
- B — a filter cannot remove what was never gathered under the missing subtasks.
- D — a gap-check is useful, but this is a downstream symptom; the omission is "baked in" before synthesis.
</details>

**Q3 · medium**
You build a research coordinator that spawns three subagents. You notice that a subagent keeps re-deriving a constraint the coordinator already established earlier in its own conversation ("only sources after 2023"). What is the most likely cause and fix?

A) Subagents inherit the parent's conversation history, so the constraint is being overwritten — increase the subagent's context window.
B) Subagents start with isolated context and do not inherit the coordinator's history; include the constraint explicitly in the Agent prompt passed to the subagent.
C) The coordinator must enable a shared-memory flag so all subagents read the same conversation state.
D) The synthesis step should re-apply the constraint, so the fix belongs in the synthesis agent, not the delegation.

<details><summary>Show answer</summary>

**Correct answer: B.** A subagent starts in a fresh context and does not see the coordinator's history; the only channel is the `Agent` prompt string, so the constraint must be placed there explicitly.
- A — there is no inheritance of history; enlarging the window will not pass what was never in the context.
- C — a "shared-memory flag" is invented: there is no shared memory by default.
- D — this shifts the problem downstream without removing the cause (the subagent already worked without the constraint).
</details>

**Q4 · hard**
A coordinator routes *every* query through a fixed pipeline: web-search → document-analysis → synthesis, regardless of the query. Simple factual lookups now incur high latency and cost, while broad survey topics still come back incomplete. Which single change best addresses *both* problems?

A) Cache subagent outputs so repeated factual lookups skip the pipeline entirely.
B) Have the coordinator dynamically choose which subagents to invoke by query complexity, and broaden decomposition (with a gap-driven refinement loop) for survey topics.
C) Add retry logic at the coordinator so failed pipeline stages are re-run automatically.
D) Move synthesis before search so the synthesis agent can request only the sources it needs.

<details><summary>Show answer</summary>

**Correct answer: B.** The two problems are two sides of one orchestration mistake. For simple queries: dynamic selection of subagents (not pushing everything through the full pipeline) removes the unnecessary latency/cost. For broad topics: a broader decomposition + iterative refinement (gap assessment → targeted re-delegation → re-synthesis) fills in the coverage. The coordinator owns both routing and decomposition.
- A — caching only treats repeated lookups, not coverage and not the first query.
- C — retry is about failures, not about an excess pipeline or under-covered scope.
- D — reordering stages solves neither the latency of a simple query nor the narrow split.
</details>

### Tricky question **

**Q★★ (tricky):** A coordinator manages 4 subagents over a 12-step research run. Midway, the coordinator's own context compacts to free room. After compaction, you observe that the still-running subagents continue and complete their assigned tasks correctly, but the coordinator can no longer recall two early sub-findings when it aggregates. A teammate concludes: "Compaction corrupted the subagents' memory — we should disable compaction so subagents keep their state." What is the most accurate assessment?

A) Correct — main-conversation compaction also truncates each subagent's transcript, so disabling it preserves subagent state.
B) Wrong — subagent transcripts are stored separately and are unaffected by main-conversation compaction; the real gap is that the coordinator lost durable references to early sub-findings, so persist aggregation inputs (e.g., write results to a durable store / file) rather than disabling compaction.
C) Correct in spirit — subagents share the coordinator's context, so any compaction there propagates into them.
D) Wrong — the fix is to give each subagent a larger context window so they can resend their early findings to the coordinator.

<details><summary>Show answer</summary>

**Correct answer: B.** Two principles are woven together here. First, **context isolation**: subagent transcripts are stored as separate files and are **not affected** by compaction of the main conversation — that is why the subagents finished correctly. The symptom "the subagents' memory got corrupted" is false. Second, **context engineering / error propagation**: the one that suffered is the *coordinator* — it lost durable references to early sub-findings. The right treatment is the root cause: make the aggregation inputs durable (write the subagents' results to a long-term store/file that the coordinator reads during aggregation), not disable compaction.
- A and C — both falsely claim that parent compaction "breaks through" into the subagents or that they share its context; subagents are isolated, their transcripts are separate.
- D — a larger subagent context will not bring back what the coordinator already dropped; it treats the wrong node. The tricky part: A/C sound "logical" via the word "compaction," but it is exactly context isolation that makes them wrong about the precise cause.
</details>
