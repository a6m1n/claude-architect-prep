# 1.6 — Design task decomposition strategies for complex workflows

> **Domain 1: Agentic Architecture & Orchestration** (27%) · **Task Statement 1.6** | **Scenarios:** Multi-Agent Research, CI, Developer Productivity | **What it tests:** when to take a fixed `prompt chaining` for predictable multi-aspect work, and when to take dynamic adaptive decomposition for open-ended investigation.

## ⚡ TL;DR

- Work is **predictable and breaks down in advance** into fixed steps → `prompt chaining` (a fixed sequential pipeline). Each step receives the output of the previous one.
- Work is **open-ended**, subtasks emerge from intermediate findings → **dynamic adaptive decomposition** (orchestrator-workers): the orchestrator itself decides which subtasks to create for a specific input.
- Large code review: **per-file local passes** (even depth of attention) **+ a separate cross-file integration pass** (data flow between files). Neither one alone covers the task.
- The main decision axis: **determinism vs flexibility**. A fixed pipeline = predictability and control; dynamic decomposition = adaptability at the cost of predictability.
- Both extremes of decomposition are equally bad: **one giant pass** over all files (attention dilution, uneven depth) and **too narrow a decomposition** (whole domains of the topic are not covered).

## 📘 Theory

### Fixed pipeline (prompt chaining)

`Prompt chaining` breaks a task into a **sequence of fixed steps**, where each LLM call processes the output of the previous one. This is ideal when the task can be **cleanly and in advance** split into subtasks — the structure of the steps is known before the run and does not depend on the input. The upside: predictability, debuggability, and the ability to insert a deterministic check (gate) between steps. The canonical example from the task statement is a multi-aspect review: "analyze each file separately, then do a cross-file integration in a separate pass." The steps are known in advance → a fixed chain.

### Dynamic adaptive decomposition (orchestrator-workers)

Here a central LLM (the orchestrator) **dynamically splits** the task, delegates subtasks to workers, and synthesizes the results. The key difference: **the subtasks are not predefined — the orchestrator creates them for a specific input**. This is needed for open-ended investigation, where you cannot know in advance how the work will break apart: scout the structure, identify high-impact areas, build a prioritized plan that **adapts as dependencies are discovered** (for example, adding full test coverage to a legacy codebase — what to test first becomes clear only after mapping the dependencies).

### The choice principle: matching workload to mechanism

This is a question of matching the **workload to the mechanism**, and at the same time — **determinism vs probabilism**. If you *can* list the steps before the run — take a fixed pipeline (cheaper, more predictable, more controllable). You take dynamic decomposition only when listing the subtasks in advance is **impossible**. Applying an orchestrator to a predictable task is overengineering; applying a fixed chain to a genuinely open-ended task — it will "break" on what was not anticipated.

### Granularity: between two extremes

- **Too coarse** (one pass over 14 files): attention gets diluted (attention dilution) — depth is uneven, obvious bugs get lost, the same pattern is flagged in one file and approved in another. The fix is **architectural** (split the passes by scope), not increasing the context window: window size ≠ attention quality.
- **Too narrow** (decomposing "AI in creative industries" → only digital art / graphic design / photography): whole domains of the topic (music, writing, film) are never assigned to a worker → a structural gap in coverage. A worker cannot be more complete than the slice it was handed; an ideal downstream will not recover an unassigned domain.

Rule: **cut each pass to fit what it can analyze with quality, then add a separate pass for the links between scopes** — and make sure the decomposition covers the whole problem space.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **Fixed `prompt chaining` for an open-ended investigation** | A fixed chain assumes the steps are known in advance. In a genuinely open task (prioritizing work in an unfamiliar legacy codebase) subtasks emerge from intermediate findings — you need an adaptive orchestrator, otherwise the plan won't account for discovered dependencies. |
| **One giant pass over all files** | Attention dilution: uneven depth, missed bugs, contradictory verdicts on identical code. This is a symptom of the pass scope, not the model. |
| **"Let's take a model with a larger context window"** | A classic substitution: **window size ≠ attention quality**. All files will *fit*, but uneven depth and contradictions are about the distribution of attention, and context doesn't fix that. |
| **Too narrow a decomposition** | The orchestrator is responsible for covering the whole problem space at the decomposition stage. A narrow breakdown leaves whole domains without a worker → incomplete coverage that downstream agents will not recover. |
| **"Let the synthesis agent catch the coverage gaps"** | Good practice, but not the **root cause**: if a domain isn't assigned at the decomposition stage, by synthesis time there is simply no data for it. The gap is baked in upstream. |
| **Consensus voting (≥2 of 3 full runs)** | Suppresses the real, intermittently caught bugs (exactly the ones a diffuse single-pass loses), triples the cost, and doesn't add the cross-file pass that's actually needed against contradictions. |

## 🔑 Remember

- **Steps known in advance → `prompt chaining`** (fixed pipeline). **Subtasks depend on findings → orchestrator-workers** (dynamic decomposition).
- Large review = **per-file local pass (depth) + a separate cross-file integration pass (data flow)**.
- **"All workers succeeded, but coverage is incomplete" → a defect in the decomposition** (root cause at the orchestrator), not in the workers' logic.
- **Context window ≠ attention.** Increasing the window does not cure attention dilution.
- The decision axes: **determinism vs flexibility** and **matching workload to mechanism**. Don't complicate a predictable task with a dynamic orchestrator.

## 📝 Practice

*Answer for yourself first, then expand the "Answer".*

**Q1 · easy.** A documentation pipeline must, for every release, run the same three steps in order: extract API signatures, generate reference docs from them, then lint the docs against a style guide. The step sequence never changes between releases. Which decomposition strategy fits best?

A) A fixed sequential pipeline (prompt chaining) where each step consumes the previous step's output
B) An orchestrator that dynamically generates subtasks per release based on what it finds
C) A single prompt that does extraction, generation, and linting in one pass
D) Three independent agents run in parallel with no ordering, then a vote on the final docs

<details><summary>Show answer</summary>

**Correct answer: A.** The steps are known in advance, don't change between releases, and each depends on the previous one's output — this is the gold standard for fixed `prompt chaining`. B is overengineering: a dynamic orchestrator is not needed where the subtasks are predefined (a violation of matching workload to mechanism). C — a single pass mixes three different tasks and loses the deterministic gate between steps (attention dilution in miniature). D — parallelism and voting break the mandatory order (generation depends on the extracted signatures) — the wrong mechanism for the workload.
</details>

**Q2 · medium.** Your team must add comprehensive tests to a large legacy codebase nobody fully understands. You cannot list the test subtasks up front: what to test first depends on dependencies you'll only discover while mapping the code. What is the most effective decomposition approach?

A) Write one fixed chain: read all files, then generate tests for all of them in a single pass
B) Map the structure and high-impact areas first, then build a prioritized plan that adapts as dependencies are discovered
C) Predefine a fixed prompt chain of exactly five steps and run it unchanged regardless of findings
D) Split the work strictly per-file with no integration step, generating tests file-by-file in isolation

<details><summary>Show answer</summary>

**Correct answer: B.** The task is genuinely open-ended — the priority of the subtasks emerges from the discovered dependencies, so you need adaptive decomposition: scout the structure, find high-impact areas, build a plan that adapts. A — one giant pass → attention dilution and uneven depth. C — a fixed chain for an open-ended task: "run it unchanged whatever turns up" contradicts the very nature of the task. D — too narrow a decomposition: tests often verify the interaction of modules, and per-file isolation without integration will leave gaps in coverage.
</details>

**Q3 · hard.** A PR touches 14 files. Your single-pass review gives deep feedback on some files, superficial comments on others, misses obvious bugs, and even flags a pattern in one file while approving identical code in another. What is the most effective restructuring?

A) Switch to a higher-tier model with a larger context window so all 14 files get attention in one pass
B) Require developers to split the PR into 3–4 file submissions before the automated review runs
C) Analyze each file individually for local issues, then run a separate integration pass over cross-file data flow
D) Run three full-PR passes and only flag issues that appear in at least two of the three runs

<details><summary>Show answer</summary>

**Correct answer: C.** The symptoms (uneven depth + contradictory verdicts) = attention dilution; it's cured by splitting by scope: a per-file local pass gives even depth, and a separate integration pass catches the cross-file data flow. A — the "window ≠ attention" substitution: the files will fit, but the quality of attention distribution will not improve. B — a process band-aid: it shifts the work onto the developers, while the review system itself still doesn't know how to handle a large PR. D — consensus voting suppresses the intermittently caught real bugs, triples the cost, and doesn't add a cross-file pass.
</details>

### Tricky question **

**Q★★ (tricky):** A multi-agent research system answers "Impact of AI on creative industries." Every subagent succeeds — search finds relevant articles, the analysis agent summarizes papers correctly, synthesis produces coherent output — yet the final report covers only visual arts and entirely omits music, writing, and film. The orchestrator's log shows it split the topic into "AI in digital art," "AI in graphic design," and "AI in photography." What is the **root cause**, and what does it imply about strategy?

A) The web search agent's queries weren't broad enough; widen them to cover more creative sectors
B) The synthesis agent lacks instructions to detect coverage gaps in the findings it receives
C) The orchestrator's task decomposition was too narrow, excluding entire domains — fixed prompt-chaining wouldn't have helped because the defect is in how the problem space was carved, not in step ordering
D) The system should have used a fixed sequential pipeline instead of an orchestrator, which would have guaranteed full coverage

<details><summary>Show answer</summary>

**Correct answer: C.** All workers succeed, yet coverage is incomplete → the defect is at the **decomposition** stage, not at execution: the orchestrator assigned only three visual subtasks, and music/writing/film weren't handed to any worker. A worker is no more complete than its slice — an ideal downstream won't recover an unassigned domain. The second half of C is why this is tricky: the problem is **not** in the choice of "fixed vs dynamic" and not in the ordering of steps, but in the **granularity/coverage** of the split; changing the mechanism by itself doesn't cure it.

Why the others are traps: **A** — the search agent only searches within the assigned subtasks; it was never asked to cover music/writing/film, so the limitation is in the assignment, not in the breadth of the queries. **B** — gap-checking at synthesis is useful, but it's not the root cause: by synthesis time there is already no data for the missed domains, the gap is baked in higher up. **D** — a tempting substitution: a fixed pipeline by itself does **not** guarantee completeness — if its steps are likewise loaded with only the visual domains, coverage will be just as incomplete; what cures it is not the type of mechanism, but a correct decomposition of the problem space.
</details>
