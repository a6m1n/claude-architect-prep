# 2.1 — Design effective tool interfaces with clear descriptions and boundaries

> **Domain 2: Tool Design & MCP Integration** (18%) · **Task Statement 2.1** | **Scenarios:** Multi-Agent Research, Customer Support | **What it tests:** the tool description is the main selection signal; expanding a thin description is the cheapest first step; keywords in the system prompt can override even a good description.

## ⚡ TL;DR

- **The description (`description`) is the primary tool-selection mechanism.** The model routes almost entirely by the names and descriptions it sees. Thin or similar descriptions → unreliable choice among close tools.
- **The root cause of misrouting is the interface itself**, not orchestration. Fix the tool contract; do not add classifiers or few-shot on top.
- **Expanding the description is a low-cost first step** (input formats, example queries, edge cases, the "when this one, not the similar one" boundary). Renaming and splitting a tool are valid too, but heavier.
- **The system prompt is keyword-sensitive and can OVERRIDE a good description.** If the choice correlates with a single word in the input, look for an explicit rule in the system prompt instead of editing an already-clean tool.
- Diagnose the **layer**: base model → tool descriptions → system prompt. Editing the wrong layer will not work.

## 🧠 In plain words

> **Analogy:** 🔑 two doors with similar signs — a visitor walks through the wrong door; rewrite the signs, do not station a traffic warden.
>
> The model picks a tool by its `description` — that is the first and main signal. If two descriptions are thin or look alike, the model guesses between neighbors and ends up in the wrong place. The cheapest first step is to EXPAND the description: inputs, example queries, edge cases, "use THIS one for X, not the similar one for Y". Note: a keyword in the system prompt ("for policy, search everywhere") can OVERRIDE even a clean description.
>
> 🚩 **Detector:** if the answer jumps to few-shot, a new classifier/routing layer, or merging tools — that is `symptom-not-root-cause` (a symptom, not the root); the right move is to first separate and expand the descriptions themselves (lever 4), and when there is a deterministic "keyword → tool" link, fix the system prompt.
>
> 🧷 **Reflex:** the sign matters more than the traffic warden — first rewrite the `description`, then look at the system prompt.

## 📘 Theory

### Why the description is the routing contract

When the model (or the coordinator in a multi-agent system) decides which tool to call, it relies on the text loaded into context: **the tool names and `description`s**. That is the selection interface. A minimal description ("analyzes content") gives the model nothing to grab onto to tell a tool apart from its neighbor. The official Anthropic article directly calls "prompt-engineering your tool descriptions" one of the most effective ways to improve tools — even small edits to descriptions give "dramatic improvements".

The design principle here is **root-cause-over-symptom**: wrong routing almost always means the interface *by which* routing happens is ambiguous. Audit the interface first, then the orchestration.

### What a good description should contain

- **Input formats** — the expected formats and terms (make specialized query formats explicit).
- **Example queries** — typical requests the tool is meant for.
- **Edge cases** — what the tool does at the boundary.
- **When-to-use-vs-similar** — an explicit boundary: "use THIS one for X, not the similar one, which is for Y". This removes the collision between close tools.

### Overlapping descriptions → misrouting

A classic trap: `analyze_content` ("analyzes content and extracts key information") versus `analyze_document` ("analyzes documents and extracts key information"). They are almost identical semantically → the model guesses. The fix: **rename and separate the descriptions** so that no neighboring tool fits the same request (for example, namespacing: `extract_web_results`, `asana_search` / `jira_search`).

### The system prompt can override the description

Tool choice is dictated by the **entire** context, not just the schemas. If the logs show an almost-deterministic pattern of "the word `account` in the input → a call to `get_customer` 78% of the time", and the descriptions are already clean — that is the signature of an **explicit keyword-sensitive rule in the system prompt** that the model dutifully follows. It is cured by revising the system prompt and removing the brittle lexical steering, not by editing the tools.

### Ladder of fixes (from cheap to expensive)

1. **Expand/clarify the description** (+ when-to-use boundaries). ← cheapest, start here.
2. **Rename** the tool to remove the name overlap.
3. **Split** a generic tool into purpose-specific ones with clear input/output contracts.
4. **Consolidate** duplicates — valid, but this is already a heavier refactor.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **Immediately jumping to few-shot examples** in the coordinator prompt | Treats the symptom, not the root. Adds tokens, is brittle, generalizes poorly to new phrasings. The thin/overlapping description remains. |
| **Building a keyword routing layer / classifier** | Over-engineering: a new component and latency for something solved by a clear description. Bypasses the model's natural-language understanding instead of using it. |
| **Consolidating tools as the FIRST step** | Consolidation is valid, but heavier. Expanding the description is cheaper and often enough — that is the "low-effort, high-leverage" first step. |
| **Ignoring keywords in the system prompt** | When the choice deterministically correlates with a word in the input, the cause is steering in the system prompt. Editing an "already-clean" tool (e.g. adding negative examples) will not work and contradicts the premise. |
| **Fixing only one of two overlapping tools** | A half-fix: if the neighbor's name/description still overlaps, the misrouting persists. |
| **Fine-tune / "we need more data"** | Ignores that the behavior is explicitly *prescribed* by an instruction; further training will not override a live routing rule. |

## 🔑 Remember

- A tool's `description` = the **primary** selection signal. A thin description → unreliable choice among similar tools.
- Put into the description: **input formats · example queries · edge cases · when-to-use-vs-similar**.
- **Namespacing** separates close tools: `extract_web_results`, `asana_search` vs `jira_search`.
- Order of fixes: **expand description → rename → split → consolidate**.
- An almost-deterministic "keyword in the input → specific tool" link = a rule in the **system prompt**, not the model's fuzzy judgment.
- Diagnose the layer: **base model / tool descriptions / system prompt** — fix exactly the source.
- `tool_choice: {"type":"auto"}` — the model decides on its own; steering goes through the system prompt; a hard guarantee comes through `tool_choice`.

## 📝 Practice

*Answer for yourself first, then expand "Answer".*

**Q1 · easy**
A research agent has two tools: `summarize` ("summarizes text") and `summarize_doc` ("summarizes a document"). Logs show the coordinator picks the wrong one ~40% of the time on uploaded-file requests. What is the most effective FIRST step?

A) Add a separate classifier component that decides "file vs text" before delegation.
B) Rewrite the descriptions to be distinct and specific, including when to use each vs the other (e.g. `summarize_doc`: "Use for uploaded PDFs/Word/spreadsheet files").
C) Add few-shot routing examples to the coordinator's system prompt.
D) Merge both tools into one generic `summarize` tool.

<details><summary>Show answer</summary>

**Correct answer: B.** Descriptions are the primary selection signal; two near-identical descriptions give the model nothing to grab onto. The cheapest and most durable fix is to separate the descriptions and add a when-to-use boundary. A is an over-engineered classifier (a patch on top of the root). C — few-shot treats the symptom, is brittle, generalizes poorly. D — consolidation is valid, but it is a heavier refactor, not the first step; and merging may hide the very distinction you need.
</details>

**Q2 · medium**
In a customer-support agent, `analyze_content` ("analyzes content and extracts key information") on the web-search subagent collides with `analyze_document` ("analyzes documents and extracts key information") on the docs subagent, causing 45% misrouting of "analyze the report I uploaded". Which fix best addresses the root cause?

A) Add few-shot examples in the coordinator prompt mapping "uploaded report → document subagent".
B) Expand ONLY `analyze_document`'s description with example file types, leaving `analyze_content` unchanged.
C) Rename the web-search tool to `extract_web_results` and rewrite its description as "processes information retrieved from web searches and URLs".
D) Insert a pre-routing classifier that detects uploaded files before delegation.

<details><summary>Show answer</summary>

**Correct answer: C.** The root is the semantic overlap of names/descriptions. Renaming + separating the description makes each tool's purpose unambiguous, removing the collision at the source (namespacing). A — few-shot, a symptom-patch, brittle. B — a half-fix: the neighbor `analyze_content` still overlaps, so the misrouting persists. D — an extra component and latency instead of removing the ambiguity.
</details>

**Q3 · hard**
The tool descriptions are well-written and unambiguous. Yet logs show: messages containing "account" call `get_customer` first 78% of the time; the same intent phrased without "account" calls `lookup_order` first 93% of the time. What is the most likely root cause?

A) The tool descriptions need negative examples specifying when NOT to use each tool.
B) The system prompt contains keyword-sensitive instructions that steer on terms like "account".
C) The base model's pretraining associates "account" with customer operations, overriding descriptions.
D) The agent needs fine-tuning on multi-concept messages mixing account and order language.

<details><summary>Show answer</summary>

**Correct answer: B.** An almost-deterministic "keyword → specific tool" link (78%/93%) is the signature of an explicit rule in the system prompt that the model dutifully follows. A contradicts the premise that "the descriptions are already clean" and fixes the wrong layer. C — base-model associations are real, but diffuse; such a systematic split points to an explicit rule, not to priors. D — you cannot override a live routing rule with further training; this is over-engineering that misses the cause.
</details>

### Tricky question **

**Q★★ (tricky):** A multi-agent system has `search_kb` ("searches the internal knowledge base") and `search_web` ("searches the web"). Descriptions are distinct and well-namespaced. Still, "find our refund policy" goes to `search_web` ~30% of the time. The system prompt includes the line: *"For any question about company policies, search broadly across all available sources for the most up-to-date answer."* What is the single most effective fix?

A) Expand `search_kb`'s description with example queries like "refund policy, PTO policy, security policy".
B) Edit the system prompt line so it does not steer policy questions toward broad/all-sources search; make it route policy questions to the internal knowledge base.
C) Add a `tool_choice`-style forcing rule that always calls `search_kb` first for policy questions.
D) Merge `search_kb` and `search_web` into one `search` tool that picks the source internally.

<details><summary>Show answer</summary>

**Correct answer: B.** Two principles combine here: (1) the descriptions are already distinct and well-named — so the defect is NOT in the tool layer; (2) the phrase in the system prompt ("search broadly across all available sources") is keyword-sensitive steering that pulls policy questions toward `search_web`, overriding the clean descriptions. The root is in the system prompt layer, so that is where the fix goes.

Why the others are traps: **A** is tempting ("expand the description — that's the low-effort first step!"), but the descriptions are already distinct and good; expanding will not remove the live instruction in the prompt that is actively pulling the other way — it fixes the wrong layer. **C** — hard forcing masks the symptom, leaving the contradicting instruction in the prompt, and makes behavior brittle for edge cases. **D** — consolidation removes a useful separation and does not eliminate the steering phrase; it is the heaviest and least precise fix.
</details>
