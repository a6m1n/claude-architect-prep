# 2.4 — Integrate MCP servers into Claude Code and agent workflows

> **Domain 2: Tool Design & MCP Integration** (18%) · **Task Statement 2.4** | **Scenarios:** Code Generation, Developer Productivity | **What it tests:** the right scope for an MCP server (`.mcp.json` vs `~/.claude.json`), `${ENV}` substitution for secrets, the fact that the tools of ALL servers are available at once, and when you need a resource catalog rather than yet another tool.

## ⚡ TL;DR

- **Scope = the sharing boundary.** A team server → project-scope `.mcp.json` (in Git, everyone gets it). A personal/experimental one → `local`/`user`-scope in `~/.claude.json` (private, just for you).
- **Secrets — only through `${VAR}`.** Commit the config structure; the token is substituted from each developer's environment: `"Authorization": "Bearer ${API_KEY}"`. Never hardcode a token in `.mcp.json` — it will end up in Git.
- **The tools of ALL servers are available at the same time.** They are discovered on connect (`tools/list`) and gathered into a single registry. You do not pick a server by hand before each call.
- **A resource is a content catalog, not a tool.** Issue summaries, doc hierarchies, DB schemas are better served as a resource (`@server:proto://path`), so the agent can see the available data without exploratory calls.
- **Take a ready-made community server** (Jira, GitHub) for standard integrations; custom — only for unique team tasks. Write rich tool descriptions, otherwise the agent will pick a built-in (`Grep`) over a more powerful MCP tool.

## 📘 Theory

### The three scopes and where they are stored

Claude Code distinguishes three levels of MCP-server configuration. The key thing on the exam — do not confuse **where it is stored** with **who gets it**:

| Scope | Loads in | Shared with team | Stored in |
| --- | --- | --- | --- |
| `local` (default) | current project | No | `~/.claude.json` |
| `project` | current project | **Yes, via version control** | `.mcp.json` at the project root |
| `user` | all your projects | No | `~/.claude.json` |

The principle is **least privilege on sharing**: share exactly as much as you need. `local` (the default) is a private server only in this project, for experiments and personal credentials you do not want in Git. `user` is for personal utilities available across all your projects, but not the team. **`project` is the only scope that travels through Git**, so this is the one that shares a server with the team: when the repo is cloned, everyone gets the same set of MCP servers from `.mcp.json`. (For project servers Claude Code asks for approval on first use — this protects against someone else's code in the repository.)

### `${ENV}` substitution: share the config without sharing the secret

The conflict: `.mcp.json` needs to be committed (otherwise the team will not get it), but the token must not be committed. The solution is **environment variable expansion**. Claude Code supports:

- `${VAR}` — the value of the environment variable `VAR`;
- `${VAR:-default}` — `VAR`, or `default` if it is not set.

Substitution works in the `command`, `args`, `env`, `url`, and `headers` fields. Each developer puts their own token in the environment, and only the structure travels through Git:

```json
{ "mcpServers": { "api-server": {
  "type": "http",
  "url": "${API_BASE_URL:-https://api.example.com}/mcp",
  "headers": { "Authorization": "Bearer ${API_KEY}" }
}}}
```

The behavior is **fail-loud**: if a required variable is not set and there is no default, Claude Code **cannot parse the config** — which is correct, an explicit error is better than a silent start without authorization.

### Discovery: the tools of all servers are available at once

When the connection is initialized, the client sends `tools/list` to each connected server and gathers the results into a **single registry** available to the model. This means: with N servers connected, the agent sees the tools of all N at the same time — there is **no manual server selection before a call**. (Additionally, `list_changed` notifications let a server dynamically change its set of tools without reconnecting.)

### Resources: a content catalog instead of exploratory calls

An MCP server exposes three primitives: **tools** (actions), **resources** (context sources), **prompts** (command templates). A **resource** is a URI-addressable data source: file contents, **DB schemas, issue summaries, documentation hierarchies**. In Claude Code a resource is referenced with an `@`-mention: `@github:issue://123`, `@postgres:schema://users`. When there is a fixed catalog of available data, a resource is more effective than yet another tool: the agent immediately sees what exists at all, and **does not burn exploratory tool calls** on reconnaissance. A tool is for an action/parameterized query; a resource is to show "here is what data I have."

### Build vs buy and tool descriptions

For standard integrations (Jira, GitHub) take a **ready-made community server** — do not reinvent it. A custom server is justified only for a team-specific workflow. And a separate quality trap: if an MCP tool is poorly described, the agent will prefer a built-in (`Grep`, `Read`) over a more powerful MCP tool. The cure is **enriching the tool's description** (what it does, when to use it, what it returns) — this is part of tool design, not prompt tuning.

## ⚠️ Traps and distractors

- **A team server in user/local scope (`~/.claude.json`).** It sounds "safe — the token is not in the repository," but `~/.claude.json` is private and does not travel through Git: on clone the teammates get **nothing**. This is the home for *personal/experimental* servers, not for shared tooling. It violates the requirement to "share through the repository."
- **Hardcoding the token in `.mcp.json`.** The scope is right (project will share the server), but the secret is now in version control — exactly what is forbidden. It looks correct, but fails on credential hygiene. The right way is `${VAR}`.
- **A secret in `CLAUDE.md` / a separate file.** `CLAUDE.md` is also tracked by Git, so the token leaks the same way as with hardcoding; plus a manual copy-paste breaks "get it automatically on clone." The supported mechanism is `${VAR}`, not a side file.
- **"Only one server loads" / "the server is selected by hand before a call."** Wrong: the tools of all connected servers are discovered on connect and available to the model at the same time. There is no manual switching.
- **Adding a tool where a resource is needed.** When the task is to show the agent a *catalog of available data* (schemas, issue summaries, doc tree), the right primitive is a **resource**, not another tool. An extra tool provokes exploratory calls instead of showing what exists right away.
- **"Fix the prompt" when the agent ignores an MCP tool.** If the model reaches for `Grep` instead of the MCP tool — that is a symptom of a weak tool description. The root cause is the tool's description/contract, not the prompt.

## 🔑 Remember

- **`.mcp.json`** (project root, in Git) = `project` scope = sharing with the team. **`~/.claude.json`** = `local`/`user` scope = private.
- `local` — default, only the current project; `user` — all your projects; both are private and live in `~/.claude.json`.
- `${VAR}` and `${VAR:-default}`; works in `command`, `args`, `env`, `url`, `headers`. A required variable with no value → config parsing fails (fail-loud).
- The tools of **all** servers are discovered on connect and available at the same time; there is no manual server selection.
- Resource = a content catalog by URI (`@server:proto://path`): DB schemas, issue summaries, doc tree — reduces exploratory calls.
- Build vs buy: a ready-made server for the standard case (Jira/GitHub), custom — for team-specific. A rich tool description so it does not lose to the built-ins.

## 📝 Practice

*Answer on your own first, then expand "Answer."*

**Q1 · easy** — A platform team wants every developer who clones the repo to automatically get the same internal MCP server, and the team uses standard `git`. Where should the server be configured?

A) In each developer's `~/.claude.json` under the `user` scope.
B) In a project-scoped `.mcp.json` file at the repo root, committed to version control.
C) In a `local`-scoped entry so it stays private per machine.
D) Pasted into `CLAUDE.md` so teammates copy it manually.

<details><summary>Show answer</summary>

**Correct answer: B.** Only `project`-scope (`.mcp.json` at the root, under version control) travels through Git and shares the server with the whole team on clone.
- A — *wrong scope*: `~/.claude.json` is private, the teammates will get nothing.
- C — *wrong scope*: `local` is also private and tied to one machine.
- D — *secret/config in a side file + manual copy-paste*: breaks "automatically on clone."
</details>

**Q2 · medium** — The same project server authenticates with a personal token. The team must commit the configuration but must NOT commit any token. What is the correct setup?

A) Hardcode the token in the `headers` block of `.mcp.json` so the server works on clone.
B) Reference the token as `${GITHUB_TOKEN}` in `.mcp.json`; each developer supplies the value via their environment.
C) Put a placeholder in `.mcp.json` and paste the real token into `CLAUDE.md` for teammates.
D) Move the server to `~/.claude.json` so the token never touches the repo.

<details><summary>Show answer</summary>

**Correct answer: B.** `${VAR}` substitution commits the structure but not the secret: the token resolves from each machine's environment on connect.
- A — *committed secret*: the scope is right, but the token is now in Git — forbidden.
- C — *secret in a tracked file*: `CLAUDE.md` is also in Git, it will leak the same way; plus a manual step.
- D — *wrong scope*: `~/.claude.json` will not share the server with the team at all.
</details>

**Q3 · hard** — An agent has GitHub, Jira and PostgreSQL MCP servers configured. Developers complain the agent burns many tool calls just exploring which database tables and which open issues exist before doing real work. What is the most effective fix?

A) Tell developers to manually select the relevant server at the start of each task so fewer tools load.
B) Merge all three into one custom server so only one set of tools loads at connection time.
C) Expose the DB schema and open-issue summaries as MCP **resources** so the agent sees the available data catalog up front instead of probing with tools.
D) Add a stronger system prompt instructing the agent to avoid unnecessary tool calls.

<details><summary>Show answer</summary>

**Correct answer: C.** The problem is exploratory calls to scout "what exists at all." A resource catalog (DB schema, issue summaries) shows the available data right away, by URI, without probing calls. This is an application of the principle "a resource instead of a tool for a content catalog."
- A — *the myth of manual selection*: servers are not selected by hand, and the tools of all servers are discovered on connect anyway.
- B — *wrong root cause*: merging into one server does not remove the exploratory calls; the tools still load, and build-vs-buy is violated.
- D — *fix the prompt, not the source*: a prompt will not replace a structured data catalog; the root cause is the absence of resources.
</details>

### Tricky question **

**Q★★ (tricky):** A team adds a project-scoped `.mcp.json` with a custom server whose tool `code_search` is more capable than the built-in `Grep`. Each developer exports `SEARCH_TOKEN` and the config uses `"Authorization": "Bearer ${SEARCH_TOKEN}"`. In practice the agent still keeps using `Grep` and rarely calls `code_search`. The token resolves fine and the server connects (`/mcp` shows it green with its tools). What is the MOST likely root cause and fix?

A) The server is in the wrong scope; move it to `~/.claude.json` so its tools take precedence over built-ins.
B) `${SEARCH_TOKEN}` expansion silently fails, so `code_search` is disabled — switch to hardcoding the token.
C) The agent never discovers `code_search` because only one server's tools load at a time — reduce the number of configured servers.
D) The `code_search` tool description is too thin, so the agent defaults to the more familiar built-in `Grep`; enrich the tool's description (what it does, when to use it, what it returns).

<details><summary>Show answer</summary>

**Correct answer: D.** The key is the condition in the task: the server is **connected, green, the token resolves, the tool is visible** (`/mcp` confirms this). So scope, `${VAR}` and discovery all work — every "infrastructure" cause is already ruled out. What remains is the quality of the **tool's description**: with a weak description the agent prefers the familiar built-in (`Grep`) over the more powerful MCP tool. The cure is enriching the description — this is tool design, not a change of scope or prompt.
- A — *wrong scope as a false cure*: moving it to `~/.claude.json` does not raise the tool's "priority" over the built-in, and on top of that removes the server from the team; scope precedence is about name-duplicate servers, not about choosing tool vs built-in.
- B — *contradicts the condition*: if substitution were broken, the config **would not parse** (fail-loud) and the server would not be green; the task explicitly says the token resolves. Hardcoding is a secret-hygiene anti-pattern.
- C — *the myth of "one server at a time"*: the tools of all servers are discovered on connect at the same time; the tool in the condition **is visible**. Reducing the number of servers is not related to the cause.
</details>
