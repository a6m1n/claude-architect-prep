# 2.2 ‚Äî Implement structured error responses for MCP tools

> **Domain 2: Tool Design & MCP Integration** (18%) ¬∑ **Task Statement 2.2** | **Scenarios:** Customer Support, Multi-Agent Research | **What it tests:** can you design the error contract of an MCP tool so the agent picks the right recovery path (retry / fix / escalate / accept) instead of guessing.

## ‚ö° TL;DR

- The `isError: true` flag tells you *that* the tool failed, but not *which kind* of failure it is. Return structured metadata: `errorCategory` (`transient` / `validation` / `business` / `permission`), an `isRetryable` boolean (also called `retriable`), and a human-readable description.
- A uniform `"Operation failed"` with no category and no flag is an anti-pattern: the agent cannot pick a recovery, and either retries the non-retryable or gives up on the retryable.
- **Access failure ‚âÝ valid empty result.** A timeout is an access failure (a retry decision). `0 results found` is a *successful* query with no matches ‚Äî it is **not** an error, but a valid finding.
- You can only retry `transient`. `business`/`permission` ‚Üí `isRetryable: false` + a friendly explanation for the customer. `validation` ‚Üí fix the input, do not retry blindly.
- In multi-agent: the subagent **first recovers locally** on transient failures, and forwards to the coordinator only what could not be resolved locally ‚Äî together with partial results and a description of the attempts.

## üßÝ In plain words

> **Analogy:** üö¶ a tool error is the board at a train station. It should say "train delayed, reason ‚Äî weather, wait 20 min", not flash "something went wrong".
>
> When a tool fails, it must tell the agent THREE things: which kind of failure it is (`errorCategory`), whether it can be retried (`isRetryable`), and what happened in human language. The catch: an empty result is NOT a failure. `0 matches` is a successful query that simply found nothing (a valid finding), while a timeout is an access failure where you have to decide about a retry. Merging them into one loses the signal. A temporary failure (`transient`, will pass on retry) the agent fixes itself; it sends up only what it could not fix itself, plus "what I tried".
>
> üö© **Detector:** if in the response all failures are collapsed into `"Operation failed"` with no category and no flag ‚Äî that is symptom-not-root-cause; the right way is a typed error (`errorCategory` + `isRetryable` + description). If `0 results` is marked as an error (or a timeout is returned as "empty") ‚Äî that is conflation/an extreme; the right way is to preserve the meaning of the outcome.
>
> üß∑ **Reflex:** say WHICH failure it is and WHETHER it can be retried; empty is not a failure; fix the temporary one yourself, send up only the unresolved.

## üìò Theory

### Why structure an error (principle: error propagation)

A tool is a contract. The agent makes a recovery decision exactly as well as the tool's response is informative. If all failures collapse into one `"Operation failed"`, the agent loses the distinction between "retry", "fix the input", "apologize to the customer", and "escalate". The MCP specification distinguishes two error channels: **protocol errors** (JSON-RPC ‚Äî unknown tool, invalid arguments) and **tool execution errors**, which are returned in the result with `isError: true`. It is the second channel that you enrich with metadata.

### The four failure categories

| `errorCategory` | Example | `isRetryable` | What the agent does |
|---|---|---|---|
| `transient` | timeout, service unavailable, rate limit | `true` | retry (preferably locally) |
| `validation` | invalid input, missing field | `false` | fix the arguments and call again |
| `business` | policy violation (refund denied) | `false` | explain to the customer in a friendly way |
| `permission` | no rights / auth required | `false` | escalation / re-authorization |

For `business` violations, add a **customer-friendly explanation** so the agent can phrase a proper response to the customer instead of returning a raw error code.

### Access failure versus valid empty result (principle: preserve the semantics of the outcome)

The subtlest part of the topic. A patent-database timeout and `0 results found` from an industry report are **categorically different events**, even though both are "not 15 articles". A timeout = access failure ‚Üí the coordinator decides about a retry. `0 results` = a successful, completed query with an empty set ‚Üí it is an informative finding, not an error. Collapsing them into one "error" (or returning an empty result on an access failure) destroys the signal needed for smart recovery.

### Local recovery before propagating (principle: recover locally before propagating)

In a multi-agent system the subagent should handle transient failures itself (retry, fallback) and forward to the coordinator **only** what cannot be resolved locally ‚Äî with partial results and a description of the attempts. But "locally" does not mean "silently forever": if a transient failure has exhausted the local attempts, the coordinator must learn about it so it can manage the time and retry budget. In building a multi-agent research system, Anthropic notes directly: "letting the agent know when a tool is failing and letting it adapt works surprisingly well" ‚Äî model adaptivity plus deterministic retries/checkpoints.

## ‚öÝÔ∏è Traps and distractors

- **Generic `"Operation failed"`.** It sounds "safe", but without `errorCategory` and `isRetryable` the agent does not pick a recovery: it retries a denied refund (`business`) and gives up on a timeout (`transient`). This is the root of the problem in the task.
- **Valid empty result treated as an error** (or an access failure treated as an empty success). `0 results` is a successful query; marking it as an error wastes the coordinator's attention and throws away the finding. The reverse case (returning empty on a timeout) feeds the agent a false "no data" instead of a retry.
- **Retrying the non-retryable.** A blanket retry "up to 3 times on any `isError`" applies the transient strategy to all categories: it burns attempts on `business`/`permission`/`validation` and still loses the category for downstream decisions.
- **Propagating every transient failure without local recovery.** It floods the coordinator with noise and load; it violates the principle "recover locally, propagate only the unresolved". The mirror error is a **silent** local retry with no report: it hides budget spend from the coordinator and strips it of power over the retry policy.
- **"Always escalate to a human on `isError`".** It kills first-contact-resolution: timeouts and fixable validation errors should be resolved locally, not by people.

## üîë Remember

- `isError` = *what* failed; `errorCategory` + `isRetryable` + description = *which kind* of failure and *what to do*.
- Categories: `transient` (retry) ¬∑ `validation` (fix input) ¬∑ `business` (`isRetryable:false` + customer-friendly) ¬∑ `permission` (non-retryable, escalate/auth).
- MCP tool response: `{ "content": [...], "isError": true }`. In the Claude API the tool error channel is a `tool_result` with `is_error: true`.
- Write **instructive** error messages: not `"failed"`, but `"Rate limit exceeded. Retry after 60 seconds."` (the official recommendation).
- `0 results` ‚âÝ error. A timeout = access failure (a retry decision). Never return empty on an access failure.
- Subagent: local recovery for transient ‚Üí to the coordinator only the unresolved + partial results + "what was tried".

## üìù Practice

*Answer for yourself first, then expand "Answer".*

**Q1 ¬∑ easy.** A Customer Support agent calls `process_refund`, which already sets `isError: true` but always returns the body `"Operation failed"`. The agent keeps retrying refunds that were denied for policy reasons. What is the most effective change to the tool contract?

A) Wrap every `process_refund` call in an automatic retry of up to 3 attempts before surfacing the error.
B) Return structured metadata ‚Äî `errorCategory` (`transient`/`validation`/`business`/`permission`), an `isRetryable` boolean, and a human-readable description.
C) On any `isError`, immediately escalate the conversation to a human agent.
D) On failure, log internally and return an empty result so the agent can continue the conversation.

<details><summary>Show answer</summary>

**Correct answer: B.** The root of the problem is typed, machine-readable errors: with `errorCategory` and `isRetryable` the agent retries only `transient`, and explains a refund denial (`business`, `isRetryable:false`) to the customer without useless retries.
A ‚Äî *retrying the non-retryable*: a blanket retry applies the transient strategy to `business`/`permission` and burns attempts. C ‚Äî *always escalate*: it kills first-contact-resolution for fixable cases. D ‚Äî *valid-empty/access-failure conflation*: returning empty hides the failure as "no data".
</details>

**Q2 ¬∑ medium.** A web-search subagent queries three sources: academic DBs return 15 papers, industry reports return `0 results found`, and the patent DB returns `Connection timeout`. How should the subagent report outcomes so the coordinator can make the best recovery decisions?

A) Aggregate all three into a single coverage metric (e.g. `67% source coverage`), with detailed logs available on request.
B) Report both the `0 results` and the `Connection timeout` as failures requiring coordinator intervention.
C) Distinguish the access failure (timeout ‚Üí retry decision) from the valid empty result (`0 results` ‚Üí successful query, an informative finding).
D) Retry the timeout internally and report only the two successful sources, omitting the timeout entirely.

<details><summary>Show answer</summary>

**Correct answer: C.** A timeout and `0 results` are semantically different: the first is an access failure (the coordinator decides about a retry), the second is a successful query with an empty set (a valid finding). Preserving the semantics at the report boundary is what makes recovery smart.
A ‚Äî *collapsing into a metric*: `67%` erases the per-source semantics, the coordinator does not know what to retry. B ‚Äî *valid-empty as an error*: marking `0 results` as a failure throws away the finding and wastes attention. D ‚Äî *silent local retry with no report*: it hides budget spend and strips the coordinator of power over the retry policy.
</details>

**Q3 ¬∑ hard.** A `lookup_inventory` MCP tool can fail in several ways. Which response design lets the agent recover correctly across all cases without burning effort on the wrong path?

A) Always return `isError: true` with `errorCategory` so the agent has a category for every outcome, including empty searches.
B) Return `isError: true` + `errorCategory`/`isRetryable` + a human-readable description for genuine failures, and return `isError: false` with an empty list for a successful search that matched nothing.
C) Return `isError: true` for timeouts and for empty results, but set `isRetryable: true` on both so the agent always retries.
D) Return a single `"Operation failed"` body for all failures but attach a numeric internal error code for the logs.

<details><summary>Show answer</summary>

**Correct answer: B.** The key is not to mark a *successful* query with no matches as an error: an empty list goes with `isError: false`, while real failures go with a category, a retry flag, and a description. That way the agent retries transient, fixes validation, explains business, and accepts an empty finding as a fact.
A ‚Äî *valid-empty as an error*: hanging an `errorCategory` on an empty search makes a successful query a "failure". C ‚Äî *retrying the non-retryable + conflation*: `isRetryable:true` on an empty result forces a retry of something that already succeeded. D ‚Äî *generic `"Operation failed"`*: an internal code for the logs does not help the agent at runtime.
</details>

### Tricky question **

**Q‚òÖ‚òÖ (tricky):** A research subagent hits a `transient` rate-limit on one of its data sources. It retries locally twice with backoff; both retries also hit the rate limit. The source is still unavailable. What should the subagent do?

A) Keep retrying locally with longer backoff until the source succeeds, since the error is `transient` and therefore `isRetryable: true`.
B) Treat the exhausted source as a valid empty result (`0 matches`) and report success, so the coordinator's budget is not disturbed.
C) Propagate to the coordinator the unresolved `transient` failure plus any partial results gathered and what was attempted, letting the coordinator own the retry/budget decision.
D) Silently drop the failing source and report only the sources that succeeded, to keep the report clean.

<details><summary>Show answer</summary>

**Correct answer: C.** Two principles meet here. Yes, `transient` means `isRetryable: true` ‚Äî so the subagent **first** recovers locally (and it did make two attempts with backoff). But `isRetryable` is permission to retry, **not** an obligation to retry forever. When the local attempts are exhausted, the failure becomes "unresolved locally", and the principle shifts to *propagate the unresolved error with partial results and what was attempted*. Only the coordinator owns the time/retry budget for the whole workflow and must explicitly decide: retry later, switch the source, or accept the gap.
A ‚Äî it is tempting and sounds technically correct (`transient` ‚áí retryable), but it confuses "can retry" with "retry infinitely locally"; this quietly burns the budget and takes power over the policy away from the coordinator. B ‚Äî *access failure as a valid empty result*: an exhausted source did not give "0 matches", it stayed unavailable; marking it an empty success means feeding the coordinator false data. D ‚Äî *silent drop*: it hides the failure, depriving the coordinator of visibility and the chance to recover.
</details>
