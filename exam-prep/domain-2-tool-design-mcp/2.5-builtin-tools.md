# 2.5 — Select and apply built-in tools (Read, Write, Edit, Bash, Grep, Glob) effectively

> 🔖 **Study emphasis:** Built-in-tool selection is a high-yield, frequently-tested area. Drill **`Grep` vs `Glob`**, **`Bash` not as a substitute** for the dedicated tools, and the **`Edit` → `Read`+`Write` fallback** until they're automatic.

> **Domain 2: Tool Design & MCP Integration** (18%) · **Task Statement 2.5** | **Scenarios:** Developer Productivity with Claude | **What it tests:** the ability to pick the built-in tool that fits the task (`Grep` = contents, `Glob` = paths, `Edit` = targeted change on a unique match) and to explore a codebase incrementally instead of reading everything at once.

## ⚡ TL;DR

- `Grep` searches the **contents** of files (function names, error strings, `import`s). `Glob` searches by file **paths/names** (`**/*.test.tsx`). Don't swap them — that's the main trap of this topic.
- `Read`/`Write` are **whole-file** operations; `Edit` is a targeted change on a **unique** text anchor.
- If `Edit` fails because of a **non-unique** match, fall back to `Read` + `Write` (a reliable fallback), not a retry of `Edit` on the same anchor.
- Explore the codebase **incrementally**: first `Grep` finds the entry points, then `Read` follows the imports and traces the flow. Don't read the whole repo up front — that bloats the context window and lowers quality.
- To trace usage through wrapper modules: first find all exported names, then `Grep` for each one across the codebase.

## 📘 Theory

### Each tool has its own semantics

The principle of this topic is **matching workload to mechanism**: pick the tool whose semantics match the question.

- **`Grep` → contents.** When you need to find *where* in the code something happens (who calls `validate_token`, where a handler is defined, where an error string appears) — that's a search by file contents.
- **`Glob` → paths.** When you need to find files *by name or extension* (all tests `**/*.test.tsx`, all `*.config.js`) — that's path matching by pattern. `Glob` knows nothing about what's *inside* a file.
- **`Read` / `Write` → whole file.** `Read` loads the entire file, `Write` writes/overwrites it entirely.
- **`Edit` → targeted change.** It changes a specific fragment on a **unique** text anchor (`old_string` → `new_string`).

### Why `Edit` requires uniqueness and what to do when it fails

`Edit` finds the `old_string` match and replaces it. If the anchor appears in the file **several times**, `Edit` cannot unambiguously choose the spot and **fails** — this is a guard against editing the wrong fragment. The right response is to **not** retry `Edit` on the same non-unique anchor (it will fail again) and to **not** guess. The reliable fallback: `Read` the whole file → make the change → `Write` the entire file. An alternative is to widen the anchor by adding surrounding context so it becomes unique. This is an example of **root-cause-over-symptom**: the cause of the failure is the non-unique anchor, not "the tool is buggy".

### Incremental exploration instead of "read everything"

The key skill of this topic is **context engineering**. To understand an unfamiliar codebase, you can't `Read` all the files up front: that overflows the context window, and model quality drops as it fills (performance degrades as the context grows). The right pattern is **explore-first, narrow-down**:

1. `Grep` finds the entry points and key symbols (a handler, function calls, an error string).
2. `Read` selectively opens the files you found and follows their `import`s, tracing the real execution flow.
3. To trace through wrapper modules: first gather all exported names, then `Grep` for each one across the codebase.

This way the context fills only with relevant code, not noise. The same principle applies to delegating heavy exploration to a subagent that reads files in *its own* context and returns only a summary.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
| --- | --- |
| **`Glob` for a content search** (or `Grep` for finding file names) — the tools are swapped | `Glob` matches **paths**, not contents: file names can't tell you who calls a function or where an error string is. `Grep` is the opposite — it's about contents. This is the most common distractor of the topic. |
| **`Read` all files up front** (or `Bash` `cat` over the whole `src/`) | Overflows the context window with irrelevant code, model quality drops, the signal drowns in noise. The opposite of incremental exploration: the right way is `Grep`, then a selective `Read`. |
| **Retrying `Edit` on a non-unique anchor** | If `old_string` is non-unique, `Edit` will keep failing. You need to either widen the anchor until it's unique, or fall back to `Read` + `Write`. |
| **`Glob` alone, to "understand the architecture from file names"** | File names don't reveal behavior or relationships; you need `Grep` over contents + `Read` along the imports. |

## 🔑 Remember

- Search by **contents** → `Grep` (functions, errors, `import`s). Search by **paths/names** → `Glob` (`**/*.test.tsx`).
- Whole file → `Read` / `Write`. Targeted change on a **unique** anchor → `Edit`.
- `Edit` fails on a non-unique match → fall back to `Read` + `Write` (or widen the anchor until it's unique).
- Explore incrementally: `Grep` (entry points) → `Read` (follow the imports). Don't read the whole repo.
- Tracing through wrappers: first all exported names, then `Grep` each one.

## 📝 Practice

*Answer for yourself first, then expand "Answer".*

**Q1 · easy.** You need to find every place in the codebase that calls the function `validate_token`. Which built-in tool is the right choice?

A) `Glob`, because it pattern-matches across the repository.
B) `Grep`, because it searches file contents for the symbol.
C) `Read` on each file in `src/` until you find the calls.
D) `Edit`, to locate and annotate the call sites.

<details><summary>Show answer</summary>

**Correct answer: B.** `validate_token` is code **contents**, and a content search is a job for `Grep`. A is the "swapped tools" trap: `Glob` searches by file paths/names, not by the text inside. C is "read everything": it bloats the context and is inefficient. D — `Edit` edits, it doesn't search, and it requires a unique anchor.
</details>

**Q2 · easy.** You want to locate all test files in a large monorepo to run them. They follow the pattern `**/*.test.tsx`. Which tool fits best?

A) `Grep` for the string `.test.tsx` inside every file.
B) `Read` the directory tree and eyeball the filenames.
C) `Glob` with the pattern `**/*.test.tsx`.
D) `Bash` `cat` of the repo to spot test files in the output.

<details><summary>Show answer</summary>

**Correct answer: C.** Finding files **by name/extension** is path matching by pattern, exactly the job of `Glob`. A swaps the tools: `Grep` is about contents, but here you need a path. B/D — "read everything"/`cat` the whole thing: context overflow and poor signal-to-noise.
</details>

**Q3 · medium.** You ask `Edit` to replace the string `return result` in a file, but it fails because that exact text appears in three functions. What is the most reliable next step?

A) Retry the same `Edit` with the identical `old_string` until it succeeds.
B) Read the whole file, make the change, and Write the file back (or extend the anchor with surrounding context to make it unique).
C) Switch to `Glob` to find the right line number first.
D) `Grep` for `return result` and delete all but one occurrence.

<details><summary>Show answer</summary>

**Correct answer: B.** The cause of the failure is a **non-unique** anchor, so the correct fix is `Read` + `Write` as a reliable fallback (or widen `old_string` with context until it's unique). This is root-cause-over-symptom. A — a retry on the same anchor will fail again. C — `Glob` searches paths, not strings inside a file. D — deleting matches breaks the code; that isn't editing the right spot.
</details>

### Tricky question **

**Q★★ (tricky):** You join a team and must add a feature to an unfamiliar auth flow that is wired through several thin **wrapper modules** (each re-exports symbols from another). Reading the whole repo would blow up the context window. What is the most effective approach to trace how a given handler is actually used across the wrappers?

A) `Glob` for `**/*auth*` to list every auth-related file, then infer the call graph from the filenames.
B) `Read` all the wrapper modules up front so the full re-export chain is in context before you start.
C) First identify all the exported names from the wrapper modules, then `Grep` each exported name across the codebase, and `Read` only the files that actually use them.
D) `Grep` once for the handler's name and assume those are all the call sites.

<details><summary>Show answer</summary>

**Correct answer: C.** The subtlety is the combination of two principles: **incremental exploration** (context engineering) AND correct **tracing through wrappers**. When a symbol passes through a chain of re-exports, a single `Grep` on the original name is not enough: a wrapper module may export it under a different name, and direct calls are hidden behind the wrapper. So the right order is to first gather **all** exported names, then `Grep` each one across the codebase, and only then `Read` selectively. D is tempting ("just `Grep`"), but the trap is exactly that a single search on the original name will miss usages through renamed re-exports. A — `Glob` gives paths, but not relationships/calls (file names don't reveal the call graph). B — "read everything up front" overflows the context, which the task explicitly forbids.
</details>
