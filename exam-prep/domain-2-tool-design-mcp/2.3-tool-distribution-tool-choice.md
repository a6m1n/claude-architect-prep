# 2.3 — Distribute tools appropriately across agents and configure tool choice

> **Domain 2: Tool Design & MCP Integration** (18%) · **Task Statement 2.3** | **Scenarios:** Multi-Agent Research | **What it tests:** least privilege at the role level, narrow cross-role tools for frequent cases, and configuring `tool_choice` (`auto` / `any` / forced).

## ⚡ TL;DR
- **Least privilege at the tool level.** An agent's behavior is limited not by its prompt but by the set of tools it can call. Give each agent only what its role needs.
- **Too many tools hurts.** 18 tools instead of 4–5 raises the difficulty of choosing and lowers the reliability of tool selection. Fewer, but more precise.
- **A narrow fast-path instead of blanket access.** Cover the frequent common case with one scoped tool (for example `verify_fact`); route the rare complex case through a coordinator. Do not hand the agent the whole arsenal.
- **Replace generic with constrained.** `fetch_url` → `load_document` with format validation: making something impossible by contract is better than saying "don't do that" in the prompt (root cause, not symptom).
- **`tool_choice`:** `auto` (may return text), `any` (must call some tool, but picks it itself), `tool`+`name` (forces a specific tool), `none` (tools forbidden).

## 🧠 In plain words

> **Analogy:** 🔑 tools are keys: give an agent only the key to its own office, not the master key with a note saying "go only to your own room."
>
> What an agent **can** do is decided by its tools, not by the text of the prompt. Give a role the 4–5 keys it needs, not 18: with a big keyring the model gets confused and grabs the wrong one. Cover the frequent common case with one narrow key (`verify_fact`); hand the rare complex case off through a 🗼 coordinator. And `tool_choice` decides something else: `auto` (may answer with text) · `any` (must call some tool) · forced `{"type":"tool","name":"X"}` (must call exactly this one).
>
> 🚩 **Detector:** if the answer is "give the agent all the tools" or "remove the tool entirely / route everything through the coordinator" — that is an extreme (over- / under-engineered); the right move is to **reshape the tool itself** to fit the role's work (a narrow key for the task).
>
> 🧷 **Reflex:** a key sized to the lock: not a master key, not an empty keyring — one precise key per role.

## 📘 Theory

### Tool distribution = least privilege for agents
An agent's capabilities are defined by its tools, not its instructions. A generic tool implicitly gives an agent every capability that tool provides — including unplanned ones. So proper tool distribution is an application of the principle of least privilege: the surface the agent can access should match what its role actually requires, not the union of everything it might occasionally use.

Two concrete failures the exam tests:
- **Overload.** 18 tools instead of 4–5 degrades tool selection: the model picks the wrong tool more often because the decision complexity grows.
- **Tools outside the specialization.** A synthesis agent with access to web-search tools starts doing ad-hoc searches — that is not its job, and it does it poorly, breaking separation of concerns.

### Scoped cross-role tools: optimize the frequent path, escalate the rare one
Sometimes a role needs access to another role's capability — but not all of it. If 85% of cases are simple fact checks (dates, names, numbers) and 15% require deep investigation, the right solution is to give the agent a **narrow single-purpose tool** for the common case (`verify_fact`) and **leave complex verifications to delegation** through the coordinator to the web-search agent. That way you remove most round-trips (a latency win) without expanding the agent's surface beyond its role. This is exactly "optimize the dominant path, escalate the rare hard case."

### Replace generic tools with constrained alternatives
The best way to remove unwanted behavior is to make it impossible at the level of the tool contract, not to discourage it with a prompt. `fetch_url` retrieves any URL, so searching via a search engine's result page is reachable *by design*. Replacing it with `load_document`, which validates that the URL points to a document, removes that possibility while keeping the legitimate work. This is root-cause-over-symptom: you fix the tool's affordance instead of whispering rules to the model. Likewise, the official recommendation is to replace broad tools (`list_contacts`) with targeted ones (`search_contacts`), consolidate related operations, and use namespacing (`asana_search`, `jira_search`) so the choice is unambiguous.

### `tool_choice`: controlling *whether* and *which* tool the model calls
- **`auto`** — the model decides for itself whether to call a tool or answer with text. The default when `tools` is passed.
- **`any`** — the model **must** call one of the tools, but picks which one itself.
- **`tool` + `name`** (forced) — the model **must** call the specific named tool. Example: forcing `extract_metadata` as the first step.
- **`none`** — tools forbidden (the default when `tools` is not passed).

An important detail: with `any` and `tool` the API prefills the assistant message, so the model **will not emit natural-language text** before `tool_use`. If you need explanatory text together with a guaranteed call — use `auto` plus an explicit instruction in the `user` message, or `any` + `strict: true` to guarantee both the call and schema conformance.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **Give the synthesis agent all web-search tools** | Over-provisioning: breaks separation of concerns, expands the surface beyond the role, degrades the agent on its main task, and raises the risk of misuse. It solves latency at the cost of least privilege. |
| **Blanket tool access for all agents** | The same thing at system scale: rising decision complexity (18 vs 4–5), agents reaching into others' specializations. Convenience ≠ reliability. |
| **Batching all verifications into the coordinator / to the end of the pass** | Creates blocking dependencies: verification is needed *during* synthesis to continue; deferring all checks to the end breaks the dynamic flow and still drives everything through the heavy path. |
| **Speculative caching of context "just in case"** | You can't predict in advance which dynamic claims synthesis will want to verify; you waste work and context, and you still miss the check you actually need. |
| **Fix it with a prompt / block domains (for a generic tool)** | Instructions only *discourage*; the capability remains; domain block-lists are fragile, need maintenance, and miss new search engines. A symptom, not the root. |
| **Remove the tool entirely and route everything through the coordinator** | Over-correction: you throw away a legitimate capability (loading documents) and add an extra hop with latency. Reshape the tool, don't remove the work. |

## 🔑 Remember
- 4–5 tools per agent is the guideline; 18 is a red flag (decision complexity).
- Generic → constrained: `fetch_url` → `load_document` (validates format); `list_contacts` → `search_contacts`.
- A scoped fast-path (`verify_fact`) for the frequent common case + delegation of the rare complex one through the coordinator.
- `tool_choice`: `auto` (may return text) · `any` (must call, the model chooses) · `{"type":"tool","name":"..."}` (forces a specific one) · `none` (forbidden).
- `any` and `tool` suppress preamble text before `tool_use`. `strict: true` guarantees schema conformance.

## 📝 Practice
*Answer it yourself first, then expand "Answer."*

**Q1 · easy**
You are building a multi-agent research system. The synthesis agent is responsible only for combining findings into a final report. During design, what tool distribution is most appropriate for it?

A) Give it the full set of web search, document-fetch, and code-execution tools so it never has to delegate.
B) Give it only the tools its role requires, adding limited cross-role tools only for specific high-frequency needs.
C) Give every agent in the system the same complete toolset for consistency.
D) Give it no tools and have it ask the coordinator for everything via prompts.

<details><summary>Show answer</summary>

**Correct answer: B.** Least privilege at the tool level: the agent's surface matches its role, plus narrow cross-role tools for frequent needs.
- A — *over-provisioning*: breaks separation of concerns; the synthesis agent will misuse capabilities that aren't its own.
- C — *blanket access*: growing decision complexity and agents stepping beyond their specialization.
- D — the extreme in the other direction: it deprives the agent even of the scoped tool it needs and loads the coordinator with extra round-trips.
</details>

**Q2 · medium**
A document-analysis agent was given a general-purpose `fetch_url` tool to load documents from URLs. Logs show it now frequently fetches search-engine result pages to run ad-hoc web searches, which should route through the web-search agent. What is the most effective fix?

A) Add a line to the agent's prompt telling it to use `fetch_url` only for document URLs, not for searching.
B) Block `fetch_url` calls to known search-engine domains while allowing all other URLs.
C) Replace `fetch_url` with a `load_document` tool that validates the URL points to a document format.
D) Remove `fetch_url` entirely and route all URL loading through the coordinator to the web-search agent.

<details><summary>Show answer</summary>

**Correct answer: C.** This is a root-cause fix: you narrow the affordance at the level of the tool contract, making search *impossible*, while keeping legitimate document loading.
- A — *prompt-nudge*: it only discourages; the capability remains, and the model drifts back.
- B — *domain-blocklist*: fragile, needs maintenance, misses new search engines — it treats the symptom.
- D — *over-correction*: it throws away a needed capability and adds a hop with latency.
</details>

**Q3 · medium**
You need to guarantee that on every request the model calls a tool (rather than replying with plain text), but you do not care which of the available tools it picks. Which `tool_choice` setting fits?

A) `tool_choice: {"type": "auto"}`
B) `tool_choice: {"type": "any"}`
C) `tool_choice: {"type": "tool", "name": "extract_metadata"}`
D) `tool_choice: {"type": "none"}`

<details><summary>Show answer</summary>

**Correct answer: B.** `any` requires calling some tool but leaves the choice to the model — exactly what the task requires.
- A — `auto` allows returning text without a call, so there is no guarantee.
- C — `tool`+`name` forces a *specific* tool, but the choice doesn't matter to us (over-constrained).
- D — `none` forbids tools entirely.
</details>

**Q4 · hard**
The synthesis agent currently returns control to the coordinator whenever it must verify a claim; the coordinator calls the web-search agent and re-invokes synthesis. This adds 2–3 round-trips and ~40% latency. Evaluation shows 85% of verifications are simple fact-checks (dates, names, stats) and 15% need deep investigation. What is the most effective approach?

A) Give the synthesis agent access to all web-search tools so it can verify anything directly without round-trips.
B) Have the web-search agent proactively cache extra context around each source so synthesis rarely needs to verify.
C) Have the synthesis agent accumulate all verification needs and send them as one batch at the end of its pass.
D) Give the synthesis agent a scoped `verify_fact` tool for simple lookups; keep complex verifications delegating through the coordinator.

<details><summary>Show answer</summary>

**Correct answer: D.** A narrow fast-path handles 85% of the common case directly (removing most round-trips), while the 15% of complex ones stay on delegation — latency drops without expanding the agent's surface.
- A — *over-provisioning*: blanket access breaks separation of concerns and least privilege.
- B — *speculative caching*: you can't guess which dynamic claims will be needed; it wastes context and still misses.
- C — *batching*: verification is needed *during* synthesis; deferring everything to the end breaks the dynamic flow and creates blocking dependencies.
</details>

### Tricky question **
**Q★★ (tricky):** A structured-extraction agent must always start each request by calling `extract_metadata`, and you also want the model to emit a short natural-language note to the user explaining what it's about to extract, in the same turn, before the tool call. Which configuration achieves both?

A) `tool_choice: {"type": "tool", "name": "extract_metadata"}` — it forces the right tool, and the preamble text comes along automatically.
B) `tool_choice: {"type": "any"}` plus `strict: true` on `extract_metadata`.
C) `tool_choice: {"type": "auto"}` with an explicit instruction in the user message to call `extract_metadata` first.
D) `tool_choice: {"type": "tool", "name": "extract_metadata"}` plus a system-prompt instruction to "explain before calling the tool".

<details><summary>Show answer</summary>

**Correct answer: C.** The key subtlety: with `tool_choice` `any` or `tool`, the API **prefills the assistant message and suppresses natural-language text before `tool_use`** — even if you explicitly ask for an explanation, there won't be one. So any forced variant is incompatible with the requirement of "a short note to the user before the call." The only way to get *both* the explanatory text *and* the right tool is `auto` (which allows a preamble) plus an explicit instruction in the `user` message to call `extract_metadata` first. In practice the model almost always obeys such an instruction, and the text is preserved.
- A — sounds right ("forced + text will come along on its own"), but the forced mode is exactly what **removes** the preamble — a correct-sounding-but-wrong trap.
- B — `any` also suppresses the preamble; `strict: true` only validates the schema and does not solve the text problem.
- D — `tool` forces the tool, but a system prompt asking to "explain before calling" is powerless: the mode still suppresses text before `tool_use`.
</details>
