# 5.4 — Manage context effectively in large codebase exploration

> **Domain 5: Context Management & Reliability** (15%) · **Task Statement 5.4** | **Scenarios:** Developer Productivity with Claude | **What it tests:** how to fight context degradation during long exploration of a large codebase — scratchpad files, delegation to subagents, a manifest for crash recovery, `/compact`.

## ⚡ TL;DR

- **Degradation symptom (context rot):** the model starts answering with generic "typical patterns" instead of the specific classes and flows it found earlier. This is a signal that the window is full — you cannot ignore it.
- **Scratchpad files** move key findings outside the context window, and they survive context boundaries; you refer to them on later questions.
- **Delegate verbose exploration to a subagent**: it reads files in its own clean window and returns only a condensed summary — the main agent keeps just high-level coordination.
- **Summarize between phases** and inject the summary at the start of the next phase; `/compact` shrinks a bloated session.
- **Crash recovery:** each agent exports its state (`manifest`) to a known location; on resume the coordinator loads the manifest and injects it into the prompts.

## 📘 Theory

### Why context degrades (principle: context is a finite budget)

The context window is a **finite resource** that the architect must actively curate, not leave to chance. The more tokens pile up (verbose output from reading files, dependency traces), the worse the model is at accurately retrieving specific information from that window — this phenomenon is called **context rot**. The practical symptom in Developer Productivity: after long exploration of the refund pipeline, the model stops naming the specific classes it found earlier and falls back on generalizations like "a refund flow usually works like this." This is not "the model got dumber" — it is window overflow, and you fix it structurally, not by "asking more insistently."

### Scratchpad files: durable facts outside the window

A key context-engineering technique is to **write key findings into note files outside the context** and load them back when needed. A specific class, a service name, an entry point into the flow — this is a **durable fact**, and you must not lose it when crossing a context boundary. A scratchpad file survives any window compaction/restart; referring to it on the next question restores accuracy.

### Subagent delegation: isolating verbose output

Instead of reading **everything** into the main agent (which is exactly what overflows the window), delegate noisy tasks to a subagent: "find all test files," "trace the dependencies of the refund flow." The subagent explores in **its own clean window** and returns only a **condensed summary** upward. The main agent keeps a high-level understanding and coordination, while the messy output of reading hundreds of files never enters its context. This is separation of concerns at the context level.

### Manifest and `/compact`: state persistence and compaction

For **crash recovery**, each agent exports structured state — a `manifest` — to a known location. On resume the coordinator loads the manifest and injects it into the agents' prompts, so the multi-phase exploration continues without losing what was found. Between phases, **summarize the key findings before launching the next phase's subagents**. When the session bloats with verbose discovery output, `/compact` shrinks the conversation: it summarizes the content and reinitializes the window with the summary, keeping the important decisions and discarding redundant output.

#### `/compact` and CLAUDE.md: what survives

An important edge case for the exam: after `/compact` the session context is compacted, but not everything is lost equally:

- **project-root CLAUDE.md** (`.claude/CLAUDE.md` or a root `CLAUDE.md`) — is **re-injected** automatically after compaction. Claude Code reloads it as part of context restoration.
- **Nested / directory CLAUDE.md** (for example, `src/CLAUDE.md`) — is **not re-injected** automatically; it was loaded "on demand," and after compaction it may be unavailable until the next access to that directory.

Implication for D3: `/compact` is a tool for managing **session size**, not a way to reset configuration. Root-level instructions (`CLAUDE.md`) are preserved; specific directory rules are in question.

This same distinction explains why **additional instructions are better kept in a project-level CLAUDE.md** or `.claude/rules/` with path-scoping — they survive compaction more reliably than nested directory files.

## ⚠️ Traps and distractors

| Trap archetype | Why it's wrong |
|---|---|
| **No scratchpad → findings are lost** | If you do not move key findings into a file, the specific classes/flows are lost when you cross a context boundary, and the model falls back to "typical patterns" again. Keeping everything "in the head" of the session is exactly the cause of degradation. |
| **Read everything into the main agent** (instead of delegating) | Loading all the verbose output into the main window **maximizes** the token load that caused context rot in the first place. You delegate to a subagent precisely to isolate the noise; "reading everything yourself" makes the problem worse. |
| **No manifest for crash recovery** | Without exporting structured state to a known location, a session crash wipes out hours of exploration — there is nothing to restore. Relying on "I'll just pick up where I left off" without a manifest is fragile. |
| **Ignoring the signs of degradation** | Answers about "how it usually works" instead of specifics are not cosmetic — they are a signal of overflow. Ignoring it and "asking more insistently" / "raising `max_tokens`" does not fix the cause: `max_tokens` limits the length of the **output**, not the memory of the input context. |
| **Paste the entire repository into context** | "So all classes are always available" sounds logical, but it is a direct route to window overflow and maximal recall degradation. |
| **"`/compact` resets all CLAUDE.md instructions"** | No. The project-root CLAUDE.md is re-injected automatically. `/compact` compacts the conversation history, not the project configuration. |

## 🔑 Remember

- **context rot symptom** = generic "typical patterns" instead of the specific classes/flows found earlier → the window is full.
- **Scratchpad files** = durable facts outside the context; refer to them on later questions.
- **Subagent** reads in **its own** window, returns only a **summary**; the main agent = coordination.
- **Summarize between phases**, then launch the next phase's subagents.
- **`manifest`** = export state to a known location; the coordinator loads it on resume and injects it into prompts.
- **`/compact`** = shrink a bloated session during long exploration.
- **`max_tokens`** ≠ memory; it limits the length of the **output**, it does not help with recall of the input context.

## 📝 Practice

*Answer it yourself first, then expand the "Answer."*

**Q1 · easy.** During a long Claude Code session exploring a legacy billing service, the agent starts answering questions about the refund pipeline with generic statements like "a typical refund flow usually validates then posts a credit" instead of naming the specific classes it identified an hour earlier. What does this most likely indicate?

A) The model needs a higher `max_tokens` to remember earlier discoveries.
B) Context degradation (context rot) — the window is saturated and recall of specific findings has dropped.
C) The repository is too small for the model to reason about.
D) A bug in the `Read` tool that corrupted the earlier file contents.

<details><summary>Show answer</summary>

**Correct answer: B.** The shift from specific classes to "how it usually works" is the canonical symptom of context rot: the window is full of verbose output, and the accuracy of retrieving specifics has dropped. A is a misconception trap: `max_tokens` limits the length of the output, not the memory of the input context. C is irrelevant: the problem is overflow, not a shortage of material. D is an invented cause (root-cause-over-symptom: look for a structural cause, not an imaginary tool bug).
</details>

**Q2 · medium.** You need a subagent to trace all dependencies of the refund flow across hundreds of files, but you do not want to pollute the main agent's context with verbose file-read output. What is the most effective approach?

A) Read every file into the main agent's context so all classes are always available, then ask the question.
B) Delegate the dependency trace to a subagent that explores in its own context window and returns only a condensed summary to the main agent.
C) Raise `max_tokens` on the main agent so it has room for all the file contents.
D) Keep asking follow-up questions in the same session and prompt the model more firmly to recall details.

<details><summary>Show answer</summary>

**Correct answer: B.** The subagent explores in **its own** clean window and returns only a condensed summary upward — the main agent keeps high-level coordination, while the noise of reading hundreds of files never enters its context. A is "read everything into the main agent": it maximizes the token load and worsens the degradation. C — `max_tokens` is about output length, not memory. D — "ask more insistently": it does not fix the structural cause of overflow.
</details>

**Q3 · hard.** A multi-phase research run over a large monorepo (find test files → trace refund dependencies → map service boundaries) occasionally crashes mid-run, losing hours of discovery. Which combination most effectively sustains accuracy AND survives crashes?

A) Persist key findings to scratchpad files, delegate verbose exploration to subagents, summarize between phases, and have each agent export a `manifest` to a known location that the coordinator reloads on resume.
B) Run the whole exploration in one long session and rely on `--continue` to resume after a crash.
C) Paste the entire monorepo into context at the start so nothing can be lost, and increase `max_tokens`.
D) After each phase, run `/compact` and trust the compacted summary to fully reconstruct state after any crash.

<details><summary>Show answer</summary>

**Correct answer: A.** This is the full set: scratchpad files (durable facts outside the window) + delegation to subagents (noise isolation) + summarizing between phases + a `manifest` in a known location that the coordinator loads on resume — it is the manifest that provides crash recovery. B relies on "I'll pick up where I left off" without structured state (no manifest → fragile). C — paste the whole repo + `max_tokens`: a double trap that maximizes overflow and confuses output with memory. D — `/compact` reduces context **in a live session**, but that is not persistent state on disk; after a crash the compacted session summary will not restore the exported state the way a manifest does.
</details>

### Tricky question **

**Q★★ (tricky):** Your context is filling with verbose discovery output and answers are starting to drift toward generic patterns. A teammate says: "Just run `/compact` — it summarizes the conversation, so all your specific findings about the discovered classes are now safely preserved and you never need scratchpad files." For a long, crash-prone multi-phase exploration, what is the most accurate assessment?

A) Correct — `/compact` preserves all specific findings, so scratchpad files and manifests are redundant.
B) `/compact` is the right tool to *reduce in-session context usage*, but it is a lossy in-session summary, not durable persistence — specific findings must still be externalized to scratchpad files, and crash recovery still requires a `manifest` written to a known location.
C) Wrong — `/compact` increases context usage, so it should never be used during exploration.
D) `/compact` raises the effective `max_tokens`, which is what actually restores recall of earlier classes.

<details><summary>Show answer</summary>

**Correct answer: B.** The subtlety is in mixing up two different functions. `/compact` really does **reduce context usage** during long exploration and it is the right tool against a bloated window — but it is a **lossy** compaction of the live session: the summary may lose specifics, and it lives **inside** the session, not on disk. So (1) durable facts still have to be moved out into **scratchpad files**, and (2) crash recovery requires a **`manifest`** in a known location that the coordinator reloads on resume — `/compact` will not survive a crash. A is a "correct-sounding" trap: it overrates `/compact` and wrongly declares scratchpad/manifest unnecessary. C is factually wrong: `/compact` actually **reduces** context usage. D is the `max_tokens` misconception (output length ≠ input memory), and `/compact` does not "raise" it anyway.
</details>
