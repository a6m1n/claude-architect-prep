# 5.3 — Implement error propagation strategies across multi-agent systems

> **Domain 5: Context Management & Reliability** (15%) · **Task Statement 5.3** | **Scenarios:** Multi-Agent Research System | **What it tests:** how a subagent reports failures to the coordinator so it can make a sensible recovery decision — without hiding errors and without killing the whole workflow.

## ⚡ TL;DR

- **Handle an error at the lowest level that can solve it.** The subagent absorbs transient failures itself (retry with backoff, skip a broken file) and escalates to the coordinator only what it cannot solve — **with context**: what it tried to do, what partial results it got, what alternatives exist.
- **Structured error context > generic status.** `failure type` + `attempted query` + `partial results` + `alternatives` let the coordinator make a smart decision. A generic «`search unavailable`» / «`operation failed`» hides the context and makes recovery impossible.
- **Access failure ≠ valid empty result.** A timeout is an access failure that needs a retry-decision. «`0 results`» is a **successful** query with an empty set — a valid finding. You cannot dump them into the same bucket.
- **Two anti-patterns, both "failure":** (1) silently suppress the error (return emptiness as `success`) and (2) crash the whole workflow because of one failure. Both destroy the system's reliability.
- **Synthesis of partial data:** graceful degradation with **coverage annotations** — deliver the value that is ready plus an explicit label of what is well-supported and where the gaps are because of unavailable sources.

## 📘 Theory

### Why you need structured error context

A subagent in a multi-agent system is not just an executor, but an **outcome reporter**. The coordinator does not see the subagent's internal context (in the Agent SDK the subagent returns only its final message to the parent — the intermediate tool calls stay inside it). So the only channel through which the coordinator learns about a failure is the structure of what the subagent escalates upward.

That is why you must structure the error so the coordinator can **branch** on it without re-investigating. The minimal useful context:

- **failure type** — what exactly happened (timeout, parsing error, auth/password-protected, rate limit).
- **attempted query / operation** — what the subagent tried to do.
- **partial results** — what it did manage to get before the failure.
- **alternative approaches** — what other paths exist (a different source, an extended timeout, a skip).

This is a direct application of the **root-cause / informed-recovery** principle: the coordinator must decide retry/skip/fail based on *why* the outcome happened, not only on *succeeded/failed*.

### Local recovery vs escalation

The **least-capable-level** principle: fix a failure where it is cheap. The subagent that owns the task absorbs its own routine transient failures (retry with backoff on a timeout, skip on a broken/protected file) and **does not bother the coordinator for every exception**. Only the residual goes upward — what the subagent could not resolve, and always with context.

An important caveat about the visibility boundary: the subagent absorbs transient failures locally, **but does not hide them completely** — the coordinator must have visibility into the retry policy and the budget. That is, "fixed locally" means "resolved the outcome," not "pretended nothing happened and burned the budget invisibly." The timeout/retry policy and the budget allocation are the coordinator's responsibility (separation of concerns).

### Access failure vs valid empty result

Two "not-15-results" outcomes are categorically different:

| Outcome | What it is | What the coordinator does |
|---|---|---|
| `Connection timeout` | **access failure** — the query did not complete | make a retry-decision (retry, extend the timeout, budget) |
| `0 results found` | **valid empty result** — the query **succeeded**, it is just empty | accept it as an informative finding, do not retry |

Dumping them into one metric («67% coverage»), into hidden internal retries, or under a generic «failure» label destroys exactly the signal the coordinator needs.

### Synthesis of partial data: graceful degradation

When some upstream sources worked and some did not, the synthesis subagent should neither hide the gaps, nor hard-fail, nor block on a speculative retry. The right approach is **coverage annotations**: deliver the value that is ready plus structured metadata about the uncertainty («this is well-supported by three sources; on topic X there is no data — news archive timed out»). This way the downstream consumer/user calibrates their own trust.

## ⚠️ Traps and distractors

- **Generic error status («`search unavailable`», «`operation failed`»).** It sounds like a clean, uniform contract, but it hides the failure type, attempted query, and partial results from the coordinator. Without this context the coordinator cannot choose retry vs skip vs fail — a generic status *hides valuable information*.
- **Silent suppression — an empty result marked `success`.** The most insidious: it simplifies the coordinator's branching, but it destroys observability and lets quiet data-quality problems flow straight into synthesis. Masking a failure as a success = losing reliability for the sake of convenience.
- **Escalating the exception to a top-level handler that kills the whole workflow.** "Fail loud" on a single transient failure throws away all the work already done and over-escalates a degraded-but-usable result. One broken PDF should not crash a research task.
- **Not distinguishing an access failure from a valid empty result.** Labeling «`0 results`» as a failure wastes the coordinator's attention and throws away an informative finding; the reverse mistake — accepting a timeout as "just empty" — loses the needed retry.
- **Hidden internal retries with no report.** Internal retry of transient failures is a real pattern, but if the subagent retries silently and only reports persistent errors, it invisibly burns the budget and takes the retry policy away from the coordinator.
- **The subagent dictates retry to the coordinator («first retry the timeout sources with an extended timeout»).** This breaks separation of concerns, assumes the retry will succeed, and stalls the already-available useful output.
- **Pre-validation by the coordinator before dispatch.** It sounds like prevention, but it duplicates the parsing logic and lifts the handling upward; many failures (timeout, hidden corruption) cannot be predicted without processing the file.
- **A separate error-handling agent through a shared queue.** It looks like a clean separation of responsibility, but it adds coupling and a bottleneck and centralizes decisions that the owning subagent would solve locally.

## 🔑 Remember

- Structured error context = **failure type + attempted query/operation + partial results + alternatives**.
- Subagent: **local recovery for transient → propagate only the unresolvable**, always with context and partial results.
- **`timeout` = access failure (retry-decision)**; **`0 results` = valid empty result (success, a finding)**. Do not confuse them.
- Two forbidden anti-patterns: **silent suppression** (emptiness as `success`) and **kill-the-whole-workflow** on a single failure.
- Synthesis of partial data → **coverage annotations** (what is supported / where the gaps are because of unavailable sources), not silent omission / hard fail / speculative retry.
- The retry policy and the budget belong to the coordinator, not the subagent (separation of concerns).
- In MCP, tool execution errors are reported through `isError: true` in the result (not as a protocol error) — that is exactly how a structural tool failure reaches the model/agent.

## 📝 Practice

*Answer on your own first, then expand "Answer".*

**Q1 · easy** — A document-analysis subagent occasionally hits a corrupted PDF that throws a parsing exception. Currently every exception is re-raised to a top-level handler that aborts the entire research task. What is the most effective improvement?

A) Have the subagent catch the exception and return an empty result marked `success` so the workflow never stops.
B) Have the subagent retry/skip the bad file locally and propagate to the coordinator only errors it cannot resolve, including what was attempted and any partial results.
C) Wrap the whole pipeline so any subagent exception triggers a full restart of the research task.
D) Return a generic `operation failed` status to the coordinator for every parsing problem.

<details><summary>Show answer</summary>

**Correct answer: B.** Local recovery of transient failures at the subagent level + escalating only the unresolvable with context — this is the least-capable-level and informed-recovery principle. A is silent suppression (emptiness as `success`, observability is lost). C is kill-the-whole-workflow on a single failure (throws away all the work). D is a generic status that hides the failure type/attempted query/partial results from the coordinator.
</details>

**Q2 · medium** — A web-search subagent queries three categories: academic DB returned 15 papers, industry reports returned `0 results found`, patent DB returned `Connection timeout`. How should it report to the coordinator for the best recovery decisions?

A) Aggregate into a single `67% source coverage` metric with detailed logs available on request.
B) Report both the `0 results` and the timeout as failures requiring coordinator intervention.
C) Distinguish the access failure (timeout → retry-decision) from the valid empty result (`0 results` → successful query, accept as a finding).
D) Retry the timeout internally and report only persistent errors, omitting the rest.

<details><summary>Show answer</summary>

**Correct answer: C.** Timeout and `0 results` are semantically different outcomes: the first is an access failure (needs a retry-decision), the second is a valid empty result (a successful query, an informative finding). A — aggregation erases the per-source semantics. B — labeling valid emptiness as a failure wastes attention and throws away a finding. D — a hidden internal retry with no report burns the budget invisibly and takes the retry policy away from the coordinator.
</details>

**Q3 · hard** — The web-search subagent returned results for only 3 of 5 source categories (two timed out); document analysis succeeded fully. The synthesis subagent must now produce a findings summary from this mixed-quality input. Most effective error-propagation strategy?

A) Have the synthesis subagent ask the coordinator to retry the timed-out sources with extended timeouts before synthesizing, to guarantee complete coverage first.
B) Synthesize using only the successful sources and produce output without indicating which data was unavailable.
C) Structure the synthesis output with coverage annotations: which findings are well-supported vs which topic areas have gaps from unavailable sources.
D) Return an error to the coordinator about incomplete upstream data, triggering a full retry or task failure.

<details><summary>Show answer</summary>

**Correct answer: C.** Graceful degradation with transparency: deliver the value that is ready + structured uncertainty (coverage annotations) so the consumer calibrates trust. A — the subagent dictates the retry policy to the coordinator (a breach of separation of concerns) and stalls the output. B — silent omission: it removes the uncertainty signal and misleads about how well the findings are supported. D — a hard fail/full retry on a partial success throws away the work done and over-escalates.
</details>

**Q4 · hard** — A subagent fetches from an external API. On `429 Too Many Requests` it backs off and retries twice, then succeeds. On a `403 Forbidden` (revoked credential) it cannot proceed. How should it handle each so the coordinator can recover intelligently?

A) On both cases, immediately propagate the raw exception to a top-level handler that aborts the run.
B) Resolve the `429` locally with backoff/retry (no escalation needed once it succeeds); for the unrecoverable `403`, propagate a structured error with failure type, the attempted query, partial results, and alternative approaches.
C) For both, return an empty result marked `success` and embed the status in metadata for synthesis to filter later.
D) Report both as a generic `service error` so the coordinator can uniformly decide what to do.

<details><summary>Show answer</summary>

**Correct answer: B.** `429` is transient: the subagent absorbs it itself (backoff/retry) and, since it resolved, there is nothing to escalate. `403` (revoked credential) is unresolvable locally: it needs structured error context (type + attempted query + partial results + alternatives) so the coordinator can choose an alternative source or escalate. A — kill-the-workflow. C — silent suppression. D — a generic status hides the failure type and the context that is the only thing you can branch on.
</details>

### Tricky question **

**Q★★ (tricky):** A research subagent must call a flaky internal search service. The architect wants the system to (a) survive intermittent `5xx` blips without bothering the coordinator and (b) still let the coordinator manage the overall time/retry budget. Which design best satisfies BOTH goals?

A) The subagent silently retries every failure internally until it succeeds, and reports only the final outcome — keeping the coordinator out of routine errors entirely.
B) The subagent retries transient `5xx` blips locally with bounded backoff; if they resolve it proceeds, but if retries are exhausted it propagates a structured error (failure type, attempted query, partial results, retry attempts made) so the coordinator owns the budget decision.
C) The subagent never retries; it propagates every `5xx` upward immediately so the coordinator has full visibility and total control over the budget.
D) The subagent returns partial results with a `success` status and embeds the `5xx` details in metadata, letting the coordinator filter problems during synthesis.

<details><summary>Show answer</summary>

**Correct answer: B.** This combines two principles that seem to be in conflict: **local recovery** (least-capable-level) and **the coordinator's visibility into the budget** (separation of concerns). B solves both: bounded local retry absorbs the routine blips (goal a), and on exhaustion — structured propagation with the number of attempts hands the *budget decision* to the coordinator (goal b).

Why the traps fail:
- **A** — the most tempting: formally "does not bother the coordinator" (goal a) is met, but "silently retries until it succeeds" violates goal b — invisible retry with no report burns the time/retry budget that the coordinator should manage. This is the hidden-internal-retry anti-pattern: it wins only because it sounds like "clean autonomy."
- **C** — over-escalation: it gives visibility (goal b), but it fails goal a — the coordinator is bothered on every transient blip, which is the original problem of excessive coordinator involvement.
- **D** — silent suppression: it masks the `5xx` as `success`, destroys observability; "details in metadata" do not help, because the outcome is marked successful and never enters the retry logic.
</details>
